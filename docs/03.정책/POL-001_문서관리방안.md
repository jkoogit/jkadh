# POL-001 문서관리방안

> 문서 ID: POL-001  
> 문서 유형: 정책  
> 상태: Draft  
> 성숙도: Candidate  
> 버전: v0.1  
> 소유자: jk  
> 작성 에이전트: Codex  
> 기준 브랜치: dev  
> 작업 브랜치: task_codex/001-docs-governance-foundation  
> 최종 수정일: 2026-07-05

## 1. 목적

본 문서는 JKADH 저장소의 공식 문서를 생성, 명명, 변경, 검토, 승격하는 기준을 정의한다.

문서는 단순한 설명서가 아니라 세션이 바뀌어도 AI와 사람이 같은 기준으로 작업하기 위한 공식 Artifact이다.

## 2. 적용 범위

이 문서는 JKADH 저장소의 모든 공식 문서에 적용한다.

다음 항목은 본 문서의 적용 대상이다.

- 문서 폴더
- 문서 ID
- 파일명
- 문서 Header
- 문서 상태
- 문서 성숙도
- 작업 이력
- 문서 변경 절차

## 3. 문서의 역할

JKADH에서 문서는 다음 역할을 가진다.

| 역할 | 설명 |
|---|---|
| 기억 | 세션이 바뀌어도 결정사항을 잃지 않게 한다. |
| 계약 | 사람과 AI가 같은 기준으로 작업하게 한다. |
| 실행 기준 | Codex, TIO, Harness가 Work Order를 수행할 때 기준으로 사용한다. |

공식 문서는 사람이 읽는 설명서에 머무르지 않고, AI가 작업에 사용할 수 있는 Specification이어야 한다.

## 4. 폴더 규칙

문서 폴더는 읽기 순서를 나타내는 번호와 한글명을 함께 사용한다.

폴더 번호는 문서 ID에 포함하지 않는다. 폴더 구조가 바뀌어도 문서 ID가 불필요하게 흔들리지 않도록 하기 위함이다.

| 폴더 | Prefix | Prefix Full Name | 목적 |
|---|---|---|---|
| `00.시작` | `STA` | Start | 먼저 읽는 문서와 진입점 |
| `01.프로젝트` | `PRJ` | Project | 프로젝트 정체성, 비전, 목표, 서비스, 범위 |
| `02.조직` | `ORG` | Organization | 역할, 책임, AI 개발팀 구조 |
| `03.정책` | `POL` | Policy | 우선순위, Branch, Commit, Review, Prompt, Issue, PR, Release 규칙 |
| `04.결정` | `DEC` | Decision | 반복 결정을 줄이기 위한 선택 근거와 결정 기록 |
| `05.설계` | `DSN` | Design | Architecture, API, DB, ERD, Prompt, MCP, Flow 설계 |
| `06.기획` | `PLN` | Planning | 요구사항, 서비스, 기능, Use Case, User Story |
| `07.로드맵` | `RDM` | Roadmap | Phase별 전체 일정과 추진 계획 |

| `10.운영` | `OPS` | Operations | Token, Cost, KPI, Dashboard, Monitoring, Incident 운영 |
| `11.점검` | `CHK` | Check | QA, Checklist, Security, Performance 점검 |
| `12.회고` | `RET` | Retrospective | 작업별 잘된 점, 문제, 개선 사항 |
| `13.실험` | `EXP` | Experiment | 실험, 검증, 후보 자산 |
| `14.학습` | `LRN` | Learning | 학습 자료와 지식 축적 |
| `15.로그` | `LOG` | Log | 작업 중 발생한 백로그, 기록, 로그성 항목 |

## 5. 문서 ID 규칙

문서 ID는 다음 형식을 사용한다.

```text
{PREFIX}-{SEQ}
```

예:

```text
POL-001
STA-001
ORG-001
DSN-001
ADR-001
```

규칙은 다음과 같다.

- `PREFIX`는 문서가 속한 성격을 나타내는 3자 내외의 영문 대문자이다.
- `SEQ`는 같은 Prefix 안에서 증가하는 3자리 숫자이다.
- 폴더 번호는 문서 ID에 포함하지 않는다.
- 문서를 단순히 다른 폴더로 이동하는 경우 ID는 유지할 수 있다.
- 문서의 성격이 바뀌면 새 ID를 발급한다.

## 6. 파일명 규칙

파일명은 다음 형식을 사용한다.

```text
{문서ID}_{문서명}.md
```

예:

```text
POL-001_문서관리방안.md
STA-001_시작문서_인덱스.md
ORG-001_AI_역할정의.md
DSN-001_Capability_관리모델.md
ADR-001_문서번호체계_결정.md
```

파일명 규칙은 다음과 같다.

- 문서명은 한글을 기본으로 한다.
- 공백은 사용하지 않고 `_`를 사용한다.
- 파일 확장자는 `.md`를 사용한다.
- 같은 개념은 항상 같은 이름으로 부른다.
- 약어만으로 의미가 불명확한 파일명은 피한다.

## 7. 문서 Header 규칙

모든 공식 문서는 제목 아래에 Header를 둔다.

최소 Header 형식은 다음과 같다.

```markdown
# POL-001 문서관리방안

> 문서 ID: POL-001
> 문서 유형: 정책
> 상태: Draft
> 성숙도: Candidate
> 버전: v0.1
> 소유자: jk
> 작성 에이전트: Codex
> 기준 브랜치: dev
> 작업 브랜치: task_codex/001-docs-governance-foundation
> 최종 수정일: 2026-07-04
```

필요한 경우 다음 항목을 추가할 수 있다.

- 관련 Work Order
- 관련 ADR
- 관련 Issue
- 관련 PR
- 원천 세션
- 승인자

## 8. 문서 상태

문서 상태는 검토 절차상의 위치를 나타낸다.

| 상태 | 한글명 | 의미 |
|---|---|---|
| Draft | 초안 | 작성 또는 수정 중인 문서 |
| Review | 검토 중 | 사람 또는 에이전트 검토가 진행 중인 문서 |
| Approved | 승인됨 | 공식 채택된 문서 |
| Deprecated | 폐기됨 | 더 이상 사용하지 않는 문서 |

## 9. 문서 성숙도

문서 성숙도는 조직 자산으로서의 신뢰 수준을 나타낸다.

| 성숙도 | 한글명 | 의미 |
|---|---|---|
| Candidate | 후보 | 제안된 문서 |
| Validated | 검증됨 | 하나 이상의 작업에서 검증된 문서 |
| Promoted | 승격됨 | JKADH 공통 자산으로 승격된 문서 |
| Core | 핵심 | JKADH 핵심 원칙으로 채택된 문서 |

상태는 검토 절차를 뜻하고, 성숙도는 조직 자산의 신뢰 수준을 뜻한다.

## 10. 기본 본문 구조

공식 문서는 기본적으로 다음 구조를 따른다.

```markdown
## 1. 목적
## 2. 적용 범위
## 3. 규칙
## 4. 절차
## 5. 관련 문서

---

## 작업 이력
```

문서 성격에 따라 필요한 섹션은 추가할 수 있다.

설계 문서처럼 실행 기준에 가까운 문서는 다음 항목을 사용할 수 있다.

```markdown
## 1. 목적
## 2. 책임
## 3. 입력
## 4. 출력
## 5. Capability
## 6. Workflow
## 7. 관련 문서
```

상태, 단계, 전이값처럼 운영 중 반복해서 참조하는 영문 상태값을 정의할 때는 한글명을 함께 병기한다.

표로 정의하는 경우에는 `상태`, `한글명`, `의미`를 기본 컬럼으로 사용한다.

문장이나 템플릿 안에서 나열하는 경우에는 `Accept Candidate(수용 후보)`처럼 영문 상태값 뒤에 한글명을 괄호로 병기한다.

## 11. 작업 이력 규칙

모든 공식 문서 끝에는 작업 이력을 둔다.

형식은 다음과 같다.

```markdown
---

## 작업 이력

| 작업일시 | 작업 도구 | AI 모델 | 에이전트 역할 | 작성자 | 변경 유형 | 내용 |
|---|---|---|---|---|---|---|
| 2026-07-04 | Codex | GPT-5 | 문서 작성 | jk / Codex | Create | 문서 최초 작성 |
```

작업 도구, AI 모델, 에이전트 역할은 분리해서 기록한다.

- `작업 도구`는 실제 작업을 수행한 제품 또는 실행 환경 이름을 기록한다.
- `AI 모델`은 작업에 사용한 모델명을 기록한다.
- `에이전트 역할`은 해당 작업에서 수행한 역할을 기록한다.
- `에이전트 역할` 값에는 `에이전트` suffix를 붙이지 않고 `CTO`, `기획`, `설계`, `구현`, `리뷰`, `QA`, `리서치`, `하네스`, `운영`처럼 간결하게 기록한다.
- 동일한 AI가 여러 역할을 수행할 경우 작업 단위에 맞는 역할을 기록한다.
- 특정 AI 제품명이 바뀌어도 역할 기준으로 작업 맥락을 추적할 수 있어야 한다.

변경 유형은 다음 값을 기본으로 사용한다.

| 변경 유형 | 의미 |
|---|---|
| Create | 최초 작성 |
| Revise | 내용 수정 |
| Rename | 이름 변경 |
| Move | 위치 변경 |
| Promote | 성숙도 승격 |
| Deprecate | 폐기 |

## 12. 변경 원칙

문서 변경은 변경 성격에 따라 다르게 처리한다.

| 변경 성격 | 처리 방식 |
|---|---|
| 오탈자 수정 | 바로 수정 가능 |
| 링크 수정 | 바로 수정 가능 |
| 문장 개선 | 의미가 같으면 바로 수정 가능 |
| 규칙 변경 | 작업 이력 기록 필요 |
| 정책 변경 | Review 상태 필요 |
| 핵심 원칙 변경 | ADR 필요 |

정책 문서가 바뀌면 그 정책을 참조하는 문서에 영향이 있는지 확인한다.

## 13. 대화에서 문서로 승격하는 절차

대화는 공식 기억이 아니다.

대화 내용은 다음 절차를 통해 공식 문서로 승격한다.

```text
대화
↓
핵심 주장 추출
↓
Adopted / Backlog / Rejected 분류
↓
문서 후보 작성
↓
검토
↓
공식 문서 반영
↓
작업 이력 기록
```

분류 기준은 다음과 같다.

| 분류 | 의미 |
|---|---|
| Adopted | 지금 공식 반영할 내용 |
| Backlog | 좋지만 지금은 반영하지 않을 내용 |
| Rejected | 이번에는 제외할 내용 |

## 14. Git 작업관리와의 관계

문서 변경은 Git 작업 단위로 추적한다.

브랜치, 커밋, PR, 리뷰, 승급, 릴리즈 규칙은 문서관리방안의 세부 규칙이 아니라 Git 작업관리 정책에서 정의한다.

향후 Git 작업관리 정책은 다음 문서에서 다룬다.

```text
POL-002_Git_작업관리방안.md
```

Git 작업관리 정책에는 다음 원칙을 포함한다.

- 한 `task_` 브랜치는 하나의 검토 가능한 작업 단위에 대응한다.
- PR 검토가 끝나고 `dev`에 머지되면 해당 `task_` 브랜치의 역할은 종료된다.
- 이어서 작업을 계속할 경우 최신 `origin/dev`에서 새 `task_` 브랜치를 만들어 진행한다.

본 문서는 문서 자체의 생성, 명명, 변경, 이력 규칙만 정의한다.

## 15. 후속 작업 관리

문서 작성 중 발생한 후속 작업 후보는 본 문서에서 직접 관리하지 않는다.

후속 작업 후보는 [Backlog 미해결 인덱스](../15.로그/backlog/README.md)에 등록하고, 해결되면 해당 인덱스에서 제거한다.

공식 문서 본문에는 확정된 내용만 기술한다. 후속 작업 후보, 설계 보강 항목, 별도 검토가 필요한 항목은 문서 안에서 목록으로 중복 관리하지 않고 Backlog로 분리한다.

공식 문서에서 후속 작업의 존재를 안내해야 하는 경우에는 개별 항목을 나열하지 않고 [Backlog 미해결 인덱스](../15.로그/backlog/README.md)를 관련 문서로 연결한다.

백로그 개별 파일은 생성일 기준 날짜 폴더에 고정한다. 해결 여부에 따라 파일 위치를 이동하지 않는다.
