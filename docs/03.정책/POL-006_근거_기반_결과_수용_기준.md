## 1. 목적

본 문서는 JKADH에서 Harness가 작업 결과를 수용 후보로 판단할 때 필요한 검증 근거 기준을 정의한다.

근거 기반 수용은 사람 리드의 최종 판단을 대체하지 않는다. Harness가 명확한 완료 조건과 검증 근거를 바탕으로 Low-risk 작업 결과를 수용 후보로 분류하고, 그렇지 않은 작업을 리뷰, 승인 요청, Backlog로 넘기기 위한 운영 기준이다.

## 2. 적용 범위

본 문서는 다음 작업에 적용한다.

- Issue 기반 작업 결과 검증
- Work Order 완료 조건 확인
- Harness의 Low-risk 수용 후보 판단
- Report 또는 PR에 남길 검증 근거 정리
- 검증 근거 부족으로 인한 Review(리뷰), Need Approval(승인 필요), Backlog(백로그) 전환 판단

코드 변경 결과의 최종 수용, 배포, main 영향 작업, 외부 시스템 쓰기 작업은 본 문서만으로 수용할 수 없다. 해당 작업은 [POL-005 사람 승인 경계 정책](./POL-005_사람_승인_경계_정책.md)을 따른다.

## 3. 기본 원칙

근거 기반 수용의 기본 원칙은 다음과 같다.

- 수행하지 않은 검증은 근거로 인정하지 않는다.
- 완료 조건과 직접 연결되지 않은 검증은 보조 근거로만 사용한다.
- 검증 결과는 명령, 파일, diff, 링크, 리뷰 기록처럼 추적 가능한 형태로 남긴다.
- 검증 실패, 미실행, 부분 실행은 통과로 해석하지 않는다.
- Low-risk 여부가 불명확하면 Harness 수용 후보로 분류하지 않는다.
- 수용 후보 판단은 작업 단위별로 수행한다.

## 4. 수용 판단 상태

Harness는 검증 이후 다음 상태 중 하나로 결과를 분류한다.

| 상태 | 한글명 | 의미 | 후속 처리 |
|---|---|---|---|
| Accept Candidate | 수용 후보 | 완료 조건과 필수 근거가 충족된 Low-risk 결과 | Report 또는 PR로 연결 |
| Review Required | 리뷰 필요 | 결과 또는 근거 보강이 필요한 상태 | 리뷰, QA, Execute 재진입 |
| Need Approval | 승인 필요 | 사람 승인 경계에 해당하거나 수용 판단이 불확실한 상태 | 승인 요청 |
| Backlog Required | 백로그 필요 | 현재 Issue 범위를 벗어난 후속 작업이 필요한 상태 | Backlog 등록 또는 연결 |
| Stop Required | 중단 필요 | 검증 방법, 권한, 범위 충돌로 진행할 수 없는 상태 | Stop 또는 Report |

Accept Candidate는 최종 병합 또는 최종 승인 상태가 아니다. Harness가 근거를 갖춘 수용 후보로 정리할 수 있다는 의미다.

## 5. 필수 근거

Harness가 결과를 Accept Candidate로 분류하려면 다음 근거가 모두 있어야 한다.

| 근거 | 설명 |
|---|---|
| 작업 기준 | Issue 또는 Work Order의 목적, 수행 범위, 제외 범위, 완료 조건 |
| 변경 근거 | 어떤 파일, 문서, 상태, 산출물이 변경되었는지 확인할 수 있는 기록 |
| 완료 조건 확인 | 완료 조건별 충족 여부 |
| 검증 실행 기록 | 실제 실행한 명령, 점검, 리뷰, 링크 확인, diff 확인 결과 |
| 승인 확인 | Approval Pass 또는 승인 상태와 판단 근거 |
| 후속 흐름 | Report, PR, Backlog, 공식 문서 중 연결 대상 |

필수 근거 중 하나라도 없으면 Accept Candidate로 분류하지 않는다.

## 6. 근거 유형

근거 유형은 다음과 같이 구분한다.

| 근거 유형 | 예시 | 사용 기준 |
|---|---|---|
| 기계 검증 | `git diff --check`, 테스트 실행, 빌드, 링크 검사 | 실행 결과와 성공/실패를 기록한다. |
| 구조 검증 | 문서 인덱스 갱신, 참조 경로 확인, 상태값 일치 확인 | 변경 대상과 관련 인덱스의 일관성을 확인한다. |
| 내용 검증 | 완료 조건 대조, 정책 위반 여부 확인, 범위/제외 범위 확인 | 사람이 읽고 판단한 근거를 짧게 기록한다. |
| 문서 탈고 | 문맥 흐름, 문서 간 충돌, 중복, 쉬운 표현, Header와 작업 이력 확인 | 공식 문서 작업을 태스크 정리하기 전에 수행하고 결과를 기록한다. |
| 리뷰 근거 | 리뷰 의견, QA 결과, 보강 요청 처리 기록 | 외부 검토가 필요한 작업에서 사용한다. |
| 승인 근거 | Approval Pass(승인 통과), Need Approval(승인 필요), Approval Granted(승인됨) 기록 | POL-005 기준과 연결한다. |

기계 검증이 가능한 항목은 가능한 한 기계 검증을 우선한다. 다만 문서 의미, 정책 적합성, 범위 판단처럼 기계 검증이 어려운 항목은 내용 검증을 함께 남긴다. 공식 문서 변경이 있는 경우 문서 탈고 결과도 수용 후보 판단 근거에 포함한다.

## 7. 수용 가능 조건

Harness는 다음 조건을 모두 만족할 때 결과를 Accept Candidate로 분류할 수 있다.

- 작업이 Low-risk 후보에 해당한다.
- 수행 범위와 제외 범위를 침범하지 않았다.
- 완료 조건별 확인 결과가 있다.
- 필수 근거가 모두 남아 있다.
- 승인 상태가 Approval Pass이거나 이미 승인된 범위 안에 있다.
- 검증 명령 또는 점검 결과에 실패가 없다.
- 결과가 추적 가능한 문서, PR, Report, Backlog 중 하나와 연결된다.

초기 운영에서 Accept Candidate는 문서, 메타데이터, 인덱스, Backlog 정리처럼 영향 범위가 좁고 되돌리기 쉬운 작업에 한정한다.

## 8. 수용 불가 조건

다음 조건 중 하나라도 해당하면 Harness는 결과를 Accept Candidate로 분류하지 않는다.

- 완료 조건이 없거나 완료 조건과 검증 결과가 연결되지 않는다.
- 검증을 실행하지 않았거나 실행 실패를 해소하지 않았다.
- 변경 범위가 Work Order의 수행 범위를 넘었다.
- 제외 범위를 침범했다.
- Secret, 배포, main 영향, 외부 시스템 쓰기, 자동 승급이 포함된다.
- 코드 변경 결과를 사람 리뷰 없이 최종 수용하려 한다.
- 정책 변경, 제품 방향 변경, 핵심 원칙 변경이 포함된다.
- Low-risk 여부를 명확히 판단하기 어렵다.

수용 불가 조건이 발생하면 Harness는 Review Required, Need Approval, Backlog Required, Stop Required 중 하나로 전환한다.

## 9. Work Order 기록 기준

Work Order에는 결과 수용을 위해 다음 항목을 포함한다.

```markdown
## 결과 수용 기준

- 수용 판단 상태: Accept Candidate(수용 후보) / Review Required(리뷰 필요) / Need Approval(승인 필요) / Backlog Required(백로그 필요) / Stop Required(중단 필요)
- 완료 조건:
- 필수 검증:
- 보조 검증:
- 승인 상태:
- 수용 불가 조건 확인:
- 후속 연결:
```

Work Order 작성 시점에는 기대 검증을 기록하고, Report 작성 시점에는 실제 수행한 검증과 결과를 기록한다.

## 10. Report 기록 기준

Harness Report에는 최소한 다음 내용을 남긴다.

| 항목 | 설명 |
|---|---|
| 작업 요약 | 수행한 작업과 제외한 작업 |
| 변경 대상 | 변경된 파일, 문서, 상태, 산출물 |
| 완료 조건 대조 | 완료 조건별 충족 여부 |
| 검증 결과 | 실행한 명령, 점검 결과, 실패 여부 |
| 승인 상태 | Approval Pass(승인 통과) 또는 승인 관련 기록 |
| 수용 판단 | Accept Candidate(수용 후보) 또는 다른 전환 상태와 이유 |
| 후속 작업 | Backlog, 추가 Issue, PR, 리뷰 필요 여부 |

Report는 Harness의 판단을 설명하는 근거 문서다. 검증하지 않은 항목을 완료된 것처럼 기록하지 않는다.
