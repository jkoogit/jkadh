## 1. 목적

본 문서는 JKADH에서 Harness와 AI 개발자가 사람 리드의 승인 없이 진행할 수 없는 작업 경계를 정의한다.

사람 승인 경계는 자동화를 막기 위한 규칙이 아니라, 자동화가 넘지 말아야 할 통제선을 명확히 하기 위한 정책이다.

## 2. 적용 범위

본 문서는 다음 작업에 적용한다.

- Issue 기반 작업
- Backlog를 Issue 후보로 승격하는 작업
- Harness가 작성하는 Work Order
- AI 개발자에게 전달되는 작업 지시
- PR, Report, Backlog로 연결되는 후속 흐름

Harness는 실행 전에 본 정책의 승인 항목을 확인해야 한다.

## 3. 기본 원칙

사람 승인 경계의 기본 원칙은 다음과 같다.

- 사람 리드는 최종 승인권과 예외 판단권을 가진다.
- Harness는 승인 필요 여부를 판단하고 근거를 정리한다.
- Harness는 승인 없이 고위험 작업을 진행하지 않는다.
- 승인 필요 항목이 없더라도 확인 결과를 기록한다.
- 승인이 필요한 작업은 Work Order 또는 Report에 승인 요청 근거를 남긴다.
- 승인 경계가 불명확하면 자동 진행하지 않고 사람 리드에게 확인한다.

## 4. 승인 상태

Harness는 작업마다 다음 승인 상태 중 하나를 기록한다.

| 승인 상태 | 한글명 | 의미 | 처리 |
|---|---|---|---|
| Approval Pass | 승인 통과 | 사람 승인 필수 항목이 확인되지 않음 | Work Order에 pass 근거를 기록하고 다음 단계로 진행 |
| Need Approval | 승인 필요 | 사람 승인 필수 항목이 있음 | 승인 요청을 작성하고 사람 리드의 판단을 기다림 |
| Approval Granted | 승인됨 | 사람 리드가 승인함 | 승인 범위 안에서 진행 |
| Approval Rejected | 승인 거절 | 사람 리드가 승인하지 않음 | Stop, Backlog, 또는 범위 재조정 |
| Approval Deferred | 승인 보류 | 지금 결정하지 않음 | Backlog 또는 Report로 연결 |

Approval Pass는 승인 생략이 아니다. 승인 필수 항목을 점검했고 해당 항목이 없었다는 기록이다.

## 5. 승인 필수 조건

다음 조건 중 하나라도 해당하면 사람 리드의 승인이 필요하다.

| 조건 | 설명 |
|---|---|
| 제품 방향 변경 | 서비스 정체성, 비전, 목표, 범위에 영향을 주는 변경 |
| 정책 변경 | 반복 적용되는 운영 규칙 또는 승인 경계를 바꾸는 변경 |
| 핵심 원칙 변경 | Core 또는 상위 기준 문서의 의미를 바꾸는 변경 |
| 비용 한도 변경 | Token Budget, 비용 한도, 사용량 제한을 바꾸는 결정 |
| Secret 접근 | 인증키, 토큰, 비밀번호, 운영 Secret 접근 |
| 배포 | 사용자 또는 운영 환경에 영향을 주는 배포 작업 |
| main 영향 작업 | `main`, `dev`, `stg` 직접 변경 또는 승급 |
| 외부 시스템 쓰기 | GitHub, Notion, 배포 시스템 등 외부 시스템에 쓰기 작업 |
| 자동 승급 | PR 병합, dev/stg/main 승급을 자동으로 진행하는 작업 |
| 대규모 삭제 | 문서, 코드, 설정을 되돌리기 어렵게 삭제하는 작업 |
| 대규모 리팩터링 | 동작 범위가 넓고 영향 분석이 필요한 구조 변경 |
| 코드 변경 수용 | 코드 변경 결과를 사람 리뷰 없이 최종 수용하려는 경우 |
| 검증 근거 부족 | 완료 조건 또는 검증 결과가 부족한 상태에서 결과를 수용하려는 경우 |
| Harness 판단 불확실 | Low-risk 여부를 명확히 판단하기 어려운 경우 |

승인 필수 조건은 비용 또는 속도보다 우선한다.

## 6. Approval Pass 조건

Harness는 다음 조건을 모두 만족할 때 Approval Pass를 기록할 수 있다.

- 승인 필수 조건에 해당하지 않는다.
- 작업 범위와 제외 범위가 명시되어 있다.
- 완료 조건과 검증 방법이 명시되어 있다.
- 작업이 문서, 메타데이터, 인덱스, Backlog 정리처럼 Low-risk 후보에 속한다.
- 외부 시스템 쓰기, 배포, Secret 접근, main 영향이 없다.
- 결과를 검증할 수 있는 근거가 있다.

초기 운영에서 Approval Pass는 주로 다음 작업에 적용한다.

- 문서 링크 수정
- 오탈자 또는 표현 정리
- 공식 문서 인덱스 갱신
- Backlog 상태 갱신
- `git diff --check`처럼 기계적으로 확인 가능한 형식 검증

코드 변경은 초기 운영에서 Approval Pass 대상이 아니다. 코드 변경은 리뷰 또는 사람 승인을 거친다.

## 7. Work Order 승인 확인

Work Order에는 승인 확인 항목을 반드시 포함한다.

최소 형식은 다음과 같다.

```markdown
## 승인 확인

- 승인 상태: Approval Pass(승인 통과) / Need Approval(승인 필요) / Approval Granted(승인됨) / Approval Rejected(승인 거절) / Approval Deferred(승인 보류)
- 확인한 승인 필수 조건:
  - 제품 방향 변경:
  - 정책 변경:
  - Secret 접근:
  - 배포:
  - main 영향:
  - 외부 시스템 쓰기:
  - 자동 승급:
  - 코드 변경 수용:
  - 검증 근거 부족:
- 판단 근거:
- 승인자:
- 승인 범위:
```

승인자가 없는 Approval Pass는 가능하지만, 판단 근거는 반드시 남겨야 한다.

## 8. 승인 요청 기준

Need Approval 상태가 되면 Harness는 사람 리드에게 다음 정보를 제공한다.

- 작업 목적
- 승인 필요한 이유
- 영향 범위
- 선택지
- 권장안
- 승인하지 않을 경우 처리 방향
- 관련 Issue, 문서, PR, Backlog

사람 리드가 승인하면 Harness는 승인 범위 안에서만 진행한다. 승인 범위를 벗어나는 새 요구가 발생하면 다시 승인 확인을 수행한다.

## 9. 승인 없이 중단해야 하는 경우

다음 경우 Harness는 승인 요청 없이도 즉시 Stop 또는 Backlog로 연결할 수 있다.

- 작업 목적이 현재 문서 또는 Issue 범위와 충돌한다.
- Secret 또는 외부 시스템 접근이 필요한데 권한 정보가 없다.
- 검증 방법이 존재하지 않는다.
- 승인 조건이 명확하지만 승인자를 확인할 수 없다.
- 작업을 계속하면 기존 결정이나 정책을 훼손할 가능성이 높다.

중단된 작업은 실패로 처리하지 않는다. 중단 이유와 후속 처리를 Report 또는 Backlog에 남긴다.

## 10. 필수 실행조건 미충족 전환 기준

세션 태그나 Git 상태 변경 게이트의 필수 실행조건이 충족되지 않으면 실행을 계속하지 않고 다음 기준으로 전환한다.

| 전환 상태 | 적용 조건 | 처리 |
|---|---|---|
| `Need Approval` | 작업 범위, 제외 범위, 완료 조건, 승급 여부처럼 사람 리드의 판단이 필요한 경우 | 선택지와 권장안을 보고하고 승인 후 진행한다. |
| `Stop` | 실행하면 기존 정책, 브랜치 상태, Issue 추적성, 검증 기준을 훼손할 가능성이 있는 경우 | 실행을 중단하고 중단 이유와 재개 조건을 보고한다. |
| `사후 보정` | 이미 일부 작업이 진행되었지만 Issue, Backlog, PR 연결, 작업 이력처럼 추적 정보만 보완하면 되는 경우 | 누락 항목을 먼저 보정하고 이후 절차를 재개한다. |
| `Backlog` | 현재 Issue 또는 세션 범위를 벗어나며 독립 판단이 필요한 경우 | Backlog로 분리하고 현재 작업에서는 제외한다. |

전환 판단은 다음 원칙을 따른다.

- Issue 없는 큰 작업은 `사후 보정`으로 계속 진행하지 않고 Issue 생성 또는 `Stop`으로 전환한다.
- 완료 조건이나 검증 방법이 없으면 `Stop` 또는 `Need Approval`로 전환한다.
- 외부 시스템 쓰기, PR 병합, `dev/stg/main` 승급은 승인 범위가 명확할 때만 진행한다.
- 자동 종료 키워드, Issue 종료, `main` 영향 작업은 완료 조건과 승인 범위가 모두 명확해야 한다.
- 사후 보정은 경미한 추적 누락을 회복하는 보조장치이며, 표준 절차를 생략하는 근거로 사용하지 않는다.
