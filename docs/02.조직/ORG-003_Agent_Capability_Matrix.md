## 1. 목적

본 문서는 JKADH에서 작업 성격과 난이도에 따라 어떤 에이전트 역할을 배치할지 판단하는 Capability Matrix를 정의한다.

Capability Matrix는 특정 AI 제품의 성능표가 아니다. Harness가 작업을 접수한 뒤 역할 후보를 고르고, 보강 역할과 검증 필요 수준을 판단하기 위한 운영 기준이다.

## 2. 적용 범위

본 문서는 다음 판단에 적용한다.

- Issue 또는 Work Order의 작업 성격 분류
- 주 담당 에이전트 역할 선정
- 보조 에이전트 또는 리뷰 에이전트 필요 여부 판단
- 단독 수행, 분할 수행, 검토 보강 판단
- 역할 부적합 또는 사람 승인 필요 조건 확인

실제 AI 제품, 모델, 도구별 세부 성능은 본 문서에서 확정하지 않는다. 본 문서는 역할 기준의 배치 원칙을 정의한다.

## 3. Capability 판단 축

Harness는 에이전트 배치 시 다음 축을 확인한다.

| 판단 축 | 한글명 | 의미 |
|---|---|---|
| Domain Fit | 도메인 적합성 | 작업 성격이 해당 역할의 책임과 맞는지 |
| Context Load | 맥락 부하 | 필요한 문서, 코드, 결정 맥락의 양 |
| Execution Risk | 실행 위험도 | 변경 결과가 정책, 코드, 배포, 외부 시스템에 미치는 영향 |
| Verification Need | 검증 필요도 | 완료 주장을 뒷받침할 검증 강도 |
| Autonomy Fit | 자율 수행 적합성 | 사람 확인 없이 어느 정도까지 진행 가능한지 |

Capability는 절대 점수가 아니라 작업 배치 판단을 위한 운영 관점이다.

## 4. 작업 난이도

작업 난이도는 다음 기준으로 구분한다.

| 난이도 | 한글명 | 의미 | 기본 처리 |
|---|---|---|---|
| L1 | 단순 | 범위가 작고 완료 조건이 명확하며 영향 범위가 좁은 작업 | 단일 에이전트 수행 가능 |
| L2 | 보통 | 여러 문서나 파일을 함께 보아야 하고 판단 근거가 필요한 작업 | 주 담당 + 리뷰 또는 QA 보강 |
| L3 | 복합 | 여러 역할 판단, 설계 선택, 리스크 평가, 후속 분리가 필요한 작업 | Work Order 분해 후 역할 분담 |
| L4 | 고위험 | 정책 변경, 코드 수용, 배포, Secret, 외부 시스템 쓰기, main 영향이 있는 작업 | 사람 승인 필요 |

난이도는 작업량만으로 판단하지 않는다. 영향 범위와 검증 가능성이 함께 고려되어야 한다.

## 5. 역할별 Capability Matrix

| 에이전트 역할 | 강점 | 적합 작업 | 부적합 작업 | 검증 보강 |
|---|---|---|---|---|
| CTO | 방향, 정책, 우선순위, 구조 판단 | 비전, 범위, 정책, 승인 경계, 의사결정 후보 정리 | 확정된 구현 세부를 직접 수행하는 작업 | 관련 상위 문서와 충돌 여부 확인 |
| 기획 | 사용자 가치, 요구사항, 범위 정리 | 요구사항, 시나리오, 성공 기준, 제외 범위 정리 | 기술 구현 세부 확정 | CTO 또는 설계 검토 |
| 설계 | 책임 경계, 구조, 인터페이스 설계 | 아키텍처, 상태 모델, 데이터 흐름, API, Work Order 구조화 | 검증 없이 구현을 확정하는 작업 | 구현 가능성 검토와 리뷰 보강 |
| 구현 | 확정된 작업 수행, 코드/문서 반영 | 범위가 명확한 코드 수정, 문서 반영, 스크립트 작성 | 요구사항이 불명확하거나 승인 경계가 걸린 작업 | 테스트, diff, 리뷰 필수 |
| 리뷰 | 결함, 리스크, 누락 확인 | PR 검토, 문서 충돌 확인, 완료 조건 대조 | 제품 방향을 새로 결정하는 작업 | 재현 근거와 파일 위치 제시 |
| QA | 검증 기준, 실패 조건, 테스트 결과 확인 | 테스트 계획, 검증 체크리스트, 미검증 항목 정리 | 설계 방향 단독 결정 | 테스트 결과와 완료 조건 연결 |
| 리서치 | 외부 정보 조사, 대안 비교 | 최신 정보 확인, 제품/기술 비교, 근거 링크 수집 | 출처 없는 결론 확정 | 출처, 날짜, 신뢰도 표시 |
| 하네스 | 작업 접수, 분해, 배정, 통제 | Work Order 작성, 역할 배치, Token/승인/상태 흐름 관리 | 사람 승인 없는 고위험 자동화 | POL-005, POL-006 기준 확인 |
| 운영 | 비용, 안정성, 운영 영향 판단 | Token, 비용, 모니터링, Incident, 운영 KPI | 제품 기능 요구사항 단독 확정 | 운영 지표와 영향 범위 확인 |

하나의 작업에 여러 역할이 필요할 수 있다. 이 경우 Harness는 주 담당 역할과 보강 역할을 분리한다.

## 6. 작업 성격별 기본 배치

| 작업 성격 | 주 담당 | 보강 역할 | 기본 검증 |
|---|---|---|---|
| 비전, 목표, 범위 정리 | CTO | 기획, 설계 | 상위 문서 충돌 여부 |
| 요구사항, 사용자 흐름 정리 | 기획 | CTO, 설계 | 범위와 완료 조건 확인 |
| 설계, 상태 모델, 운영 흐름 | 설계 | CTO, 리뷰 | 책임 경계와 전이 조건 확인 |
| 코드 구현 | 구현 | 리뷰, QA | 테스트, 빌드, diff 확인 |
| 문서 작성 또는 정리 | CTO 또는 설계 | 리뷰 | 링크, 상태, 관련 문서 확인 |
| 정책 수립 | CTO | 리뷰, 운영 | 승인 경계와 반복 적용 가능성 확인 |
| PR 검토 | 리뷰 | QA, 설계 | 결함, 누락, 테스트 공백 확인 |
| 검증 기준 작성 | QA | 구현, 리뷰 | 완료 조건과 검증 방법 연결 |
| 외부 조사 | 리서치 | CTO, 설계 | 출처와 최신성 확인 |
| 운영 정책 | 운영 | CTO, 하네스 | 비용, 권한, 중단 조건 확인 |
| Work Order 작성 | 하네스 | CTO 또는 설계 | 목적, 범위, 제외 범위, 완료 조건 확인 |

주 담당은 산출물을 만든다. 보강 역할은 산출물의 누락, 충돌, 검증 공백을 줄이는 역할이다.

## 7. 배치 판단 규칙

Harness는 다음 순서로 에이전트 역할을 배치한다.

```text
작업 성격 확인
↓
난이도 확인
↓
위험 조건 확인
↓
주 담당 역할 선정
↓
보강 역할 필요 여부 판단
↓
검증 방법 연결
↓
승인 상태 확인
```

배치 판단 규칙은 다음과 같다.

- L1(단순) 작업은 주 담당 1개 역할로 시작할 수 있다.
- L2(보통) 작업은 리뷰 또는 QA 보강을 기본 후보로 둔다.
- L3(복합) 작업은 Work Order를 세부 태스크로 분해한 뒤 역할을 나눈다.
- L4(고위험) 작업은 역할 배치 전에 사람 승인 필요 여부를 확인한다.
- 구현 작업은 초기 운영에서 사람 리뷰 없이 최종 수용하지 않는다.
- 정책, 승인 경계, 비용 한도 변경은 CTO와 사람 리드 판단을 거친다.
- 검증 방법이 없으면 구현 또는 수용 단계로 넘기지 않는다.

## 8. 부적합 판단

다음 조건 중 하나라도 해당하면 Harness는 단독 배치를 피하고 보강 역할, 사람 승인, 또는 Backlog 전환을 검토한다.

- 작업 목적이나 완료 조건이 불명확하다.
- 한 역할의 책임을 넘어서는 설계, 구현, 검증이 섞여 있다.
- Secret, 배포, 외부 시스템 쓰기, main 영향이 있다.
- 결과를 검증할 방법이 없다.
- 작업 범위가 현재 Issue보다 크다.
- 특정 AI 제품 성능에만 의존해야 한다.
- Low-risk 여부를 명확히 판단하기 어렵다.

부적합 판단은 작업 거절이 아니라 배치 조정 신호다.

## 9. Work Order 기록 기준

Work Order에는 역할 배치 판단을 다음 형식으로 남긴다.

```markdown
## 역할 배치

- 작업 성격:
- 난이도: L1(단순) / L2(보통) / L3(복합) / L4(고위험)
- 주 담당 역할:
- 보강 역할:
- 부적합 조건:
- 검증 보강:
- 승인 상태:
- 판단 근거:
```

역할 배치 판단은 산출물보다 앞선 운영 판단이다. 실행 중 작업 성격이 바뀌면 Harness는 역할 배치를 다시 확인한다.
