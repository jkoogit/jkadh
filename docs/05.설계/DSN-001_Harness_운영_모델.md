## 1. 목적

본 문서는 JKADH AI서비스플랫폼에서 Harness가 Issue 기반 작업을 어떻게 운영하는지 정의한다.

Harness는 AI 개발자를 직접 대체하는 실행자가 아니라, 작업을 접수하고, 판단 지점을 통과시키고, 적절한 에이전트 역할로 연결하고, 산출물과 검증 근거를 수집해 다음 흐름으로 넘기는 운영 시스템이다.

본 문서는 Harness의 입력, 출력, 상태 전이, 중단 조건, 승인 조건을 정의한다. 에이전트별 상세 Capability, Token 운영 정책, 사람 승인 세부 정책은 별도 문서에서 다룬다.

## 2. 책임

Harness의 책임은 다음과 같다.

| 책임 | 설명 |
|---|---|
| 작업 접수 | Issue 또는 Backlog 후보를 작업 후보로 읽는다. |
| 범위 확인 | 목적, 작업 범위, 제외 범위, 완료 조건이 충분한지 확인한다. |
| 작업 분석 | 작업 성격, 난이도, 위험도, 권한 영향을 판단한다. |
| 작업 분해 | 작업 분량을 분석하고 필요한 경우 세부 태스크로 나눈다. |
| 역할 연결 | 필요한 에이전트 역할과 AI 개발자 후보를 선정한다. |
| 실행 제약 확인 | 권한 게이트와 Token Budget을 확인한다. |
| 작업 지시 | 선택된 AI 개발자에게 범위와 완료 조건을 전달한다. |
| 결과 수집 | 코드, 문서, 리뷰, 테스트 결과, 보고서를 산출물로 수집한다. |
| 검증 근거 정리 | 완료 조건, 테스트, diff, 링크, 정책 준수 여부를 정리한다. |
| 후속 흐름 결정 | Harness 수용, 사람 승인 요청, PR, Report, Backlog 중 다음 흐름으로 연결한다. |

Harness는 제품 방향, 정책 변경, 고위험 작업 승인, 최종 병합 판단을 대신하지 않는다. 해당 판단은 사람 리드의 책임이다.

## 3. 입력

Harness의 입력은 다음과 같다.

| 입력 | 필수 여부 | 설명 |
|---|---|---|
| Issue | 필수 | 기본 작업 요청 단위다. 목적, 범위, 완료 조건을 포함해야 한다. |
| 관련 문서 | 필수 | 작업 기준이 되는 프로젝트, 정책, 조직, 설계, 운영 문서다. |
| Backlog | 선택 | Issue로 승격할 수 있는 후보 또는 범위 밖 항목이다. |
| PR | 선택 | 기존 작업 결과를 리뷰하거나 후속 처리할 때 사용한다. |
| Review 결과 | 선택 | 결함, 누락, 충돌, 검증 공백을 확인하는 입력이다. |
| Test 결과 | 선택 | 결과 수용 여부를 판단하는 검증 근거다. |
| Token 정보 | 선택 | 작업 지속, 축소, 분할, 재배치 판단에 사용하는 운영 정보다. |

Issue의 목적, 작업 범위, 제외 범위, 완료 조건이 불충분하면 Harness는 작업을 시작하지 않고 보완 요청 또는 Backlog 정리로 연결한다.

## 4. 출력

Harness의 출력은 다음과 같다.

| 출력 | 설명 |
|---|---|
| 작업 판단 | 작업 성격, 난이도, 위험도, 필요한 에이전트 역할에 대한 판단이다. |
| 세부 태스크 목록 | 작업 분량이 큰 경우 분리된 하위 작업과 우선순위다. |
| 작업 지시 | AI 개발자가 수행해야 할 범위, 제외 범위, 완료 조건이다. |
| 실행 상태 | 작업이 어느 상태에 있는지 나타내는 상태값이다. |
| 산출물 묶음 | 코드, 문서, 리뷰, 테스트 결과, 보고서 등 작업 결과다. |
| 검증 근거 | 결과를 수용하거나 보류해야 하는 근거다. |
| 승인 요청 | 사람 리드의 판단이 필요한 조건과 이유다. |
| 후속 흐름 | PR, Report, Backlog, 추가 Issue 후보 중 다음 연결 대상이다. |

출력은 개별 AI 세션에만 남지 않고 Issue, PR, Report, Backlog, 공식 문서 중 하나 이상의 추적 가능한 단위와 연결되어야 한다.

## 5. 상태 모델

Harness는 다음 상태를 사용한다.

| 상태 | 한글명 | 의미 | 다음 상태 |
|---|---|---|---|
| Candidate | 후보 | Issue 또는 Backlog에서 작업 후보를 읽은 상태 | Intake, Backlog |
| Intake | 접수 | 목적, 범위, 완료 조건을 확인하는 상태 | Classify, Need Clarification, Backlog |
| Classify | 분류 | 작업 성격, 난이도, 위험도, 권한 영향을 분석하는 상태 | Decompose, Gate Check, Need Approval, Backlog |
| Decompose | 분해 | 작업 분량을 분석하고 필요한 경우 세부 태스크로 분리하는 상태 | Gate Check, Backlog |
| Gate Check | 게이트 확인 | 권한 게이트와 Token Budget을 확인하는 상태 | Assign, Need Approval, Stop |
| Assign | 배정 | 필요한 에이전트 역할과 AI 개발자 후보를 정하는 상태 | Execute |
| Execute | 실행 | AI 개발자에게 작업을 지시하고 산출물을 받는 상태 | Collect |
| Collect | 수집 | 산출물을 수집하고 누락 여부를 확인하는 상태 | Verify, Backlog |
| Verify | 검증 | 완료 조건과 검증 근거를 확인하는 상태 | Accept Candidate, Review, Need Approval, Backlog |
| Accept Candidate | 수용 후보 | Low-risk 결과를 Harness가 수용 후보로 판단한 상태 | Report, PR |
| Need Approval | 승인 필요 | 사람 리드의 승인이 필요한 상태 | Execute, Report, Stop |
| Review | 리뷰 | 리뷰 또는 QA 보강이 필요한 상태 | Execute, Report, Backlog |
| Report | 보고 | 결과, 근거, 후속 작업을 보고로 정리하는 상태 | Closed |
| Backlog | 백로그 | 현재 범위 밖 항목을 분리한 상태 | Closed |
| Stop | 중단 | 권한, Token, 범위, 검증 문제로 중단한 상태 | Report, Closed |
| Closed | 종료 | 해당 Harness 흐름을 종료한 상태 | - |

상태는 구현 세부가 아니라 운영 추적 단위다. 실제 자동화 전에는 사람이 이 상태를 수동으로 판단하고 기록할 수 있어야 한다.

## 6. 기본 흐름

Harness의 기본 흐름은 다음과 같다.

```text
Candidate(후보)
↓
Intake(접수)
↓
Classify(분류)
↓
Decompose(분해)
↓
Gate Check(게이트 확인)
↓
Assign(배정)
↓
Execute(실행)
↓
Collect(수집)
↓
Verify(검증)
↓
Accept Candidate(수용 후보) / Need Approval(승인 필요) / Review(리뷰) / Backlog(백로그) / Stop(중단)
↓
Report(보고)
↓
Closed(종료)
```

기본 흐름은 Issue Loop를 기준으로 한다. PR 생성 이후에는 PR Review Loop로 연결할 수 있고, 작업 결과와 후속 판단은 Report Loop로 정리한다.

## 7. 판단 지점

Harness는 다음 판단 지점을 가진다.

| 판단 지점 | 질문 | 결과 |
|---|---|---|
| 범위 충분성 | Issue에 목적, 범위, 제외 범위, 완료 조건이 있는가? | 진행, 보완 요청, Backlog |
| 작업 성격 | 문서, 정책, 설계, 구현, 리뷰, QA, 운영, 리서치 중 무엇인가? | 역할 후보 선정 |
| 난이도 | 단독 수행, 분할 수행, 검토 보강 중 무엇이 필요한가? | 배치 방식 결정 |
| 작업 분량 | 하나의 Issue 안에서 수행 가능한가, 세부 태스크로 나누어야 하는가? | 단일 작업, 세부 태스크 분리, Backlog |
| 위험도 | Secret, 배포, main 영향, 외부 시스템 쓰기, 정책 변경이 있는가? | 권한 게이트 또는 승인 요청 |
| Token 가용성 | 현재 작업을 지속할 수 있는 Token Budget이 있는가? | 진행, 축소, 분할, 중단 |
| Capability 적합성 | 후보 AI 개발자가 작업을 수행하기 적합한가? | Assign 또는 후보 재선정 |
| 검증 충분성 | 완료 조건과 검증 근거가 충분한가? | 수용 후보, 리뷰, QA, 승인 요청 |
| 후속 흐름 | 결과를 PR, Report, Backlog 중 어디로 연결할 것인가? | 후속 상태 결정 |

세부 기준이 아직 별도 문서로 정의되지 않은 판단 지점은 Harness 문서에서 임의로 확정하지 않는다. Harness는 해당 판단이 필요한 위치와 필요한 입력을 정의한다.

## 8. 중단 조건

Harness는 다음 조건에서 작업을 중단하거나 보류한다.

| 조건 | 처리 |
|---|---|
| Issue 범위가 불명확함 | 보완 요청 또는 Backlog 정리 |
| 제외 범위가 없음 | 보완 요청 |
| 완료 조건이 없음 | 보완 요청 |
| 작업 분량이 현재 Issue 범위를 초과함 | 세부 태스크 분리 또는 Backlog |
| 권한 게이트를 통과하지 못함 | 사람 승인 요청 또는 Stop |
| Secret, 배포, main 영향 작업이 승인되지 않음 | Stop 또는 Need Approval |
| Token Budget이 부족함 | 축소, 분할, 재배치, Stop |
| AI 개발자 후보가 부적합함 | 후보 재선정 또는 Backlog |
| 산출물이 누락됨 | Review 또는 Execute 재진입 |
| 검증 근거가 부족함 | Review, QA, Need Approval |
| 작업 중 새 요구가 범위를 바꿈 | Backlog 분리 |

중단은 실패가 아니라 운영 통제 지점이다. 중단된 작업은 Report 또는 Backlog로 연결되어야 한다.

## 9. 승인 조건

다음 조건은 사람 리드의 승인을 요구한다.

- 제품 방향 변경
- 정책 변경
- 핵심 원칙 변경
- 비용 한도 변경
- Secret 접근
- 배포
- main 영향 작업
- 외부 시스템 쓰기 작업
- 자동 승급
- 검증 근거가 부족한 결과 수용
- Harness가 Low-risk로 판단하기 어려운 작업

승인 조건이 발생하면 Harness는 판단 근거와 선택지를 정리해 사람 리드에게 전달한다. Harness는 승인 없이 고위험 작업을 진행하지 않는다.

## 10. Low-risk 수용 후보

Harness가 Low-risk 결과를 수용 후보로 판단하려면 다음 조건을 모두 만족해야 한다.

- 작업 범위가 Issue 또는 문서에 명시되어 있다.
- 제외 범위를 침범하지 않았다.
- 권한, Secret, 배포, main 영향, 외부 시스템 쓰기 작업이 아니다.
- 완료 조건을 확인할 수 있다.
- 검증 근거가 남아 있다.
- 관련 산출물이 Issue, PR, Report, Backlog, 공식 문서 중 하나와 연결되어 있다.

이 조건은 최종 수용 정책이 아니라 Harness 운영 모델의 최소 수용 후보 조건이다. 세부 수용 기준은 별도 점검 또는 정책 문서에서 정의한다.

## 11. Backlog 연결

Harness는 다음 경우 Backlog로 연결한다.

- 현재 Issue 범위 밖의 아이디어가 나온 경우
- 선행 문서 또는 결정이 필요한 경우
- 작업은 필요하지만 지금 우선순위가 낮은 경우
- 자동화하기 전에 수동 운영 검증이 필요한 경우
- 결과 수용 기준이나 승인 기준이 아직 불명확한 경우

Backlog는 공식 문서 본문에서 중복 관리하지 않고 [Backlog 미해결 인덱스](../15.로그/backlog/README.md)에서 관리한다.
