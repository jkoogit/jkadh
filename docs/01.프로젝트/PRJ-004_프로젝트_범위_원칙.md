## 1. 목적

본 문서는 JKADH AI서비스플랫폼의 프로젝트 범위를 정할 때 적용할 원칙을 정의한다.

본 문서는 단계별 추진 계획이나 일정표가 아니다. 단계별 구현 방향은 [RDM-001 단계적 추진방향](../07.로드맵/RDM-001_단계적_추진방향.md)에서 다룬다.

## 2. 적용 범위

이 문서는 JKADH 프로젝트의 포함 범위, 제외 범위, 변경 기준, 완료 기준을 판단할 때 적용한다.

개별 단계의 추진 순서와 일정은 로드맵 문서에서 다룬다.

## 3. 범위 정의 기준

JKADH는 다음 기준으로 프로젝트 범위를 정한다.

| 기준 | 설명 |
|---|---|
| 실제 적용 가능성 | PDFowers 같은 실제 제품 개발에 적용할 수 있어야 한다. |
| 추적 가능성 | 작업은 Issue, Branch, PR, Review, Test, Report, Backlog 중 하나 이상의 단위와 연결되어야 한다. |
| 운영 가능성 | 사람이 검토하고 통제할 수 있는 수준에서 먼저 운영되어야 한다. |
| 역할 분리 | 사람 리드, CTO 에이전트, Harness, AI 개발자의 책임이 구분되어야 한다. |
| 승인 경계 | 권한, Secret, 배포, main 영향, 외부 시스템 쓰기 작업은 사람 승인 경계를 우선한다. |
| 점진적 자동화 | 반복 패턴이 검증된 영역만 부분 자동화한다. |
| 비용과 Token 제약 | Token은 작업 지속 가능성을 판단하는 운영 제약으로 사용한다. |

## 4. 포함 원칙

초기 프로젝트 범위에는 다음 성격의 작업을 포함한다.

- AI 개발팀 운영에 직접 필요한 공식 문서와 정책
- Issue 기반 작업 접수, 범위 정의, 완료 조건 관리
- 역할 기반 에이전트 운영 기준
- GitHub 기반 Branch, PR, Review, Test 흐름
- Backlog 분리와 미해결 인덱스 관리
- 사람이 확인 가능한 수동 Harness 운영
- Token Usage와 Token Budget 기록
- Low-risk 작업의 근거 기반 수용 후보 판단
- High-risk 작업의 사람 승인 요청 기준

## 5. 제외 원칙

초기 프로젝트 범위에서는 다음 성격의 작업을 제외한다.

- 사람이 없는 완전 자율 개발 조직
- 모든 AI 도구의 자동 통합
- Harness 내부 자동화 전체 구현
- 검증되지 않은 자동 승인과 자동 승급
- Token 비용 최적화 제품화
- 대시보드 UI 구현
- GitHub/Notion 전체 API 자동화
- AI 모델 성능 비교 서비스
- 범용 챗봇 서비스

제외 항목은 영구 폐기 대상이 아니다. 단, 현재 운영 기준이 검증되기 전까지 공식 범위에 포함하지 않는다.

## 6. 변경 원칙

프로젝트 범위는 진행 과정에서 조정될 수 있다.

범위 변경은 다음 기준을 따른다.

- 새 요구가 현재 문서의 목적을 바꾸면 별도 문서나 Backlog로 분리한다.
- 포함 범위를 넓히기 전 운영 가능성과 승인 경계를 먼저 확인한다.
- 자동화 범위를 넓히기 전 수동 운영에서 반복 패턴이 검증되었는지 확인한다.
- High-risk 작업은 편의성보다 승인과 검증을 우선한다.
- 범위 밖 아이디어는 공식 문서 본문에 목록으로 중복 관리하지 않고 Backlog로 분리한다.

## 7. 완료 기준

PRJ-004의 범위 원칙이 유효하려면 다음 기준을 만족해야 한다.

- 포함할 작업과 제외할 작업을 판단하는 기준을 설명할 수 있다.
- 수동 운영과 자동화 확장의 경계를 설명할 수 있다.
- 사람 승인 경계를 프로젝트 범위 판단에 반영할 수 있다.
- 범위 밖 항목을 Backlog로 분리할 수 있다.
- 단계별 추진 계획과 문서 역할이 구분된다.
