## 1. 목적

본 문서는 JKADH AI서비스플랫폼의 초기 프로젝트 범위를 정의한다.

PRJ-001은 서비스 정체성을, PRJ-002는 현실적인 비전과 목표를, PRJ-003은 서비스 정의를 다룬다. 본 문서는 그 내용을 기준으로 Phase 1부터 Phase 3까지 무엇을 포함하고 무엇을 제외할지 정한다.

본 문서는 구현 상세 범위가 아니라 프로젝트 수준의 포함 범위와 제외 범위를 정의한다.

## 2. 범위 원칙

JKADH의 초기 범위 원칙은 다음과 같다.

- 실제 제품 개발에 적용 가능한 운영 체계를 먼저 만든다.
- 문서, Issue, Branch, PR, Review, Test, Report가 연결되는 흐름을 우선한다.
- Harness는 처음부터 완전 자동화하지 않고 수동/반자동 운영으로 시작한다.
- 반복 패턴이 검증된 영역만 부분 자동화한다.
- 권한, Secret, 배포, main 영향, 외부 시스템 쓰기 작업은 사람 승인 경계를 우선한다.
- Token은 비용 계산보다 작업 지속 가능성을 판단하는 운영 제약으로 먼저 사용한다.
- 현재 범위를 벗어난 아이디어는 공식 문서 본문에 목록으로 관리하지 않고 Backlog로 분리한다.

## 3. 포함 범위

초기 프로젝트 범위에는 다음 항목을 포함한다.

| 범위 | 설명 |
|---|---|
| 공식 문서 체계 | 프로젝트, 조직, 정책, 설계, 운영, 백로그 문서를 관리하는 기준을 유지한다. |
| Issue 기반 작업 운영 | Issue를 작업 요청서, 범위 잠금 장치, 검토 기준, 추적 단위로 사용한다. |
| 역할 기반 에이전트 운영 | CTO, 기획, 설계, 구현, 리뷰, QA, 리서치, 하네스, 운영 역할을 구분한다. |
| GitHub 기반 개발 흐름 | Branch, Commit, PR, Review, Test 흐름을 작업 단위와 연결한다. |
| Backlog 관리 | 현재 범위 밖의 항목을 Backlog로 분리하고 미해결 인덱스에서 관리한다. |
| 수동 Harness 운영 | 사람이 확인 가능한 방식으로 작업 분석, 배치 판단, 결과 수집, 보고 연결을 수행한다. |
| Token 기록과 판단 | Token Usage와 Token Budget을 기록하고 작업 지속, 축소, 분할, 재배치 판단에 사용한다. |
| 근거 기반 결과 수용 | 완료 조건과 검증 근거가 명확한 Low-risk 작업은 Harness 수용 후보로 다룬다. |
| 사람 승인 경계 | High-risk 작업은 사람 리드의 승인과 검증을 요구한다. |
| PDFowers 적용 | 실제 제품 개발 사례로 PDFowers 작업 흐름에 우선 적용한다. |

## 4. 제외 범위

초기 프로젝트 범위에서 다음 항목은 제외한다.

| 제외 항목 | 제외 이유 |
|---|---|
| 완전 자율 개발 조직 | 사람 리드의 최종 승인권과 예외 판단권을 유지해야 한다. |
| 모든 AI 도구의 자동 통합 | 초기에는 운영 기준을 먼저 검증하고, 통합은 반복 패턴 확인 후 확장한다. |
| Harness 내부 자동화 전체 구현 | 수동/반자동 운영 모델이 안정된 뒤 자동화한다. |
| Token 비용 최적화 제품화 | 초기에는 비용 최적화보다 작업 지속 가능성 판단을 우선한다. |
| 대시보드 UI 구현 | 운영 지표와 보고 항목이 안정된 뒤 UI를 검토한다. |
| GitHub/Notion 전체 API 자동화 | 초기에는 사람이 검토 가능한 연동 수준을 우선한다. |
| AI 모델 성능 비교 서비스 | JKADH는 모델 비교가 아니라 AI 개발팀 운영을 목표로 한다. |
| 범용 챗봇 서비스 | JKADH는 대화형 응답 서비스가 아니라 작업 운영 서비스다. |

## 5. 단계별 범위

JKADH의 단계별 범위는 다음과 같다.

| 단계 | 포함 범위 | 제외 범위 |
|---|---|---|
| Phase 1 | 공식 문서 체계, 역할/정책, Backlog, 프로젝트 정의 | 자동화 구현, 대시보드 UI, 외부 API 자동 연동 |
| Phase 2 | GitHub Issue/Branch/PR 기반 수동 Harness 운영, 리뷰와 보고 흐름 | 완전 자동 배치, 자동 승인, 자동 승급 |
| Phase 3 | Token 기록, 상태 추적, Low-risk 판단 일부 자동화 | High-risk 자동 처리, 비용 최적화 제품화 |
| Phase 4 | 다중 AI 개발팀 운영 플랫폼화 방향 검토 | 본 문서의 초기 완료 범위에는 포함하지 않음 |

## 6. 운영 단위

초기 운영 단위는 Issue다.

Issue는 다음 정보를 포함해야 한다.

- 목적
- 배경
- 작업 범위
- 제외 범위
- 완료 조건
- 관련 문서
- 관련 브랜치
- 후속 작업 후보

Issue 진행 중 발생한 새 아이디어는 현재 Issue에 즉시 섞지 않는다. 현재 작업 완료에 필요한 항목이 아니면 Backlog로 분리한다.

## 7. 수용 가능한 자동화 범위

초기 자동화는 다음 범위 안에서만 수용한다.

| 자동화 범위 | 기준 |
|---|---|
| 문서 링크 확인 | 로컬 Markdown 링크처럼 검증 가능한 항목에 한정한다. |
| 공백과 형식 검사 | `git diff --check`처럼 결과가 명확한 검증에 한정한다. |
| 작업 상태 요약 | Issue, PR, Backlog, 문서 변경 상태를 사람이 확인할 수 있게 요약한다. |
| Low-risk 결과 수용 후보 판단 | 완료 조건과 검증 근거가 명확한 작업에 한정한다. |
| Backlog 분류 보조 | 현재 범위 밖 항목을 Ready, Blocked, Deferred 등으로 제안한다. |

자동화 결과는 사람이 검토할 수 있어야 한다. 승인 경계를 넘는 작업은 자동화 대상이 아니다.

## 8. 완료 기준

PRJ-004의 프로젝트 범위가 유효하려면 다음 기준을 만족해야 한다.

- Phase 1부터 Phase 3까지의 포함 범위와 제외 범위를 설명할 수 있다.
- Issue 기반 작업 운영이 프로젝트의 기본 단위로 고정된다.
- 수동 Harness와 부분 자동화의 경계를 설명할 수 있다.
- High-risk 작업을 초기 자동화 범위에서 제외한다.
- PDFowers 같은 실제 제품 개발에 적용할 수 있는 범위로 유지한다.
- 범위 밖 항목은 Backlog로 분리할 수 있다.
