## 1. 목적

본 문서는 JKADH가 만들려는 서비스의 정체성과 기본 운영 모델을 정의한다.

JKADH는 여러 AI 서비스를 단순 도구로 사용하는 수준을 넘어, 역할과 책임을 가진 AI 개발자들을 운영 가능한 개발팀으로 구성하기 위한 플랫폼을 지향한다.

본 문서는 후속 프로젝트 문서, 조직 문서, 운영 문서, 구현 문서의 상위 기준으로 사용한다.

## 2. 문제 정의

AI 개발 도구는 빠르게 발전하고 있지만, 여러 AI를 함께 사용하면 다음 문제가 발생한다.

- 작업 맥락이 AI별 세션과 도구 안에 흩어진다.
- 어떤 AI가 어떤 역할과 책임으로 작업했는지 추적하기 어렵다.
- 토큰과 비용 한도를 넘기기 전에 작업을 조정하기 어렵다.
- 브랜치, Pull Request, Review, 테스트, 보고 흐름이 일관되지 않다.
- 권한, Secret, 배포, main 변경처럼 위험도가 높은 작업을 AI별로 통제하기 어렵다.
- 여러 AI가 낸 결과를 누가 모으고, 검토하고, 다음 작업으로 연결할지 불명확하다.

JKADH는 이 문제를 AI 자체의 성능 문제가 아니라 개발 조직 운영 문제로 본다.

## 3. 서비스 정체성

JKADH는 소규모 AI 개발 조직을 위한 AI 개발팀 운영 플랫폼이다.

이 플랫폼에서 Codex, Claude, Gemini, Copilot, Manus 같은 AI 서비스는 단순 보조 도구가 아니라 역할을 가진 개발자로 취급한다. 각 AI 개발자는 할당된 역할, 권한, 작업 범위 안에서 결과물을 만든다.

Harness는 AI 개발팀을 운영하는 실행 시스템이다. Harness는 Loop, Token, 작업 배분, 상태 추적, GitHub/Notion 연동, 결과 수집, 다음 작업 지시를 담당한다.

사람 리드는 최종 권한자다. 제품 방향, 우선순위, 승인, 예외 판단, 비용 한도, 고위험 작업 허용 여부는 사람 리드가 결정한다.

CTO 에이전트는 Harness와 독립된 판단 보조자다. CTO 에이전트는 기술 방향, 아키텍처, 정책, 리스크, 의사결정 초안을 제안하지만 최종 결정권을 갖지 않는다.

## 4. 대상 사용자

1차 대상 사용자는 소규모 AI 개발 조직의 운영자 또는 리드다.

초기 실제 사용자는 jk이며, PDFowers를 포함한 실제 제품 개발에 먼저 적용한다. 이후 AI 기반 제품을 개발하는 소규모 팀, 스타트업, 개발 조직으로 확장할 수 있어야 한다.

대상 사용자는 다음 문제를 가진다.

- 여러 AI 개발자를 역할별로 배치하고 싶다.
- AI별 작업 품질, 비용, 토큰 사용량을 관리하고 싶다.
- GitHub 기반의 검증 가능한 개발 흐름을 강제하고 싶다.
- Notion 기반으로 대시보드, 보고서, 회의록, 리뷰, 의사결정을 관리하고 싶다.
- AI가 자율적으로 움직이더라도 사람 리드의 승인권과 통제권은 유지하고 싶다.

## 5. 핵심 가치 제안

JKADH의 핵심 가치는 AI를 많이 쓰는 것이 아니라 AI 개발팀을 통제 가능하게 운영하는 것이다.

| 가치 | 설명 |
|---|---|
| 역할 기반 운영 | AI별 강점과 역할에 따라 작업을 분리한다. |
| 통제 가능한 자동화 | Harness가 반복 실행을 담당하되 권한과 승인 경계를 지킨다. |
| 비용과 토큰 관리 | 토큰 사용량과 잔여량을 기준으로 작업 지속 여부를 판단한다. |
| 검증 가능한 개발 흐름 | Issue, Branch, PR, Review, Test, Report를 추적 가능하게 연결한다. |
| 사람 중심 의사결정 | AI는 판단을 제안하고 실행을 보조하지만 최종 결정은 사람 리드가 한다. |

| 구성원 | 책임 | 비고 |

| CTO 에이전트 | 기술 판단, 아키텍처, 정책, 리스크, 의사결정 초안 제안 | Harness와 독립성을 유지한다. |
| Harness | Loop, Token, 작업 배분, 상태 추적, GitHub/Notion 연동, 결과 수집 | 실행 운영 시스템이다. |
| AI 개발자 | 할당된 역할과 작업 범위 안에서 산출물 작성 | Codex, Claude, Gemini, Copilot, Manus 등이다. |

## 7. 운영 원칙

JKADH의 운영 원칙은 다음과 같다.

- AI는 자기 역할과 할당 범위만 수행한다.
- Harness는 결과를 수집하고 다음 작업을 지시한다.
- Harness는 실행과 통제를 담당하지만 제품/기술 방향의 최종 판단자가 아니다.
- CTO 에이전트는 판단을 제안하고, 사람 리드는 승인하며, Harness는 승인된 방향을 실행 가능한 흐름으로 변환한다.
- 모든 작업은 Issue, Branch, PR, Review, Test, Report 중 하나 이상의 추적 가능한 단위와 연결한다.
- 현재 작업 범위를 벗어난 아이디어는 즉시 포함하지 않고 Backlog로 분리한다.
- 고위험 작업은 자동 실행보다 승인과 검증을 우선한다.

## 8. Loop 모델

기본 Loop 단위는 Issue다.

Issue Loop는 하나의 GitHub Issue를 기준으로 분석, 작업 분배, 수행, 리뷰, 테스트, PR, 보고까지 이어지는 반복 흐름이다.

```text
Issue 분석
↓
위험도와 권한 확인
↓
AI 개발자 후보 선정
↓
토큰/비용 가용성 확인
↓
작업 지시
↓
결과 수집
↓
리뷰와 테스트
↓
PR 또는 Backlog 정리
↓
보고와 다음 작업 판단
```

보조 Loop는 다음과 같다.

| Loop | 목적 |
|---|---|
| Backlog Selection Loop | Notion/GitHub Backlog에서 다음 Issue 후보를 고른다. |
| PR Review Loop | PR 생성 이후 리뷰, 테스트, 수정, 병합 조건을 반복 관리한다. |
| Report Loop | 작업 결과, 토큰 사용량, 결정 사항, 후속 작업을 보고서로 정리한다. |

Loop는 AI 개발자가 아니라 Harness가 담당한다. AI 개발자는 Loop 안에서 할당된 작업만 수행한다.

## 9. AI 배치 기준

Harness는 AI 개발자를 배치할 때 다음 순서로 판단한다.

| 단계 | 기준 | 설명 |
|---|---|---|
| 1 | 권한과 위험도 | Secret, 배포, main 영향, 외부 시스템 변경 등 고위험 작업 가능 여부를 먼저 확인한다. |
| 2 | 토큰/비용 가용성 | 실행 가능한 잔여 토큰과 비용 한도를 확인한다. |
| 3 | Capability 적합도 | 코드 작성, 리뷰, 리서치, 설계, 문서화 등 작업 성격에 맞는 AI를 고른다. |
| 4 | 작업 난이도 | 난이도와 컨텍스트 규모에 따라 단독 수행, 분할 수행, 검토 보강을 결정한다. |

정책상 최상위 게이트는 권한과 위험도다. 비용이 낮아도 권한상 허용되지 않는 작업은 수행하지 않는다.

실행상 최상위 물리 제약은 토큰/비용 가용성이다. 정책상 허용되는 후보가 있어도 가용 토큰이 없으면 작업을 중단, 축소, 대체, 재배치한다.

## 10. 권한과 안전장치

JKADH는 AI의 장점을 활용하되 저장소와 운영 권한을 분리한다.

기본 안전장치는 다음과 같다.

- AI별 또는 작업별 브랜치를 분리한다.
- 모든 변경은 Pull Request를 거친다.
- main 직접 push는 금지한다.
- GitHub Actions 테스트를 필수 게이트로 둔다.
- Secret 접근은 기본 차단하고 필요한 경우 최소 범위로 제한한다.
- 배포, main 영향, 외부 시스템 쓰기 작업은 승인 조건을 둔다.
- 고위험 작업은 Harness가 자동 실행하지 않고 사람 리드의 확인을 요구한다.

## 11. 외부 시스템 연계

JKADH는 GitHub와 Notion을 기본 운영 시스템으로 사용한다.

| 시스템 | 역할 |
|---|---|
| GitHub | Issue, Branch, Pull Request, Review, Actions, Release 관리 |
| Notion | Dashboard, 보고서, 회의록, 리뷰 기록, 의사결정 기록 관리 |
| Harness | GitHub와 Notion 사이에서 작업 상태, 토큰, 보고, 다음 지시를 연결 |

초기에는 사람이 확인 가능한 수동/반자동 운영으로 시작하고, 반복 패턴이 안정되면 Harness 자동화 범위를 넓힌다.

## 12. 성공 기준

PRJ-001의 서비스 정체성이 구현으로 이어지려면 다음 기준을 만족해야 한다.

- Issue 단위 작업 흐름이 추적 가능하다.
- AI별 역할, 작업 범위, 결과물이 구분된다.
- AI별 토큰 사용량과 잔여량을 확인할 수 있다.
- 권한과 위험도에 따라 작업 배치가 제한된다.
- PR, 리뷰, 테스트, 보고가 누락 없이 연결된다.
- PDFowers 같은 실제 제품 개발에 적용할 수 있다.
- 사람 리드가 최종 승인권과 예외 판단권을 유지한다.

## 13. 비목표

JKADH는 다음을 목표로 하지 않는다.

- 일반 챗봇 플랫폼
- 단순 IDE 플러그인
- AI 모델 성능 비교 서비스
- 사람이 없는 완전 자율 개발 조직
- 모든 AI에게 동일 권한을 주는 자동화 시스템
- 제품/기술 의사결정을 AI가 최종 대체하는 시스템

| Loop | 작업을 선택하고, 수행하고, 검증하고, 보고하고, 다음 작업으로 연결하는 반복 흐름 |

| Token Budget | AI별 또는 작업별로 사용할 수 있는 토큰/비용 한도 |
| Capability | AI가 특정 작업을 수행하는 데 필요한 능력과 적합도 |
