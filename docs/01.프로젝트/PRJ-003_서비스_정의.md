## 1. 목적

본 문서는 JKADH AI서비스플랫폼이 사용자에게 제공하는 서비스를 정의한다.

PRJ-001이 서비스 정체성을 정의하고 PRJ-002가 현실적인 비전과 목표를 정의한다면, 본 문서는 JKADH를 사용하는 사람이 어떤 입력을 넣고, Harness가 어떤 운영 서비스를 제공하며, 어떤 결과를 받을 수 있는지를 설명한다.

본 문서는 구현 설계가 아니라 서비스 정의 문서다. Harness 내부 구조, 에이전트 배치 알고리즘, Token 계산식, API 연동 방식은 별도 설계와 운영 문서에서 다룬다.

## 2. 서비스 한 문장 정의

JKADH는 Issue 기반 작업을 입력으로 받아 Harness가 에이전트 배치, 토큰과 권한 통제, 결과 수집, 검증 근거 정리, 보고 연결을 수행하도록 돕는 AI 개발팀 운영 서비스다.

## 3. 대상 사용자

JKADH의 1차 대상 사용자는 소규모 AI 개발 조직의 사람 리드다.

초기 사용자는 jk이며, PDFowers 같은 실제 제품 개발에 JKADH 운영 방식을 먼저 적용한다. 이후 대상은 여러 AI 도구를 함께 사용하면서도 작업 맥락, 비용, 권한, 검증 흐름을 통제하고 싶은 소규모 개발팀으로 확장할 수 있다.

대상 사용자는 다음 요구를 가진다.

- 필요한 작업을 적절한 AI에 배치하고, 구현 과정의 작업 맥락과 산출물 기준을 일관되게 유지하고 싶다.
- 작업 성격과 난이도에 따라 적절한 AI 개발자를 활용하고 싶다.
- 토큰과 비용 사용량을 기준으로 작업 지속 여부를 판단하고 싶다.
- 권한과 위험도가 높은 작업을 자동화 흐름에서 분리하고 싶다.
- AI가 만든 결과를 검증 근거와 함께 보고받고 싶다.
- 최종 승인권과 예외 판단권은 사람 리드에게 남기고 싶다.

## 4. 서비스 입력

JKADH는 다음 입력을 서비스 운영의 출발점으로 사용한다.

| 입력 | 설명 |
|---|---|
| Issue | 가장 기본적인 작업 요청 단위다. 목적, 작업 범위, 제외 범위, 완료 조건을 포함한다. |
| Backlog | 아직 Issue로 승격하지 않은 작업 후보와 검토 항목이다. |
| 공식 문서 | 정책, 조직, 프로젝트, 설계, 운영 기준을 제공하는 실행 기준이다. |
| Pull Request | 작업 결과를 검토하고 병합 여부를 판단하는 단위다. |
| Review 결과 | 결함, 누락, 충돌, 검증 공백을 확인하는 입력이다. |
| Test 결과 | 결과 수용 또는 추가 작업 판단에 사용하는 검증 근거다. |
| Token 정보 | 작업 지속, 축소, 분할, 재배치 여부를 판단하는 운영 정보다. |

초기 서비스는 사람이 작성한 Issue와 문서를 중심으로 운영한다. 자동 입력 수집은 반복 패턴이 확인된 뒤 확장한다.

## 5. 핵심 서비스 기능

JKADH는 다음 서비스를 제공한다.

| 서비스 | 설명 |
|---|---|
| 작업 접수 | Issue 또는 Backlog를 기준으로 작업 요청을 접수한다. |
| 작업 분석 | 작업 성격, 난이도, 위험도, 권한 영향을 확인한다. |
| 에이전트 역할 판단 | 작업에 필요한 CTO, 기획, 설계, 구현, 리뷰, QA, 리서치, 하네스, 운영 역할을 구분한다. |
| AI 개발자 배치 | 역할과 작업 성격에 맞는 AI 개발자 후보를 선정한다. |
| Token 통제 | Token Budget과 Token Usage를 기준으로 작업 지속 가능성을 판단한다. |
| 권한 게이트 | Secret, 배포, main 영향, 외부 시스템 쓰기 같은 고위험 조건을 확인한다. |
| 작업 지시 | 선택된 AI 개발자에게 작업 범위와 완료 조건을 전달한다. |
| 결과 수집 | 코드, 문서, 리뷰, 테스트 결과, 보고서를 산출물로 수집한다. |
| 검증 근거 정리 | 완료 조건, 테스트, diff, 링크, 정책 준수 여부를 확인 가능한 근거로 정리한다. |
| 결과 수용 또는 승인 요청 | Low-risk 결과는 근거 기반으로 수용하고, High-risk 결과는 사람 리드에게 승인 요청한다. |
| 보고 연결 | 작업 결과와 후속 판단을 PR, Report, Backlog 중 적절한 흐름으로 연결한다. |

## 6. 서비스 흐름

JKADH의 기본 서비스 흐름은 Issue Loop를 중심으로 한다.

```text
Issue 또는 Backlog 후보 확인
↓
작업 범위와 완료 조건 확인
↓
작업 성격, 난이도, 위험도 분석
↓
필요 에이전트 역할과 AI 개발자 후보 선정
↓
권한 게이트와 Token Budget 확인
↓
작업 지시
↓
산출물 수집
↓
검증 근거 확인
↓
Harness 수용, 사람 승인 요청, PR, Report, Backlog 중 후속 흐름 결정
```

이 흐름에서 Harness는 실행 운영 시스템이다. Harness는 작업을 직접 최종 결정하지 않고, 승인된 기준에 따라 반복 판단과 연결을 수행한다.

## 7. 결과물

JKADH가 사용자에게 제공하는 결과물은 다음과 같다.

| 결과물 | 설명 |
|---|---|
| 작업 판단 | 작업 성격, 난이도, 위험도, 필요한 에이전트 역할에 대한 판단이다. |
| 작업 지시 | AI 개발자가 수행해야 할 범위와 완료 조건이다. |
| 산출물 | 코드, 문서, 리뷰, 테스트 결과, 보고서 등 검증 가능한 작업 결과다. |
| 검증 근거 | 결과를 받아들일 수 있는 이유 또는 추가 검토가 필요한 이유다. |
| 승인 요청 | 사람 리드의 판단이 필요한 조건과 근거다. |
| PR 또는 Report | 작업 결과를 GitHub 또는 운영 보고 흐름으로 연결한 산출물이다. |
| Backlog 갱신 | 이번 작업 범위 밖의 항목을 별도 후보로 분리한 기록이다. |

결과물은 AI별 세션 안에 머무르지 않고 공식 문서, Issue, PR, Report, Backlog 중 하나 이상의 추적 가능한 단위와 연결되어야 한다.

## 8. 사람 리드와 Harness의 경계

사람 리드는 JKADH의 최종 권한자다.

사람 리드는 다음을 결정한다.

- 제품 방향
- 우선순위
- 비용 한도
- 정책 변경
- 예외 판단
- 고위험 작업 허용 여부
- 최종 승인과 병합 판단

Harness는 다음을 수행한다.

- 작업 접수와 상태 추적
- 작업 성격과 위험도 분석
- 에이전트 역할과 AI 개발자 후보 선정
- Token Budget과 Token Usage 확인
- 작업 지시와 결과 수집
- 검증 근거 정리
- Low-risk 결과 수용 후보 판단
- High-risk 작업의 사람 승인 요청

Harness는 사람 리드의 최종 판단을 대체하지 않는다. 다만 반복적이고 근거가 명확한 Low-risk 작업은 승인된 기준 안에서 수용할 수 있다.

## 9. 서비스 단계

JKADH 서비스는 단계적으로 제공한다.

| 단계 | 서비스 수준 |
|---|---|
| Phase 1 | 공식 문서와 정책을 기준으로 사람이 확인 가능한 운영 체계를 제공한다. |
| Phase 2 | GitHub Issue, Branch, PR, Review, Test, Report를 연결하는 수동 Harness 서비스를 제공한다. |
| Phase 3 | Token, 상태, 에이전트 배치 근거를 기록하고 Low-risk 판단 일부를 Harness가 수행한다. |
| Phase 4 | 다중 AI 개발팀 운영 플랫폼으로 확장한다. 단, 본 문서에서는 장기 방향으로만 둔다. |

초기 서비스는 완전 자동화를 목표로 하지 않는다. 먼저 사람이 검토 가능한 수동/반자동 운영을 안정화하고, 반복 패턴이 검증된 영역부터 자동화한다.

## 10. 비목표

JKADH 서비스는 다음을 목표로 하지 않는다.

- 일반 챗봇 서비스
- 단순 AI 모델 비교 서비스
- 단순 IDE 플러그인
- 사람이 없는 완전 자율 개발 조직
- 모든 AI에게 동일 권한을 부여하는 자동화 시스템
- 검증 근거 없이 AI 결과를 자동 수용하는 시스템
- GitHub, Notion, AI 도구 자체를 대체하는 서비스

JKADH는 외부 도구를 대체하기보다 GitHub, Notion, AI 개발자를 운영 흐름 안에서 연결하고 통제하는 서비스다.

## 11. 성공 기준

PRJ-003의 서비스 정의가 유효하려면 다음 기준을 만족해야 한다.

- 사용자가 JKADH에 어떤 입력을 제공하는지 설명할 수 있다.
- Harness가 제공하는 핵심 서비스 기능을 설명할 수 있다.
- 사람 리드와 Harness의 책임 경계를 설명할 수 있다.
- Low-risk 결과 수용과 High-risk 승인 요청의 방향을 구분할 수 있다.
- 서비스 결과물이 Issue, PR, Report, Backlog와 연결된다.
- PRJ-001 서비스 정체성과 PRJ-002 비전과 목표에 모순되지 않는다.
