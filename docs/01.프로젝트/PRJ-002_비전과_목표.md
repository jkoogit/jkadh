## 1. 목적

본 문서는 JKADH AI서비스플랫폼의 현실적인 비전과 단계별 목표를 정의한다.

JKADH는 여러 AI를 개별 도구로 사용하는 데서 멈추지 않고, Issue 기반 작업을 중심으로 AI 개발자를 배치하고, Harness가 토큰, 권한, 위험도, 검증 근거를 통제하여 사람이 신뢰하고 승인할 수 있는 개발 결과로 연결하는 것을 목표로 한다.

본 문서는 장기적인 제품 구상보다 가까운 단계에서 검증 가능한 운영 목표를 우선 정의한다.

## 2. 배경

AI 개발 도구를 실제 제품 개발에 활용하면 다음 문제가 반복된다.

- 작업 지시와 결과가 AI별 세션에 흩어진다.
- 역할, 책임, 작업 범위가 명확하지 않으면 결과를 비교하고 이어받기 어렵다.
- 작업 난이도와 성격에 맞는 AI를 고르는 기준이 없다.
- 토큰과 비용 사용량이 작업 지속 여부와 연결되지 않는다.
- 단순 작업과 고위험 작업이 같은 승인 절차로 다뤄진다.
- 결과를 받아들일 수 있는 근거가 문서, 테스트, 리뷰, PR 흐름과 연결되지 않는다.

JKADH는 이 문제를 AI 성능 문제가 아니라 AI 개발팀 운영 문제로 본다. 따라서 목표는 더 강한 AI 하나를 고르는 것이 아니라, 여러 AI 개발자를 역할과 통제 기준 안에서 운영할 수 있는 체계를 만드는 것이다.

## 3. 현실적 비전

JKADH의 현실적 비전은 소규모 AI 개발 조직이 Issue를 등록하면 Harness가 작업 성격, 난이도, 위험도, 토큰 가용성, 필요한 에이전트 역할을 판단하고, 적절한 AI 개발자에게 작업을 배치하며, 결과를 검증 가능한 근거와 함께 사람 리드에게 연결하는 운영 플랫폼을 만드는 것이다.

사람 리드는 최종 승인권과 예외 판단권을 유지한다. Harness는 반복적인 운영 판단과 결과 수집을 담당하되, 제품 방향과 고위험 의사결정을 대신하지 않는다.

낮은 위험도의 명확한 작업은 Harness가 완료 조건과 검증 근거를 기준으로 수용할 수 있어야 한다. 반대로 권한, Secret, 배포, main 영향, 외부 시스템 쓰기, 정책 변경처럼 위험도가 높은 작업은 사람 리드의 승인과 검증을 요구해야 한다.

## 4. 목표 운영상

JKADH가 달성하려는 목표 운영상은 다음과 같다.

```text
Issue 등록
↓
작업 성격, 난이도, 위험도 분석
↓
필요 에이전트 역할과 후보 AI 개발자 선정
↓
권한 게이트와 Token Budget 확인
↓
작업 지시와 산출물 수집
↓
검증 근거 확인
↓
Harness 수용 또는 사람 리드 승인 요청
↓
PR, Report, Backlog 중 적절한 후속 흐름으로 연결
```

이 운영상에서 Issue는 작업 요청서이자 범위 잠금 장치다. AI 개발자는 할당된 역할과 작업 범위 안에서 산출물을 만든다. Harness는 작업을 이어 붙이고, 상태를 추적하고, 다음 판단으로 연결한다.

## 5. 단계별 목표

JKADH는 다음 단계를 현실 목표로 삼는다.

| 단계 | 목표 | 완료 기준 |
|---|---|---|
| Phase 1 | 문서와 운영 기준 정립 | 세션이 바뀌어도 서비스 정체성, 역할, 정책, 작업 흐름을 같은 기준으로 확인할 수 있다. |
| Phase 2 | GitHub 기반 수동 Harness 운영 | Issue, Branch, PR, Review, Test, Report가 하나의 작업 흐름으로 연결된다. |
| Phase 3 | Token과 상태 기반 부분 자동화 | Token Usage, Token Budget, 작업 상태, 에이전트 배치 근거를 기록하고 일부 Low-risk 판단을 Harness가 수행한다. |
| Phase 4 | 다중 AI 개발팀 운영 플랫폼화 | 장기 방향으로만 둔다. 본 문서의 현실 목표 완료 조건에는 포함하지 않는다. |

본 문서는 Phase 1부터 Phase 3까지를 현실 목표로 다룬다. Phase 4는 후속 로드맵에서 별도로 정의한다.

## 6. 핵심 목표

JKADH의 핵심 목표는 다음과 같다.

| 목표 | 설명 |
|---|---|
| Issue 기반 운영 | 모든 주요 작업은 Issue 또는 Backlog에서 시작하고 추적 가능한 산출물로 끝난다. |
| 역할 기반 AI 배치 | 작업 성격에 따라 CTO, 기획, 설계, 구현, 리뷰, QA, 리서치, 하네스, 운영 역할을 구분한다. |
| 난이도와 위험도 분리 | 쉬운 작업이라도 권한 위험이 높으면 사람 승인을 요구한다. 어려운 작업이라도 권한 위험이 낮으면 검토 보강으로 진행할 수 있다. |
| Token 기반 통제 | Token Budget과 Token Usage를 작업 지속, 중단, 축소, 분할, 재배치 판단에 사용한다. |
| 근거 기반 수용 | Harness는 테스트, diff, 링크, 완료 조건, 정책 준수 같은 근거를 확인한 뒤 Low-risk 결과를 수용할 수 있다. |
| 사람 중심 승인 | 사람 리드는 최종 승인권, 예외 판단권, 비용 한도 결정권, 고위험 작업 허용권을 유지한다. |

## 7. 통제 원칙

JKADH는 다음 통제 원칙을 따른다.

- 권한과 위험도는 Token과 비용보다 먼저 판단한다.
- Harness는 작업을 자동화하되 승인 경계를 넘지 않는다.
- AI 개발자는 자기 역할과 할당 범위를 벗어난 판단을 공식 결정으로 만들지 않는다.
- 결과 수용은 명확한 완료 조건과 검증 근거에 연결되어야 한다.
- 검증 근거가 부족하면 Harness는 결과를 수용하지 않고 리뷰, QA, 사람 승인, Backlog 중 하나로 연결한다.
- 정책 변경, 핵심 원칙 변경, 배포, Secret 접근, main 영향 작업은 사람 리드의 확인을 요구한다.
- 현재 작업 범위 밖의 아이디어는 즉시 포함하지 않고 Backlog로 분리한다.

## 8. 성공 기준

PRJ-002의 목표가 달성되려면 다음 기준을 만족해야 한다.

- Issue 등록 후 필요한 에이전트 역할을 설명할 수 있다.
- 작업 성격, 난이도, 위험도, Token Budget을 기준으로 작업 진행 여부를 판단할 수 있다.
- AI 개발자별 작업 범위와 산출물이 구분된다.
- Low-risk 작업을 Harness가 수용할 수 있는 근거 기준이 정의된다.
- High-risk 작업을 사람 리드에게 올리는 조건이 정의된다.
- PR, 리뷰, 테스트, 보고, Backlog가 누락 없이 연결된다.
- PDFowers 같은 실제 제품 개발에 적용할 수 있다.

## 9. 비목표

본 문서는 다음을 목표로 하지 않는다.

- Harness 내부 상태 머신 상세 설계
- 에이전트 배치 알고리즘 상세
- Token Budget 계산식
- GitHub/Notion API 연동 상세
- 자동 승인 체크리스트의 세부 항목
- UI와 대시보드 설계
- 사람이 없는 완전 자율 개발 조직

위 항목은 후속 설계, 운영, 정책, 로드맵 문서에서 별도로 다룬다.
