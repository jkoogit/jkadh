## 1. 목적

본 문서는 JKADH AI서비스플랫폼에서 반복 사용하는 공통 용어를 정의한다.

용어 정의를 개별 프로젝트, 설계, 운영 문서마다 반복하지 않고 한곳에서 관리하여 문서 간 의미 충돌을 줄이는 것이 목적이다.

## 2. 적용 범위

이 문서는 JKADH의 AI서비스플랫폼 공통 용어에 적용한다.

다음 문서에서 이 용어사전을 참조할 수 있다.

- 프로젝트 정체성, 비전, 범위 문서
- 조직과 역할 문서
- Harness, Loop, Token 관련 설계 문서
- 운영, 점검, 보고 문서

개별 제품에만 적용되는 도메인 용어는 해당 제품 문서에서 별도로 정의할 수 있다.

## 3. 용어 관리 원칙

- 공통 용어는 본 문서에서 먼저 정의한다.
- 개별 문서는 같은 용어를 재정의하지 않고 본 문서를 참조한다.
- 같은 용어가 문맥에 따라 다른 의미로 쓰이면 용어를 분리하거나 한정자를 붙인다.
- 용어 변경이 기존 문서 의미를 바꾸는 경우 관련 문서의 영향 범위를 함께 확인한다.

## 4. 공통 용어

| 용어 | 정의 | 비고 |
|---|---|---|
| AI서비스플랫폼 | 여러 AI 개발자와 운영 시스템을 묶어 제품 개발을 통제 가능하게 수행하는 플랫폼 | JKADH가 지향하는 상위 서비스 정체성 |
| AI 개발자 | Codex, Claude, Gemini, Copilot, Manus처럼 역할과 범위 안에서 개발 작업을 수행하는 AI 서비스 | 단순 도구가 아니라 역할 기반 수행자로 취급한다. |
| 사람 리드 | 목표, 우선순위, 승인, 예외 판단, 비용 한도, 제품 방향의 최종 결정권을 가진 사람 | 초기 역할은 jk가 수행한다. |
| CTO 에이전트 | 기술 방향, 아키텍처, 정책, 리스크, 의사결정 초안을 제안하는 독립 에이전트 | 최종 결정권자는 아니다. |
| Harness | AI 개발팀의 Loop, Token, 작업 배분, 상태 추적, 외부 시스템 연동을 담당하는 실행 운영 시스템 | CTO 에이전트와 독립성을 유지한다. |
| Loop | 작업을 선택하고, 수행하고, 검증하고, 보고하고, 다음 작업으로 연결하는 반복 흐름 | Harness가 담당한다. |
| Issue Loop | 하나의 Issue를 기준으로 작업 분배부터 PR 또는 보고까지 이어지는 기본 Loop | JKADH의 기본 작업 단위 |
| PR Review Loop | PR 이후 리뷰, 테스트, 수정, 병합 조건을 반복 확인하는 보조 Loop | 품질 게이트에 해당한다. |
| Backlog Selection Loop | Backlog에서 다음 작업 후보를 고르고 Issue로 전환하는 보조 Loop | 작업 선정 흐름에 해당한다. |
| Report Loop | 작업 결과, 토큰 사용량, 결정 사항, 후속 작업을 보고서로 정리하는 보조 Loop | Notion 연계 대상 |
| Token Budget | AI별 또는 작업별로 사용할 수 있는 토큰/비용 한도 | 실행 가능성을 결정하는 물리적 제약 |
| Capability | AI가 특정 작업을 수행하는 데 필요한 능력과 적합도 | 역할, 모델 특성, 작업 난이도와 함께 판단한다. |
| 권한 게이트 | Secret, 배포, main 영향, 외부 시스템 변경 같은 위험 요소를 기준으로 작업 허용 여부를 판단하는 기준 | 비용보다 우선하는 정책 게이트 |
| 작업 범위 | AI 개발자가 수행해야 하는 구체 작업과 제외해야 하는 내용의 경계 | Issue 또는 문서에서 명시한다. |
| 산출물 | AI 개발자가 작업 후 제출하는 코드, 문서, 리뷰, 테스트 결과, 보고서 | 검증 가능한 단위여야 한다. |

## 5. 제품별 용어와의 관계

본 문서는 AI서비스플랫폼 공통 용어를 관리한다.

PDFowers 같은 개별 제품의 도메인 용어는 해당 제품의 프로젝트, 기획, 설계 문서에서 별도로 관리할 수 있다. 단, Harness, Loop, Token Budget, AI 개발자처럼 플랫폼 공통 운영에 속하는 용어는 본 문서를 우선한다.
