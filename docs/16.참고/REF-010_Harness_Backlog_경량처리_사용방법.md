## 1. 목적

본 문서는 JKADH Harness에서 Backlog를 태스크보다 가벼운 절차로 등록, 반영, 승급, 승격하는 기준을 정리한다.

Backlog는 지금 실행하지 않지만 추적해야 하는 후속 검토, 작업 후보, 보류 항목이다. Backlog 자체는 태스크가 아니지만, Backlog 문서를 생성하거나 원격 repo에 반영하는 행위는 추적 가능한 작업이다.

## 2. 적용 범위

이 문서는 다음 범위에 적용한다.

- Backlog 문서 생성과 미해결 인덱스 갱신
- Backlog-only 문서 변경의 PR 반영과 브랜치 승급
- Backlog를 GitHub Issue로 전환하는 승격
- `#백로그추가`, `#백로그반영`, `#백로그승급`, `#백로그승격` 계열 태그의 책임 경계
- `#백로그추가반영승급`, `#백로그반영승급` 같은 조합 명령의 게이트 기준

다음 항목은 본 문서에서 구현하지 않는다.

- 실제 Harness CLI 코드 변경
- GitHub 자동 PR 생성, 머지, 승급 구현
- 기존 HCP alias 파서 변경
- 개별 Backlog의 Issue 승격 실행
- 타겟서비스 운영구조 본문 설계

## 3. 용어 기준

Backlog 경량처리에서 `승급`과 `승격`은 다음처럼 구분한다.

| 용어 | 의미 | 계층 |
|---|---|---|
| 승급 | GitHub 작업 결과를 원격 주요 브랜치인 `dev`, `stg`, `main`에 반영하는 것 | Git, PR, branch |
| 승격 | 콘텐츠를 GitHub Issue로 전환해 실행 작업 대상으로 구분하는 것 | Backlog, Issue, work item |

따라서 `백로그승급`은 Backlog 문서 변경 PR을 주요 브랜치에 반영하는 절차이고, `백로그승격`은 Backlog 항목을 Issue로 전환하는 절차다.

## 4. Backlog와 Task의 구분

Backlog와 Task는 다음 기준으로 구분한다.

| 구분 | Backlog | Task |
|---|---|---|
| 목적 | 나중에 볼 후보나 보류 항목을 기록한다. | 지금 실행할 작업을 수행한다. |
| 실행 여부 | 즉시 실행하지 않는다. | 실행과 검증을 전제로 한다. |
| 필수 식별자 | Backlog ID | GitHub Issue 또는 WorkOrder |
| 산출물 | Backlog 문서, 인덱스, 상태 이력 | 코드, 문서, 테스트 결과, PR |
| 절차 무게 | 경량 절차 | 세션/태스크 절차 |
| Issue 생성 | 승격 시에만 생성한다. | 태스크 시작 시 생성하거나 기존 Issue를 사용한다. |

Backlog 등록 시점은 태스크 중으로 제한하지 않는다. Backlog는 콘텐츠 기준으로 판단한다. 태스크 진행 중 나온 범위 밖 항목은 task-scoped Backlog이고, 태스크 없이 세션 논의 중 나온 후속 검토 항목은 session-scoped Backlog다.

## 5. Backlog 액션

Backlog 경량처리는 다음 액션으로 나눈다.

| 액션 | 태그 후보 | 책임 | 수행하지 않는 일 |
|---|---|---|---|
| 추가 | `#백로그추가` | Backlog 문서 생성 또는 수정, 미해결 인덱스 갱신, 로컬 검증 | PR 생성, 머지, 브랜치 승급, Issue 생성 |
| 반영 | `#백로그반영` | Backlog-only 변경을 커밋하고 PR을 생성한다. | `dev/stg/main` 승급, Issue 생성 |
| 승급 | `#백로그승급` | Backlog PR을 `dev`에 머지하고 `stg/main`까지 반영한다. | Backlog를 Issue로 전환 |
| 승격 | `#백로그승격` | Backlog를 GitHub Issue로 전환하고 Backlog 상태를 `Issue Linked`로 바꾼다. | PR/브랜치 승급 |

실사용 기본 흐름은 다음 세 가지를 우선한다.

```text
#백로그추가
#백로그추가반영승급
#백로그승격
```

이미 로컬에 추가된 Backlog를 원격 주요 브랜치까지 반영해야 하면 `#백로그반영승급`을 사용할 수 있다.

## 6. 조합 명령

조합 명령은 여러 액션을 한 번에 시도하는 단축 명령이다. 조합 명령은 무조건 끝까지 실행하는 명령이 아니라, 각 단계 게이트를 순서대로 통과할 때만 다음 단계로 진행한다.

| 조합 명령 | 의미 | 적합한 경우 |
|---|---|---|
| `#백로그반영승급` | 이미 작성된 Backlog 변경을 PR로 반영하고 주요 브랜치에 승급한다. | Backlog-only 변경이 로컬에 있고 내용이 확정된 경우 |
| `#백로그추가반영승급` | Backlog를 추가한 뒤 같은 흐름에서 PR 반영과 브랜치 승급까지 시도한다. | 논의 중 나온 메모 수준의 후속 검토 항목을 바로 공식 기록으로 남기는 경우 |

조합 명령은 다음 상황에서 중단하고 피드백한다.

- Backlog 내용이 단순 후보가 아니라 즉시 실행 태스크에 가깝다.
- 정책 의미 변경, 코드 변경, DB migration, workflow 변경이 섞여 있다.
- Secret, 배포, 외부 시스템 쓰기, main 직접 영향 같은 승인 조건이 있다.
- Backlog-only PR 조건을 만족하지 않는다.
- PR 생성, 머지, 승급 상태를 확인할 수 없다.

## 7. 단계별 게이트

### 7.1 추가 게이트

`#백로그추가`와 `#백로그추가반영승급`은 먼저 다음 조건을 확인한다.

| 확인 항목 | 통과 기준 |
|---|---|
| Backlog 적합성 | 지금 실행하지 않을 후속 검토, 작업 후보, 보류 항목이다. |
| 최소 필드 | 제목, 상태, 유형, 처리시점, 우선순위, 출처가 있다. |
| 처리 기준 | Issue 승격 또는 Resolved 전환 기준을 약식으로라도 포함한다. |
| 인덱스 | `docs/15.로그/backlog/README.md` 미해결 인덱스에 반영된다. |
| 출처 | task-scoped 또는 session-scoped 출처가 남는다. |

### 7.2 반영 게이트

`#백로그반영`, `#백로그반영승급`, `#백로그추가반영승급`은 다음 조건을 확인한다.

| 확인 항목 | 통과 기준 |
|---|---|
| 변경 범위 | `docs/15.로그/backlog/**`와 필요한 인덱스 변경만 포함한다. |
| 제외 파일 | `.hcp/`, `.env`, 코드, migration, workflow, lockfile 변경을 포함하지 않는다. |
| 문서 검증 | `git diff --check`를 통과한다. |
| PR 성격 | PR 제목과 본문에 Backlog-only 문서 변경임을 명시할 수 있다. |
| 연결 근거 | 관련 Issue가 없으면 Backlog-only 변경 사유를 PR 본문에 남긴다. |

### 7.3 승급 게이트

`#백로그승급`, `#백로그반영승급`, `#백로그추가반영승급`은 다음 조건을 확인한다.

| 확인 항목 | 통과 기준 |
|---|---|
| PR 상태 | Backlog-only PR이 생성되어 있고 머지 가능하다. |
| dev 반영 | PR이 `dev`에 병합된다. |
| stg/main 반영 | `dev` 병합 커밋을 `stg`, `main`에 fast-forward 반영할 수 있다. |
| 정합성 | 원격 `dev`, `stg`, `main` 커밋이 최종적으로 일치한다. |
| 보안 확인 | 승급 대상에 Backlog 외 변경이나 runtime 상태 파일이 섞이지 않는다. |

### 7.4 승격 게이트

`#백로그승격`은 다음 조건을 확인한다.

| 확인 항목 | 통과 기준 |
|---|---|
| 작업화 적합성 | Backlog가 실제 작업 Issue로 전환할 만큼 목적과 범위가 명확하다. |
| 범위 | 작업 범위, 제외 범위, 완료 조건, 검증 방법을 Issue 본문으로 만들 수 있다. |
| 우선순위 | 지금 또는 다음 작업 후보로 선정할 근거가 있다. |
| 상태 변경 | Backlog 상태를 `Issue Linked`로 변경하고 미해결 인덱스와 연결 이력을 갱신한다. |

## 8. 중단 피드백 기준

Backlog 조합 명령은 단계별 게이트를 통과하지 못하면 실행을 멈추고 중단 사유와 다음 선택지를 보고한다.

예시는 다음과 같다.

```text
중단: 추가반영승급 부적절
사유: 내용이 단순 메모가 아니라 운영 절차 변경이다.
권장: #백로그추가까지만 수행하고 별도 Issue 승격 여부를 검토한다.
```

```text
중단: 반영 게이트 실패
사유: Backlog 변경 외에 packages/harness-cli 코드 변경이 포함되어 있다.
권장: 변경 범위를 분리한 뒤 #백로그반영승급을 다시 실행한다.
```

```text
중단: 승급 게이트 실패
사유: 원격 dev/stg/main 정합성을 확인하지 못했다.
권장: PR은 생성하지 않고 로컬 Backlog 등록 상태로 유지한다.
```

중단은 실패가 아니라 절차 통제 결과다. Harness는 가능한 한 적게 멈추되, 멈출 때는 어느 액션에서 멈췄는지와 재시도 가능한 태그를 함께 보고한다.

## 9. 상태 해석

Backlog 경량처리의 상태는 다음처럼 해석한다.

| 상태 | 의미 | 다음 후보 |
|---|---|---|
| local registered | Backlog 문서와 인덱스가 로컬에 작성됐다. | `#백로그반영승급` |
| reflected | Backlog 변경 PR이 생성됐다. | `#백로그승급` |
| promoted | Backlog 변경이 `dev/stg/main`에 반영됐다. | 필요 시 `#백로그승격` |
| issue linked | Backlog가 GitHub Issue로 승격됐다. | `#태스크시작` |
| resolved | 해결 문서나 작업 결과와 연결되어 미해결 인덱스에서 제거할 수 있다. | 세션정리 또는 Backlog 해결 처리 |

예를 들어 `BLG-028`처럼 세션 논의 중 작성된 Backlog는 session-scoped Backlog이며, 문서와 인덱스가 로컬에만 있으면 `local registered` 상태다. 이를 원격 주요 브랜치까지 반영하려면 `#백로그반영승급` 또는 후속 구현에서 정의한 동등 절차를 사용한다.
