## 1. 목적

이 문서는 세션 진행 중 업무가 커지거나 방향이 넓어질 때 현재 태스크에 포함할지, 같은 Issue의 다음 태스크로 나눌지, 다음 세션으로 넘길지, Backlog로 보류할지 판단하는 기준을 정리한다.

범위 관리는 작업을 막기 위한 절차가 아니다. 진행 중 발견한 일을 어떤 단위로 검증, 리뷰, 승급, 회고할지 정하는 운영 기준이다.

## 2. 기본 원칙

세션과 태스크는 다음 단위를 기준으로 나눈다.

| 단위 | 의미 |
|---|---|
| 세션 | 하나의 작업 맥락과 Issue를 중심으로 진행, 정리, 회고하는 운영 단위 |
| 태스크 | 세션 안에서 하나의 PR, 검증 결과, 승급 단위로 닫을 수 있는 실행 단위 |
| Backlog | 지금 실행하지 않지만 추적해야 하는 검토, 개선, 보류 항목 |

작업 범위는 처음 계획과 달라질 수 있다. 다만 범위가 바뀌면 다음 항목을 먼저 확인한다.

- 기존 완료조건을 그대로 만족할 수 있는가.
- 검증 방법이 크게 바뀌지 않는가.
- 보안, 권한, 배포, DB, 외부 서비스 영향이 새로 생기지 않는가.
- PR 리뷰와 롤백 단위가 여전히 명확한가.
- Issue의 목적을 벗어나지 않는가.

## 3. 판단 구분

### 3.1 현재 태스크에 포함

다음 조건을 모두 만족하면 현재 태스크에 포함할 수 있다.

| 기준 | 포함 가능 조건 |
|---|---|
| 목적 | 기존 태스크 목적을 직접 완성하거나 설명을 보강한다. |
| 변경 범위 | 같은 문서, 같은 모듈, 같은 검증 흐름 안에 있다. |
| 검증 | 기존 검증 방법으로 충분하거나 아주 작은 보조 검증만 추가된다. |
| 위험 | 권한, 보안, 배포, DB, 외부 쓰기 영향이 새로 생기지 않는다. |
| 리뷰 | PR 하나에서 변경 의도를 쉽게 읽을 수 있다. |
| 롤백 | 함께 되돌려도 문제가 없다. |

예시는 다음과 같다.

- REF 문서 작성 중 같은 주제의 판단 기준 표를 추가한다.
- CLI 기능 구현 중 동일 명령의 안내 메시지와 테스트를 함께 보강한다.
- 문서 링크 누락처럼 완료조건 충족에 필요한 작은 수정을 반영한다.

### 3.2 같은 Issue의 다음 태스크로 분리

Issue 목적에는 포함되지만 별도 PR과 검증 단위가 더 적합하면 같은 Issue의 다음 태스크로 분리한다.

| 기준 | 분리 조건 |
|---|---|
| 목적 | 같은 Issue의 목표를 이어가지만 현재 태스크 완료조건을 넘어선다. |
| 변경 범위 | 문서에서 구현으로, 구현에서 운영 절차로 단계가 바뀐다. |
| 검증 | 검증 방법이 늘어나거나 테스트 범위가 달라진다. |
| 리뷰 | 현재 PR에 함께 넣으면 리뷰 초점이 흐려진다. |
| 운영 | 먼저 닫고 승급한 뒤 이어서 진행해도 업무 흐름이 끊기지 않는다. |

예시는 다음과 같다.

- 먼저 Backlog 절차를 REF로 정리하고, 다음 태스크에서 Harness CLI 구현을 진행한다.
- 최소 구현을 승급한 뒤, 세션 큐 조회나 대상 선택 규칙을 추가로 정의한다.
- 같은 Issue에서 정책 문서와 코드 변경을 순차 태스크로 나눈다.

### 3.3 다음 세션 또는 새 Issue로 분리

다음 조건 중 하나라도 강하게 해당하면 다음 세션 또는 새 Issue로 분리한다.

| 기준 | 분리 조건 |
|---|---|
| 목적 변경 | 현재 Issue의 문제 정의를 벗어난다. |
| 보안 | Secret, 인증, 권한, 접근 제어, 민감정보 처리 기준이 바뀐다. |
| 배포 | GitHub Actions, 운영 배포, main 직접 영향, 서비스 공개 범위가 바뀐다. |
| DB | migration 실행, reset, 운영 데이터 영향, checksum 수정이 포함된다. |
| 외부 영향 | 외부 서비스 쓰기, 결제, 메일, 알림, 원격 리소스 변경이 포함된다. |
| 검증 변화 | 로컬 문서 검증에서 통합 테스트, 실환경 검증, 운영 점검으로 바뀐다. |
| 롤백 | 기존 작업과 함께 되돌리기 어렵거나 장애 대응 단위가 다르다. |
| 크기 | PR이 커져서 리뷰와 회고가 한 문장으로 설명되지 않는다. |

보안, 권한, 배포, DB, 외부 서비스 영향은 단순 구현 범위보다 우선해서 본다. 같은 주제처럼 보여도 운영 위험이 달라지면 분리하는 편이 좋다.

### 3.4 Backlog로 보류

지금 실행하지 않아도 되는 검토 항목은 Backlog로 보류한다.

| 기준 | 보류 조건 |
|---|---|
| 긴급도 | 현재 태스크 완료에 필요하지 않다. |
| 정보 | 추가 판단, 정책 합의, 선행 설계가 필요하다. |
| 우선순위 | 중요한 아이디어지만 현재 Issue보다 낮다. |
| 성격 | 구현보다 검토, 운영 방향, 나중에 볼 개선안에 가깝다. |
| 공유 | 세션 중에는 메모로 충분하고 정리 때 문서화해도 된다. |

Backlog는 태스크가 아니다. 다만 Backlog를 문서로 만들거나 원격 주요 브랜치에 승급하는 행위는 추적 가능한 작업이므로 별도 태스크 또는 경량 Backlog 절차로 처리한다.

## 4. 범위 확장 체크 기준

태스크 진행 중 새로운 요청이나 아이디어가 나오면 다음 순서로 판단한다.

1. 현재 Issue 목적에 포함되는가.
2. 현재 태스크 완료조건을 바꾸지 않고 처리 가능한가.
3. 검증 방법이 기존 검증 안에 들어오는가.
4. 보안, 권한, 배포, DB, 외부 쓰기 영향이 새로 생기는가.
5. PR 리뷰와 롤백 단위가 여전히 자연스러운가.
6. 지금 처리하지 않으면 다음 작업이 막히는가.

판단 결과는 다음처럼 보고한다.

```text
판단: 같은 Issue 다음 태스크 권장
근거: Issue #111의 목적에는 포함되지만 구현 변경과 테스트가 필요해 현재 문서 태스크 완료조건을 넘어선다.
제안: 현재 태스크를 REF 문서로 닫고, 다음 태스크에서 Harness CLI 구현 범위를 새로 명시한다.
```

```text
판단: 현재 태스크 포함 가능
근거: 같은 REF 문서 안의 운영 기준 보강이며 검증 방법이 git diff --check와 링크 확인으로 유지된다.
제안: 완료조건 문구를 훼손하지 않는 선에서 문서에 함께 반영한다.
```

## 5. 메모성 내용 보관 아이디어

세션 중 나온 운영 판단, 회고 후보, 정책 아이디어는 매번 문서 PR로 만들면 절차가 무거워질 수 있다. 향후 Harness는 이런 메모를 HCP 세션 상태에 임시 보관하고, 세션정리 때 필요한 항목만 문서화하는 방식을 사용할 수 있다.

권장 흐름은 다음과 같다.

1. 세션 중 메모성 내용은 HCP 상태에 `sessionNotes` 또는 유사 구조로 저장한다.
2. 메모에는 제목, 본문, 출처 태스크, 추천 문서화 대상, 우선순위를 남긴다.
3. 세션정리 때 메모를 검토해 REF, Backlog, 회고, 다음 세션 인계 중 하나로 분류한다.
4. 문서화가 필요한 항목만 세션정리 산출물 PR에 함께 포함한다.
5. 단순 참고로 충분한 항목은 HCP 세션 상태에만 남기고 별도 문서로 승급하지 않는다.

이 문서는 운영 아이디어만 기록한다. HCP 메모 기능 구현, 세션정리 자동 문서화 구현, Backlog 반영승급 구현은 이 태스크 범위에 포함하지 않는다.

## 6. 운영 효과

범위를 나누는 기준을 두면 다음 이점이 있다.

- 완료조건이 흐려지지 않아 태스크를 닫기 쉽다.
- 검증 결과가 어떤 변경을 증명하는지 명확해진다.
- PR 리뷰가 목적별로 작아져 누락과 오해가 줄어든다.
- 문제가 생겼을 때 되돌릴 단위가 분명해진다.
- HCP 세션 상태, Issue, PR, 회고가 같은 작업 단위를 가리킨다.
- 다음 프롬프트나 다음 세션에서 이전 맥락을 재구성하기 쉽다.

반대로 너무 잘게 나누면 진행이 느려진다. 따라서 작은 문서 보강, 같은 명령의 테스트 보강, 완료조건 충족을 위한 직접 수정은 현재 태스크에 포함하고, 위험이나 검증 단위가 바뀌는 순간 분리한다.
