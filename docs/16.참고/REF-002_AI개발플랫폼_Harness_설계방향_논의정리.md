## 1. 목적

본 문서는 AI개발플랫폼 Harness 운영구조를 정리하는 과정에서 논의된 방향, 제외된 선택지, 아직 결정되지 않은 항목을 참고자료로 남긴다.

설계 문서인 DSN-003은 최종 운영구조를 간결하게 정의한다. 본 문서는 그 결정이 어떻게 도출되었는지 추적할 수 있게 하는 보조 문서다.

## 2. 논의 배경

초기 논의는 Harness 운영데이터 API 구현에서 출발했다. Work Order, Token Usage, 검증 결과, Report를 저장하려면 백엔드 API가 필요한지 검토했다.

검토 과정에서 JKADH가 실제 서비스 프로젝트 자체가 아니라 AI 개발을 지원하는 플랫폼이라는 점이 명확해졌다. 따라서 서비스 프로젝트의 기능 구현과 JKADH 플랫폼 자체의 Harness, 에이전트, 정책, 스킬, 계정 관리 작업을 구분해야 한다는 결론이 나왔다.

또한 2주 안에 실제 서비스 프로젝트 개발을 시작하고 싶다는 일정 제약이 있었다. 이 제약 때문에 완전한 오케스트레이션 엔진이나 백엔드 API 서버보다, 먼저 통제 가능한 한 턴 실행 사이클을 만드는 방향이 우선되었다.

## 3. 최종 결정안 요약

최종 결정안은 다음과 같다.

- JKADH 플랫폼 repo와 서비스 프로젝트 repo는 별도 repo로 관리한다.
- JKADH는 서비스 프로젝트 코드를 직접 포함하지 않는다.
- Harness는 서비스 프로젝트를 ProjectProfile로 참조한다.
- 작업은 `platform`, `service`, `integration`으로 구분한다.
- 초기 Harness는 실행 엔진이 아니라 Control Plane으로 둔다.
- Control Plane은 WorkOrder, AgentRole, ProviderProfile, PermissionProfile, Gate, Report를 관리한다.
- 첫 구현 후보는 TypeScript CLI와 JSON 저장이다.
- 백엔드 API 서버, 다중 에이전트 자동 오케스트레이션, 토큰 자동 수집은 후속 단계로 둔다.
- 첫 샘플 서비스 프로젝트는 특정 설계 본문에 고정하지 않고, 샘플 코드나 실행 예시에서만 사용할 수 있다.

## 4. 결정안 도출 흐름

처음에는 Harness 운영데이터 API를 구현하는 방향이 검토되었다. 이때 FastAPI, Fastify, NestJS, Hono 같은 백엔드 스택이 후보로 나왔다.

이후 서비스 프로젝트가 별도 repo로 존재한다면 JKADH에 상시 백엔드가 반드시 필요한지 다시 검토했다. 토큰 관리도 단순 기록이면 파일이나 SQLite로 시작할 수 있고, 여러 에이전트와 프로젝트가 공유하는 운영 시스템이 될 때 API가 필요하다는 판단이 나왔다.

다음으로 에이전트 오케스트레이션을 체계적이고 통제 가능하게 만들 필요가 논의되었다. LangGraph, AutoGen, CrewAI, Temporal 같은 실행 엔진 후보가 거론되었지만, 2주 안에 서비스 프로젝트를 시작해야 하는 조건에서는 실행 엔진 도입이 과하다고 판단했다.

따라서 초기 목표를 자동 실행 엔진이 아니라 Control Plane으로 낮췄다. Control Plane은 AI가 직접 모든 일을 자동 수행하는 구조가 아니라, 요청을 WorkOrder로 고정하고, 역할과 권한을 확인하고, feature branch에서 작업하고, 검증과 Report를 남기는 통제 구조다.

마지막으로 실제 서비스 프로젝트 작업과 JKADH 플랫폼 작업이 섞이지 않도록 작업 유형을 분리했다. 이 구분이 있어야 서비스 프로젝트 개발 중 플랫폼 정책이나 Harness 구현이 같은 Issue와 PR에 섞이는 문제를 줄일 수 있다.

## 5. 검토했으나 제외한 방향

| 방향 | 제외 또는 보류 사유 |
|---|---|
| 백엔드 API 서버부터 구현 | 현재 단계에서는 공유 상태, 대시보드, 외부 자동 수집 요구가 확정되지 않았다. 2주 안 서비스 프로젝트 시작 목표에 비해 초기 비용이 크다. |
| FastAPI 기반 운영데이터 API | AI가 다루기 쉽고 빠르지만, TypeScript 기반 플랫폼 도구와 서비스 프로젝트 연동을 고려하면 첫 Harness 구현으로 확정하기에는 이르다. |
| NestJS 기반 백엔드 | 구조와 타입 안정성은 좋지만 MVP에는 무겁다. Control Plane 검증 전에 프레임워크 구조가 먼저 고정될 수 있다. |
| Hono 또는 Fastify API | TypeScript 백엔드로 적합하지만, API 서버 자체가 필요한 시점이 아직 아니다. CLI와 JSON 저장으로 먼저 검증하기로 했다. |
| 완전 자동 에이전트 오케스트레이터 직접 구현 | 통제권은 크지만 구현 범위가 크고 서비스 프로젝트 시작을 지연시킨다. |
| LangGraph, AutoGen, CrewAI 즉시 도입 | 에이전트 실행 구조를 빨리 얻을 수 있지만 프레임워크 철학에 종속된다. JKADH의 태그, 승인, 승급 정책과 연결하는 어댑터가 먼저 필요하다. |
| Temporal 같은 워크플로 엔진 즉시 도입 | 장기 실행과 재시도에는 강하지만 AI 에이전트 모델은 별도 설계가 필요하다. 초기 운영 부담이 크다. |
| PDFowers를 설계 본문에 고정 | 설계 문서는 특정 서비스 프로젝트에 종속되지 않아야 한다. PDFowers는 샘플 후보나 실행 예시에서만 사용한다. |

## 6. 결정되지 않은 항목

다음 항목은 아직 결정되지 않았다.

| 항목 | 미결정 사유 | 다음 검토 방향 |
|---|---|---|
| Harness가 서비스 repo를 clone/fetch까지 자동 수행할지 | 로컬 경로 참조만으로 첫 사이클을 돌 수 있는지 먼저 확인해야 한다. | ProjectProfile에 `local_path`와 `repo_full_name`을 함께 두고 실제 사용성을 검증한다. |
| TypeScript CLI의 패키지 구조 | 아직 코드 구현이 시작되지 않았다. | `packages/harness-cli` 구조를 후보로 두고 첫 구현 시 확정한다. |
| JSON 저장과 SQLite 저장의 경계 | 한 턴 사이클에는 JSON이 충분할 수 있지만 조회와 집계가 늘면 SQLite가 필요하다. | WorkOrder와 Report가 누적되는 시점에 SQLite 전환 기준을 정의한다. |
| AI Provider/Profile 세부 스키마 | Provider별 실행 방식, 계정, 도구 권한이 다르다. | Codex 중심 기본 Profile을 먼저 정의하고 OpenAI API, Claude, Gemini, Copilot은 후보로 둔다. |
| AI 계정과 CredentialRef 운영 | 실제 키 저장 위치와 계정 권한 정책이 확정되지 않았다. | 키 원문은 저장하지 않고 `env:` 또는 `managed:` 참조만 남기는 원칙부터 적용한다. |
| 역할별 허용 Provider | 역할과 Provider를 직접 결합하면 모델 교체가 어렵다. | AgentRole과 ProviderProfile을 분리하고 역할별 허용 후보를 별도 매트릭스로 정리한다. |
| 토큰 사용량 관리 | 수동 기록, API 사용량 조회, Provider별 과금 데이터 수집 방식이 다르다. | 초기에는 수동 기록과 Report 연결을 우선하고 자동 수집은 보류한다. |
| 첫 샘플 서비스 프로젝트의 첫 기능 | 샘플 후보는 있으나 첫 기능 범위는 아직 정하지 않았다. | 작고 테스트 가능한 기능으로 제한해 한 턴 사이클 검증에 사용한다. |
| PR 병합과 Issue 종료 자동화 수준 | 최근 실행에서 자연어 주문 과해석 문제가 있었다. | 명시 태그와 요청 해석 게이트를 유지하고, 자동화는 Control Plane 검증 후 확대한다. |

## 7. 다음 판단 기준

다음 단계는 문서 완성도가 아니라 실제 한 턴 실행 가능성으로 판단한다.

우선 확인할 질문은 다음과 같다.

- WorkOrder 하나로 서비스 프로젝트 작업 범위가 흔들리지 않는가?
- ProjectProfile만으로 별도 repo 서비스 프로젝트를 안정적으로 참조할 수 있는가?
- AgentRole, ProviderProfile, PermissionProfile이 실제 실행 전에 필요한 제약을 충분히 표현하는가?
- feature branch에서 작은 기능을 구현하고 검증, Report, PR까지 이어갈 수 있는가?
- 플랫폼 작업과 서비스 프로젝트 작업이 같은 PR에 섞이지 않는가?

이 질문에 답하기 위해 다음 실행 후보는 Harness CLI 전체 구현이 아니라, 샘플 서비스 프로젝트를 대상으로 한 작은 한 턴 사이클 검증이다.
