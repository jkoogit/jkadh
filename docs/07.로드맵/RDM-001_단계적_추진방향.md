## 1. 목적

본 문서는 JKADH AI서비스플랫폼의 단계적 추진방향을 정의한다.

본 문서는 고정 일정표가 아니다. 진행 과정에서 검증 결과, 실제 제품 적용 결과, 사람 리드의 판단에 따라 조정될 수 있는 추진 방향 문서다.

프로젝트 범위를 판단하는 고정 원칙은 [PRJ-004 프로젝트 범위 원칙](../01.프로젝트/PRJ-004_프로젝트_범위_원칙.md)을 따른다.

## 2. 추진 원칙

JKADH는 다음 원칙으로 추진한다.

- 문서와 운영 기준을 먼저 고정한다.
- 실제 제품 개발에 적용하면서 검증한다.
- Harness는 수동 운영에서 시작한다.
- 반복 가능한 흐름만 부분 자동화한다.
- High-risk 작업은 자동화보다 사람 승인과 검증을 우선한다.
- Token, 상태, 검증 근거는 자동화 판단의 기반 데이터로 축적한다.
- 단계는 고정 일정이 아니라 운영 성숙도 기준으로 전환한다.

## 3. 전체 단계

JKADH는 다음 단계로 추진한다.

| 단계 | 이름 | 핵심 목표 |
|---|---|---|
| Phase 1 | 문서와 운영 기준 정립 | 세션이 바뀌어도 같은 기준으로 작업할 수 있는 공식 문서와 정책을 만든다. |
| Phase 2 | GitHub 기반 수동 Harness 운영 | Issue, Branch, PR, Review, Test, Report를 연결하는 수동 운영 흐름을 만든다. |
| Phase 3 | Token과 상태 기반 부분 자동화 | Token, 작업 상태, 에이전트 배치 근거를 기록하고 Low-risk 판단 일부를 자동화한다. |
| Phase 4 | 다중 AI 개발팀 운영 플랫폼화 | 검증된 운영 흐름을 다중 AI 개발팀 운영 플랫폼으로 확장한다. |

Phase 1부터 Phase 3까지를 현실적인 추진 범위로 둔다. Phase 4는 장기 방향이며 별도 검증 후 구체화한다.

## 4. Phase 1: 문서와 운영 기준 정립

Phase 1의 목표는 JKADH의 기본 운영 기준을 공식 문서로 고정하는 것이다.

| 항목 | 내용 |
|---|---|
| 주요 산출물 | 프로젝트 정체성, 비전과 목표, 서비스 정의, 범위 원칙, 조직/정책 문서 |
| 운영 방식 | 사람이 문서를 읽고 작업 범위와 역할을 확인한다. |
| 성공 기준 | 세션이 바뀌어도 같은 기준으로 작업을 시작할 수 있다. |
| 제외 범위 | 자동화 구현, 대시보드 UI, 외부 API 자동 연동 |

Phase 1에서는 빠른 자동화보다 문서 품질과 기준 일관성을 우선한다.

## 5. Phase 2: GitHub 기반 수동 Harness 운영

Phase 2의 목표는 GitHub를 중심으로 수동 Harness 운영 흐름을 검증하는 것이다.

| 항목 | 내용 |
|---|---|
| 주요 산출물 | Issue Loop 운영 방식, PR Review Loop 운영 방식, Report Loop 운영 방식 |
| 운영 방식 | 사람이 Issue와 PR을 확인하고, Harness 역할을 수동 또는 반자동으로 수행한다. |
| 성공 기준 | Issue, Branch, PR, Review, Test, Report가 하나의 흐름으로 연결된다. |
| 제외 범위 | 완전 자동 배치, 자동 승인, 자동 승급 |

Phase 2에서는 AI 개발자 배치와 결과 수집 흐름을 실제 작업에 적용해 반복 패턴을 찾는다.

## 6. Phase 3: Token과 상태 기반 부분 자동화

Phase 3의 목표는 Token과 작업 상태를 기준으로 일부 운영 판단을 자동화하는 것이다.

| 항목 | 내용 |
|---|---|
| 주요 산출물 | Token Usage 기록, Token Budget 판단, 작업 상태 추적, Low-risk 수용 후보 판단 |
| 운영 방식 | Harness가 검증 가능한 데이터를 기준으로 일부 판단을 제안하거나 수행한다. |
| 성공 기준 | Token과 상태 정보가 작업 지속, 축소, 분할, 재배치 판단에 사용된다. |
| 제외 범위 | High-risk 자동 처리, 비용 최적화 제품화, 사람 승인 없는 승급 |

Phase 3 자동화는 명확한 근거가 있는 Low-risk 작업에 한정한다.

## 7. Phase 4: 다중 AI 개발팀 운영 플랫폼화

Phase 4는 장기 방향이다.

Phase 4에서는 검증된 운영 모델을 바탕으로 여러 AI 개발자, 여러 제품, 여러 작업 흐름을 동시에 운영할 수 있는 플랫폼화를 검토한다.

Phase 4는 현재 완료 범위가 아니다. Phase 1부터 Phase 3까지의 운영 결과가 충분히 축적된 뒤 구체화한다.

## 8. 단계 전환 기준

다음 단계로 넘어가기 전에는 다음 조건을 확인한다.

| 전환 | 조건 |
|---|---|
| Phase 1 → Phase 2 | 작업 시작 문서, 역할 문서, Git 정책, Issue 정책이 실제 세션 시작에 사용 가능하다. |
| Phase 2 → Phase 3 | 수동 Harness 운영에서 반복되는 판단과 상태 정보가 확인된다. |
| Phase 3 → Phase 4 | 부분 자동화가 실제 제품 개발에서 안정적으로 동작하고 승인 경계를 지킨다. |

단계 전환은 일정이 아니라 운영 성숙도와 검증 결과를 기준으로 판단한다.

## 9. Harness 자율성 단계와 운영 경계

Harness 자율성은 한 번에 확대하지 않고, 판단 가능한 범위와 사람 승인 필수 범위를 분리해 단계적으로 확장한다. 이 절은 세부 정책을 대체하지 않고, 로드맵 관점에서 어느 수준까지 자동화할지 정리한다.

초기 기준에서 Harness가 판단할 수 있는 작업 범위는 다음과 같다.

| 단계 | 이름 | 판단 가능 범위 | 금지 조건 |
|---|---|---|---|
| A0 | 기록 보조 | 세션 시작 정보, 관련 문서, Backlog, 현재 브랜치, 작업트리 상태를 확인하고 보고한다. | 작업 범위 확정, Issue 생성, 브랜치 생성, 파일 수정 |
| A1 | 범위 점검 | Issue 또는 요청의 목적, 범위, 제외 범위, 완료 조건 충족 여부를 판단하고 보완 필요 항목을 제안한다. | 보완 없이 실행 착수, 범위 변경 확정 |
| A2 | 작업 구조화 | Work Order 초안, 세부 태스크, 검증 방법, 중단 조건, 관련 문서 목록을 작성한다. | 사람 확인 없이 Work Order 최종 확정 |
| A3 | Low-risk 실행 후보 판단 | 문서 오탈자, 링크, 인덱스, Backlog 상태, 제목 형식처럼 Low-risk 후보 작업인지 판단한다. | 정책 의미 변경, 제품 방향 변경, 코드 변경, 외부 시스템 쓰기 |
| A4 | 운영 조정 제안 | Token 제약, 작업 분량, 역할 적합성, 검증 공백을 근거로 축소, 분할, 재배치, 중단을 제안한다. | 비용 한도 변경, 작업 범위 확대, 승인 조건 우회 |
| A5 | 수용 후보 보고 | 완료 조건과 검증 근거가 충족된 Low-risk 결과를 수용 후보로 보고하고 PR 또는 Report 연결을 제안한다. | 최종 수용, 자동 병합, dev/stg/main 승급 |

Harness가 판단을 진행할 수 있는 공통 조건은 다음과 같다.

- 판단 기준이 공식 문서, Issue, Backlog, PR, 검증 결과 중 하나에 연결되어 있다.
- 승인 필수 조건에 해당하지 않는다.
- 판단 근거를 Work Order, PR 본문, Report, Backlog 중 하나에 남길 수 있다.
- 실패하거나 불확실할 때 Stop(중단), Need Approval(승인 필요), Backlog(백로그)로 전환할 수 있다.
- 판단 결과가 사람 리드의 최종 결정권을 대체하지 않는다.

다음 작업은 Harness가 독자적으로 판단하거나 확정하지 않는다.

- 제품 방향, 비전, 목표, 범위 원칙 변경
- 반복 적용되는 정책 변경의 최종 확정
- Secret 접근, 배포, 외부 시스템 쓰기
- 코드 변경 결과의 최종 수용
- 자동 병합, 자동 승급, main 영향 작업
- 검증 근거가 부족한 결과 수용
- Low-risk 여부가 불명확한 작업의 독자 진행

현재 단계에서는 `A0`부터 `A2`까지를 수동 운영에서 안정적으로 수행 가능한 범위로 본다. `A3`부터 `A5`까지는 검증 기록과 사람 승인 경계가 충분히 쌓인 뒤 부분 자동화 후보로 다룬다.

### 9.1 사람 승인 필수 조건

Harness 자율성 단계에서 사람 승인이 필요한 조건은 자율성 단계보다 우선한다. 세부 정책은 [POL-005 사람 승인 경계 정책](../03.정책/POL-005_사람_승인_경계_정책.md)을 따른다.

다음 조건이 확인되면 Harness는 자동 판단을 중단하고 Need Approval(승인 필요) 또는 Stop(중단)으로 전환한다.

| 승인 조건 | 발생 예시 | Harness 처리 |
|---|---|---|
| 방향 변경 | 서비스 정체성, 비전, 목표, 범위 원칙을 바꾸는 판단 | 변경안과 영향 범위를 정리해 사람 승인 요청 |
| 정책 변경 | 반복 적용되는 운영 규칙, 승인 경계, Git 흐름을 바꾸는 판단 | 초안과 대안을 제시하고 승인 전 확정 금지 |
| 권한 영향 | Secret 접근, 외부 시스템 쓰기, 배포, main 영향 작업 | 실행 중단 후 승인 필요 사유 기록 |
| 자동 승급 | PR 병합, `dev`, `stg`, `main` 승급을 자동으로 진행하려는 경우 | 자동 진행 금지, 사람 승인 또는 명시 명령 대기 |
| 코드 결과 수용 | 코드 변경 결과를 사람 리뷰 없이 최종 수용하려는 경우 | Review(리뷰) 또는 Need Approval(승인 필요)로 전환 |
| 검증 공백 | 완료 조건, 테스트, 링크, diff, 실행 결과 근거가 부족한 경우 | 수용 후보 금지, 검증 보강 또는 Stop(중단) |
| 비용 한도 변경 | Token Budget, 비용 한도, 사용량 제한 변경 | 선택지와 영향 범위를 정리해 승인 요청 |
| 판단 불확실 | Low-risk 여부, 범위 포함 여부, 승인 필요 여부가 불명확한 경우 | 독자 진행 금지, 질문 또는 Backlog 분리 |

사람 승인 필수 조건에 해당하지 않는 경우에도 Harness는 Approval Pass(승인 통과)를 기록한다. Approval Pass는 승인을 생략한다는 뜻이 아니라, 승인 필수 조건을 점검했고 해당 조건이 없었다는 기록이다.

초기 운영에서는 사람 승인 조건이 없는 문서 링크 수정, 오탈자 수정, 인덱스 갱신, Backlog 상태 갱신처럼 검증 가능한 Low-risk 후보만 Approval Pass 대상으로 본다.

### 9.2 Work Order 승인 Pass 기록 방식

Approval Pass(승인 통과)는 Harness가 사람 승인을 생략한다는 뜻이 아니다. 실행 전에 승인 필수 조건을 확인했고, 해당 조건이 없었다는 근거를 남기는 기록이다.

Work Order에는 다음 항목을 최소 기록한다.

| 항목 | 기록 내용 |
|---|---|
| 승인 상태 | Approval Pass(승인 통과) |
| 확인 시점 | Work Order Review(작업지시서 검토) 또는 Gate Check(게이트 확인) 시점 |
| 확인한 승인 조건 | 방향 변경, 정책 변경, 권한 영향, 자동 승급, 코드 결과 수용, 검증 공백, 비용 한도 변경, 판단 불확실 여부 |
| 판단 근거 | Issue 범위, 제외 범위, 관련 문서, 검증 방법, 변경 파일, 작업 성격 |
| 허용 범위 | Approval Pass로 진행해도 되는 작업 범위 |
| 제외 범위 | Approval Pass로 진행하지 않을 작업 범위 |
| 재확인 조건 | 범위 변경, 새 파일 유형 추가, 외부 시스템 쓰기 발생, 검증 실패, Token 제약 발생 |

Approval Pass 기록은 다음 형식을 기본으로 한다.

```markdown
## 승인 확인

- 승인 상태: Approval Pass(승인 통과)
- 확인 시점:
- 확인한 승인 필수 조건:
  - 방향 변경: 해당 없음
  - 정책 변경: 해당 없음
  - 권한 영향: 해당 없음
  - 자동 승급: 해당 없음
  - 코드 결과 수용: 해당 없음
  - 검증 공백: 해당 없음
  - 비용 한도 변경: 해당 없음
  - 판단 불확실: 해당 없음
- 판단 근거:
- Approval Pass 허용 범위:
- Approval Pass 제외 범위:
- 재확인 조건:
```

작업 중 승인 조건이 새로 발생하면 기존 Approval Pass는 더 이상 유효하지 않다. Harness는 Work Order를 갱신하고 Need Approval(승인 필요), Stop(중단), Backlog(백로그) 중 하나로 전환한다.

### 9.3 Token 제약 시 중단과 재배치 기준

Token 제약은 작업 실패가 아니라 운영 조정 신호다. Harness는 Token 상태를 근거로 작업을 계속할지, 축소할지, 분할할지, 재배치할지, 중단할지 판단한다. 세부 기준은 [OPS-001 Token 운영](../10.운영/OPS-001_Token_운영.md)을 따른다.

Token 상태별 기본 처리는 다음과 같다.

| Token 상태 | 한글명 | 판단 기준 | Harness 처리 |
|---|---|---|---|
| Available | 사용 가능 | 현재 범위와 검증, 보고를 수행할 수 있음 | 계획대로 진행 |
| Watch | 관찰 필요 | 작업은 가능하지만 검증 또는 보고 예산을 확인해야 함 | 진행하되 검증 예산과 남은 범위 기록 |
| Constrained | 제약 | 현재 범위를 그대로 유지하기 어려움 | 범위 축소, 세부 태스크 분할, 역할 재배치 검토 |
| Exhausted | 소진 | 작업 지속 또는 검증에 필요한 Token이 부족함 | Stop(중단) 또는 Backlog(백로그) 전환 |
| Unknown | 불명확 | 예산 또는 사용량 판단이 어려움 | 장기 작업, 고위험 작업, 자동 판단 확대 금지 |

Harness는 Token 제약이 확인되면 다음 순서로 조정한다.

```text
현재 Token 상태 확인
↓
완료 조건과 검증 예산 확인
↓
범위 축소 가능성 검토
↓
세부 태스크 분할 가능성 검토
↓
역할 또는 AI 개발자 재배치 검토
↓
사람 승인, Stop, Backlog 전환 여부 판단
```

중단 기준은 다음과 같다.

- 검증과 보고에 필요한 최소 Token이 남아 있지 않다.
- 범위 축소가 완료 조건을 훼손한다.
- 필요한 문서 또는 코드 맥락을 유지할 수 없다.
- 같은 실패나 불명확성이 반복되어 Token을 소모한다.
- Token 상태가 Unknown(불명확)인데 작업이 장기 작업 또는 고위험 작업이다.
- 추가 Token 또는 비용 한도 변경이 필요하지만 사람 승인이 없다.

재배치 기준은 다음과 같다.

- 현재 역할보다 다른 에이전트 역할이 작업 성격에 더 적합하다.
- 작업이 설계, 구현, 리뷰, QA, 운영처럼 서로 다른 성격으로 분리된다.
- 현재 세션의 맥락 한계를 넘어서 별도 Work Order로 나누는 편이 검증 가능하다.
- Token 제약이 특정 역할 또는 작업 구간에서 반복된다.

Token 제약으로 작업을 축소, 분할, 재배치, 중단한 경우 Harness는 Work Order 또는 Report에 Token 상태, 판단 근거, 검증 영향, 후속 처리 방향을 남긴다.

### 9.4 Low-risk 수용 후보와 최종 수용의 차이

Low-risk 수용 후보는 Harness가 결과를 최종 승인한다는 뜻이 아니다. 검토해도 될 만큼 범위, 증거, 검증, 승인 경계가 갖춰졌다는 뜻이다. 최종 수용은 사람 주도 검토, 명시 승인, PR 병합, 배포, 환경 승급 같은 별도 의사결정으로 남긴다.

| 구분 | 주체 | 의미 | 가능한 처리 | 금지/주의 |
| --- | --- | --- | --- | --- |
| Low-risk 실행 후보 | Harness | 실행 전 작업이 좁고 되돌리기 쉬우며 승인 필수 조건이 없다고 판단한 상태 | Work Order에 판단 근거 기록, 제한된 범위 실행 제안 | 정책 의미 변경, 제품 방향 변경, 코드 결과 수용, 외부 쓰기 포함 시 후보로 보지 않음 |
| 수용 후보(Accept Candidate) | Harness | 실행 후 완료 조건, 검증, 변경 증거가 맞아 결과를 사람에게 검토 대상으로 올릴 수 있는 상태 | Report/PR/Work Order에 증거와 함께 제시, 후속 리뷰 요청 | 최종 승인, 자동 병합, dev/stg/main 승급으로 해석하지 않음 |
| 최종 수용 | 사람 리드 또는 권한자 | 결과를 실제 반영, 병합, 종료, 승급하기로 결정한 상태 | Issue/PR 종료, 병합, 운영 반영, 다음 단계 착수 | 명시 승인 없이 Harness가 자동 판단으로 대체하지 않음 |

수용 후보로 올리려면 작업 범위와 제외 범위가 지켜져야 한다. 완료 조건, 필요한 검증, Approval Pass 또는 명시 승인도 확인되어야 한다. 또한 변경 증거와 후속 흐름이 문서, PR, Report, Backlog 중 하나 이상에 연결되어야 한다.

판단이 불명확하거나 승인 필수 조건이 새로 발견되면 수용 후보로 올리지 않고 Review Required, Need Approval, Backlog Required, Stop Required 중 하나로 전환한다. 특히 코드 결과의 최종 수용, 배포, 외부 시스템 쓰기, dev/stg/main 영향은 Low-risk 수용 후보 판단만으로 처리하지 않는다.

## 10. 운영 데이터 API에서 남겨야 할 자율성 판단 근거

Harness의 자율성 판단은 사후에 사람이 추적할 수 있어야 한다. 운영 데이터 API에는 단순 결과만 남기지 않고 판단 근거, 승인 경계, 검증 상태, 후속 처리를 함께 남긴다. 이 절은 로드맵 수준의 최소 기록 항목이며, 상세 저장 기준은 [OPS-001 Token 운영](../10.운영/OPS-001_Token_운영.md)과 [DSN-002 Harness 운영데이터 API 서비스](../05.설계/DSN-002_Harness_운영데이터_API_서비스.md)에서 구체화한다.

| 기록 항목 | 설명 |
| --- | --- |
| 작업 식별자 | Issue, PR, Work Order, Report, Backlog ID 중 연결 가능한 식별자 |
| 판단 시점 | 자율성 판단이 이루어진 시각과 세션 또는 실행 단위 |
| 자율성 단계 | A0~A5 중 적용한 Harness 자율성 단계 |
| 판단 유형 | Low-risk 실행 후보, Approval Pass, 수용 후보, 운영 조정, 중단, 재배치 등 |
| 판단 근거 | 범위, 제외 범위, 관련 정책, 변경 파일, 검증 방법, 관찰된 상태 |
| 승인 상태 | Approval Pass, Need Approval, Approval Granted, Approval Rejected, Approval Deferred 중 하나 |
| 결과 상태 | Accept Candidate, Review Required, Backlog Required, Stop Required 등 결과 수용 판단 상태 |
| Token 상태 | Available, Watch, Constrained, Exhausted, Unknown 중 하나와 판단 근거 |
| 검증 상태 | 실행한 검증, 생략한 검증, 실패 여부, 수용 후보 판단 가능 여부 |
| 후속 처리 | 계속 진행, 축소, 분할, 재배치, Backlog 전환, 사람 승인 요청, PR/Report 연결 |

API 기록은 Harness의 자동 판단을 정당화하기 위한 장치가 아니라 사람이 나중에 판단을 재구성할 수 있게 하는 감사 로그다. 특히 수용 후보, Approval Pass, Token 제약, Stop/Backlog 전환은 반드시 근거와 함께 남겨야 한다.

초기에는 운영 데이터 API를 별도 구현하기보다 Work Order, Report, PR 본문, Backlog 이력에 같은 필드를 남기는 방식으로 시작한다. 이후 Phase 3에서 실제 API 또는 이벤트 스키마로 승격한다.

## 11. 조정 원칙

추진방향은 진행 과정에서 조정될 수 있다.

조정은 다음 기준을 따른다.

- 실제 제품 개발 적용 결과를 우선한다.
- 불필요해진 항목은 제거한다.
- 새로 필요한 항목은 Backlog로 분리한 뒤 Issue로 승격한다.
- 단계 순서가 현실과 맞지 않으면 사람 리드의 판단으로 조정한다.
- 범위 원칙과 충돌하는 조정은 먼저 범위 원칙 문서를 검토한다.
