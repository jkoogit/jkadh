## 1. 목적

본 문서는 JKADH AI서비스플랫폼에서 Token Budget과 Token Usage를 작업 운영 판단에 사용하는 기준을 정의한다.

Token 운영은 비용 절감만을 위한 규칙이 아니다. Harness가 작업을 계속할지, 축소할지, 분할할지, 재배치할지, 중단할지를 판단하기 위한 물리적 제약 관리 기준이다.

## 2. 적용 범위

본 문서는 다음 작업에 적용한다.

- Issue 기반 작업
- Work Order 작성과 검토
- 에이전트 역할 배치
- 작업 분해와 재배치
- 결과 수집, 검증, 보고
- Harness의 Stop(중단), Backlog(백로그), Report(보고) 판단

본 문서는 구체적인 과금 단가나 월별 예산을 확정하지 않는다. 비용 한도 변경은 [POL-005 사람 승인 경계 정책](../03.정책/POL-005_사람_승인_경계_정책.md)에 따라 사람 리드의 승인 대상이다.

## 3. 기본 원칙

Token 운영의 기본 원칙은 다음과 같다.

- Token Budget은 작업 실행 가능성을 판단하는 제약이다.
- Token Usage는 작업 결과 보고에 남겨야 하는 운영 근거다.
- Token Usage는 외부 AI API 사용량뿐 아니라 내부 작업 상태, 역할 배치, 검증 결과와 함께 저장되어야 한다.
- Token 부족은 실패가 아니라 축소, 분할, 재배치, 중단 판단 신호다.
- Token 절감 때문에 검증, 승인, 보안 기준을 낮추지 않는다.
- Token 한도가 불명확하면 장기 작업이나 고위험 작업을 자동 진행하지 않는다.
- 작업 범위가 커질수록 Token Budget도 Work Order에서 명시한다.

## 4. Token 운영 단위

Token Budget은 다음 단위로 관리할 수 있다.

| 운영 단위 | 한글명 | 의미 | 사용 시점 |
|---|---|---|---|
| Task Budget | 작업 예산 | 하나의 Work Order 또는 세부 태스크에 허용된 Token 범위 | Work Order 작성 시 |
| Agent Budget | 에이전트 예산 | 특정 역할 또는 AI 개발자에게 배정한 Token 범위 | 역할 배치 시 |
| Loop Budget | 루프 예산 | Issue Loop, PR Review Loop, Report Loop 같은 흐름별 Token 범위 | Harness 흐름 시작 시 |
| Session Budget | 세션 예산 | 현재 세션에서 사용할 수 있는 Token 범위 | 세션 운영 중 |
| Reserve Budget | 예비 예산 | 검증, 보고, 복구, 사람 승인 요청을 위해 남겨야 하는 Token 범위 | 장기 작업 또는 복합 작업 시 |

초기 운영에서는 모든 단위를 수치로 강제하지 않는다. 다만 장기 작업, 복합 작업, 여러 AI 개발자를 쓰는 작업은 Work Order에 어떤 Budget 단위를 적용했는지 기록해야 한다.

## 5. Token 상태

Harness는 Token 가용성을 다음 상태로 판단한다.

| 상태 | 한글명 | 의미 | 기본 처리 |
|---|---|---|---|
| Available | 사용 가능 | 현재 범위와 검증을 수행하기에 충분함 | 계속 진행 |
| Watch | 관찰 필요 | 남은 Token이 줄어 작업 축소 가능성을 봐야 함 | 진행하되 보고와 검증 예산 확인 |
| Constrained | 제약 | 현재 범위를 그대로 유지하기 어려움 | 범위 축소, 작업 분할, 재배치 검토 |
| Exhausted | 소진 | 작업 지속 또는 검증에 필요한 Token이 부족함 | Stop(중단) 또는 Backlog(백로그) |
| Unknown | 불명확 | 예산 또는 사용량을 판단할 수 없음 | 장기/고위험 작업 자동 진행 금지 |

Token 상태는 작업 품질 판단이 아니다. 실행 가능한 운영 범위를 판단하는 신호다.

## 6. 판단 기준

Harness는 다음 질문으로 Token 상태를 판단한다.

| 판단 지점 | 질문 | 결과 |
|---|---|---|
| 작업 지속 가능성 | 현재 Token으로 작업 범위와 완료 조건을 수행할 수 있는가? | 진행, 축소, 분할, 중단 |
| 검증 예산 | 결과 검증과 보고에 필요한 Token이 남아 있는가? | 진행, 검증 우선, 중단 |
| 맥락 부하 | 필요한 문서와 코드 맥락이 현재 세션에 담길 수 있는가? | 요약, 분할, 재배치 |
| 역할 배치 | 더 적합한 역할 또는 AI 개발자에게 넘기는 것이 효율적인가? | 유지, 재배치 |
| 반복 비용 | 같은 실패나 불명확성이 반복되어 Token을 소모하고 있는가? | Work Order 재작성, 보완 요청 |
| 승인 필요성 | 비용 한도 변경이나 추가 사용 승인이 필요한가? | Need Approval(승인 필요) |

Token 판단은 단독 기준으로 결과 수용을 결정하지 않는다. 결과 수용은 [POL-006 근거 기반 결과 수용 기준](../03.정책/POL-006_근거_기반_결과_수용_기준.md)을 따른다.

## 7. 초과 대응

Token Budget이 부족하거나 초과 가능성이 있으면 Harness는 다음 순서로 대응한다.

```text
현재 사용량과 남은 범위 확인
↓
완료 조건과 검증 예산 확인
↓
작업 범위 축소 가능성 검토
↓
세부 태스크 분할 가능성 검토
↓
역할 또는 AI 개발자 재배치 검토
↓
사람 승인 또는 중단 판단
↓
Report 또는 Backlog 연결
```

초과 대응 원칙은 다음과 같다.

- 검증과 보고에 필요한 최소 Token은 남겨야 한다.
- 완료 조건을 훼손하는 축소는 허용하지 않는다.
- 범위가 바뀌면 Work Order를 갱신한다.
- 세부 태스크로 나누면 각 태스크의 완료 조건과 검증 방법을 분리한다.
- 추가 예산이 필요하면 승인 요청에 이유, 영향 범위, 선택지를 포함한다.

## 8. 운영데이터 저장 기준

Token 운영은 Report에만 남기지 않고, 향후 저장소 또는 API 서비스에서 조회 가능한 운영데이터로 축적해야 한다.

최소 저장 대상은 다음과 같다.

| 데이터 | 설명 |
|---|---|
| Work Order ID | Token 사용량이 연결되는 작업 단위 |
| 에이전트 역할 | 작업을 수행한 주 담당 역할과 보강 역할 |
| AI 개발자 또는 도구 | 실제 작업에 사용한 AI 서비스, 모델, 실행 도구 |
| Token Budget | 적용한 예산 단위와 한도 |
| Token Usage | 외부 API 또는 작업 보고에서 수집한 사용량 |
| Token 상태 | Available(사용 가능), Watch(관찰 필요), Constrained(제약), Exhausted(소진), Unknown(불명확) |
| 작업 결과 | 완료, 중단, 분할, 재배치, Backlog, Need Approval(승인 필요) |
| 품질 결과 | 검증 통과, 리뷰 보강, 재작업, 수용 후보 여부 |

이 저장 기준은 즉시 구현 요구사항이 아니라 서비스 설계 방향이다. 저장소, API, 스킬 구성은 [DSN-002 Harness 운영데이터 API 서비스](../05.설계/DSN-002_Harness_운영데이터_API_서비스.md)에서 별도로 정의한다.

## 9. Work Order 기록 기준

Work Order에는 Token 운영 판단을 다음 형식으로 남긴다.

```markdown
## Token 운영

- 적용 Budget 단위: Task Budget / Agent Budget / Loop Budget / Session Budget / Reserve Budget
- Token 상태: Available(사용 가능) / Watch(관찰 필요) / Constrained(제약) / Exhausted(소진) / Unknown(불명확)
- 예상 Token 부담:
- 검증과 보고 예산:
- 축소 조건:
- 분할 조건:
- 재배치 조건:
- 중단 조건:
- 승인 필요 여부:
```

초기 운영에서는 정확한 Token 수치를 알 수 없는 경우에도 `Token 상태`와 `판단 근거`를 기록한다.

## 10. Report 기록 기준

Harness Report에는 Token 관련 내용을 다음 수준으로 남긴다.

| 항목 | 설명 |
|---|---|
| 적용 Budget 단위 | 어떤 예산 단위를 기준으로 운영했는지 |
| Token 상태 | 작업 종료 시점의 Token 상태 |
| 사용량 판단 | 충분, 관찰 필요, 제약, 소진, 불명확 중 어떤 판단이었는지 |
| 작업 조정 | 축소, 분할, 재배치, 중단 여부 |
| 검증 영향 | Token 제약이 검증 범위에 영향을 주었는지 |
| 후속 처리 | 추가 예산 승인, Backlog, Work Order 분리 필요 여부 |

Token 정보가 정확한 수치로 남지 않는 경우에도, 운영 판단과 후속 영향은 기록한다.

## 11. 금지 조건

다음 처리는 허용하지 않는다.

- Token 절감을 이유로 승인 필수 조건을 우회한다.
- Token 절감을 이유로 검증하지 않은 결과를 수용한다.
- Token 부족 상태에서 범위가 큰 작업을 계속 진행한다.
- 사용량을 알 수 없는데 장기 작업을 자동으로 확장한다.
- 비용 한도 변경을 사람 승인 없이 적용한다.
- Report 작성에 필요한 최소 맥락을 남기지 않고 세션을 종료한다.

Token 운영은 속도보다 통제 가능성을 우선한다.
