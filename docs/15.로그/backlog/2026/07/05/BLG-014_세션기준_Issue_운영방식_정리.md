## 1. 내용

Issue를 짧은 태스크 단위가 아니라 세션 또는 업무 단위로 관리하는 방식을 정리한다.

하나의 Issue 안에서 여러 태스크와 여러 PR을 진행할 수 있게 하고, `#태스크정리` 시점에 Issue명, 브랜치명, PR명을 현재 업무 맥락에 맞게 현행화하는 기준을 마련한다.

## 2. 발생 배경

작업 절차를 보강하는 과정에서 Issue가 너무 짧게 생성되면 세션의 업무 맥락이 끊기고, 대화 중 자연스럽게 발생하는 보조 작업을 별도 Issue로 과도하게 분리하게 되는 문제가 확인되었다.

문서, 정책, 설계 작업은 하나의 세션 안에서 범위가 조금씩 조정될 수 있으므로, Issue는 업무 맥락을 담고 PR은 검토 가능한 변경 단위로 나누는 방식이 더 적합하다.

## 3. 기대 효과

- 세션 시작 시 Issue 하나로 업무 맥락을 유지할 수 있다.
- 하나의 Issue 안에서 여러 PR을 작게 나누어 리뷰와 병합을 안전하게 진행할 수 있다.
- 관련 있지만 주 작업이 아닌 항목은 `[번외]`로 표시해 맥락을 잃지 않고 관리할 수 있다.
- `[번외]` 항목이 무겁거나 별도 판단이 필요하면 Backlog 또는 새 Issue 후보로 분리할 수 있다.

## 4. 처리 기준

`POL-002`, `POL-003`, `STA-002`에서 다음 기준을 정리한다.

- Issue는 세션 또는 업무 단위로 생성한다.
- Branch는 Issue 단위로 유지한다.
- PR은 Issue 안의 정리 단위로 여러 개 생성할 수 있다.
- PR 제목의 이슈내PR번호는 해당 Issue 안에서 증가시킨다.
- 관련성이 낮은 작업은 `[번외]`로 표시한다.
- `[번외]` 작업의 무게가 크거나 독립 판단이 필요하면 Backlog 또는 새 Issue로 분리한다.
- 작업 중 현재 태스크와 맞지 않는 주제가 들어오면 현재 Issue 반영, `[번외]`, Backlog, 새 Issue 후보 중 하나로 분류해 제안한다.
- `#태스크정리` 시 Issue명, 브랜치명, PR명, Backlog 분리 여부를 현행화한다.
