## 1. 내용

프로그램 수정 이후 변경 결과를 분석해 개발자가 인사이트를 얻을 수 있는 개선사항 보고 기준을 정리한다.

보고에는 단순 작업 완료 여부뿐 아니라 다음 관점을 포함한다.

- 하나의 주제로 여러 파일이 변경된 경우 설계 개선 여지가 있는지
- 리팩토링이 필요한 부분이 남았는지
- 현시점에서는 유지하지만 향후 수정이 발생하면 대응해야 할 구조적 이슈가 있는지
- 구현 백로그로 분리해야 하는 후속 작업이 있는지
- 좋은 설계 방향은 있으나 지금 적용하지 않는 판단 근거가 있는지

## 2. 발생 배경

Harness와 AI 개발자가 작업 결과를 제출할 때 검증 근거만 남기면 작업 완료 여부는 확인할 수 있지만, 개발자 관점의 구조적 학습과 후속 개선 포인트가 누락될 수 있다.

특히 여러 파일이 같은 이유로 변경되거나, 현재 범위에서는 수용 가능하지만 반복 수정 시 설계 개선이 필요한 경우에는 별도 보고 기준이 필요하다.

## 3. 기대 효과

- 작업 결과에서 설계 개선점과 구현 백로그를 체계적으로 발견할 수 있다.
- 단기 수정과 장기 개선을 분리해 현재 Issue 범위가 흔들리지 않게 한다.
- 개발자가 변경 결과를 읽고 코드 구조, 변경 비용, 리팩토링 필요성을 빠르게 파악할 수 있다.
- Harness Report가 단순 완료 보고가 아니라 다음 작업 판단 자료로 활용된다.

## 4. 처리 기준

Harness Report 또는 구현 작업 보고 기준에 다음 항목을 추가할지 검토한다.

- 변경 범위 요약
- 변경 파일 간 공통 원인 또는 결합도 분석
- 설계 개선 후보
- 리팩토링 후보
- 지금 처리하지 않는 이유
- 구현 Backlog 후보
- 다음 수정 시 재검토해야 할 조건

공식 문서에 반영할 때는 결과 수용 기준과 중복되지 않도록, 검증 근거와 개발 인사이트 보고의 경계를 분리한다.
