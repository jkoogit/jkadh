## 1. 내용

Harness가 Work Order, Token Usage, 에이전트 배정, 검증, 리뷰, 재작업 이력을 축적하고 조회할 수 있는 운영데이터 저장소와 API 서비스 구조를 정리한다.

이 백로그는 단순 Token 집계를 넘어서 AI개발플랫폼 서비스가 내부 운영 현황을 품질 개선에 활용하기 위한 기반을 다룬다.

## 2. 발생 배경

Token 운영을 정책으로만 다루면 실제 작업 지속, 중단, 분할, 재배치 판단을 반복 가능하게 만들기 어렵다.

외부 AI API에서 Token Usage를 수집하는 것도 필요하지만, 플랫폼 내부에서는 작업 ID, 역할 배치, Work Order 상태, 검증 결과, 리뷰 결과, 재작업 여부를 함께 저장해야 한다. 이 데이터가 있어야 Harness가 다음 작업에서 더 나은 배정과 품질 판단을 할 수 있다.

## 3. 기대 효과

- Token Usage를 비용 관리뿐 아니라 품질 개선 데이터로 활용할 수 있다.
- 에이전트 역할별 작업 효율과 재작업 패턴을 분석할 수 있다.
- Work Order 분해, 재배치, 중단 판단을 데이터 기반으로 개선할 수 있다.
- 반복되는 작업을 HCP, Skill 또는 API 프로세스로 정리할 수 있다.
- Harness가 상태 보고와 배정 추천을 일관되게 수행할 수 있다.

## 4. 처리 기준

설계 문서에서 다음 항목을 정리한다.

- 운영데이터 저장소의 최소 엔티티
- Work Order, Token Usage, Agent Assignment, Verification, Review, Report 간 관계
- Token 수집 API와 내부 상태 관리 API의 경계
- 배정 추천, 상태 조회, 보고 생성에 필요한 API 후보
- 반복 작업을 HCP 또는 Skill로 구성할 기준
- 초기 구현 범위와 제외 범위

초기 구현은 대형 플랫폼 전체가 아니라 수동 운영을 대체할 수 있는 최소 저장소와 API 후보 정의부터 시작한다.
