## 1. 내용

JKADH AI개발플랫폼이 타겟서비스 개발을 시작하기 전에 필요한 운영 구조를 검토한다.

검토 범위는 다음을 포함한다.

- 플랫폼 자체에 백엔드/API/UI 배포가 필요한 시점
- GitHub Actions를 배포보다 검증 자동화로 먼저 사용할지 여부
- 타겟서비스 repo를 JKADH 플랫폼 repo와 분리해 관리하는 기준
- `ProjectProfile`, `WorkOrder`, `PermissionProfile` 기반으로 타겟서비스 작업을 접수, 배정, 검증, 보고, PR 연결하는 흐름
- 플랫폼 작업, 타겟서비스 작업, integration 작업을 분리하는 기준
- 타겟서비스 첫 기능 구현 전 필요한 최소 운영 절차

## 2. 발생 배경

현재 JKADH는 상시 실행되는 백엔드 서버보다 Harness CLI, GitHub, 문서, HCP 상태, DB를 기반으로 한 Control Plane에 가깝다.

HCP 명령과 DB 상태 관리가 진행되면서 AI 채팅 주문을 작업 단위로 구조화하고, 에이전트 역할과 검증 흐름으로 연결할 기반은 마련되고 있다. 다만 아직 실제 타겟서비스 기능 개발을 시작하지 않았기 때문에, 첫 타겟서비스 작업 전에 다음 질문을 정리할 필요가 있다.

- 플랫폼 자체를 배포해야 하는가?
- 배포보다 GitHub Actions 기반 검증 자동화가 먼저 필요한가?
- 타겟서비스 repo를 어떻게 등록하고 작업 범위를 고정할 것인가?
- AI 에이전트가 타겟서비스 코드를 수정하고 관리하는 흐름을 Harness가 어디까지 통제할 것인가?
- 타겟서비스 배포는 플랫폼이 담당할 것인가, 타겟서비스 repo가 담당할 것인가?

## 3. 기대 효과

- 타겟서비스 첫 작업을 시작하기 전에 플랫폼 repo와 서비스 repo의 경계를 명확히 한다.
- 배포, Secret, main 영향, 외부 시스템 쓰기 같은 승인 게이트를 사전에 구분한다.
- GitHub Actions를 검증 자동화, CLI 릴리즈, DB dry-run, 타겟서비스 테스트 중 어디에 먼저 적용할지 결정할 수 있다.
- 타겟서비스 기능 구현 작업이 JKADH 플랫폼 정책 변경과 한 PR에 섞이는 것을 줄인다.
- 이후 `ProjectProfile`과 `WorkOrder` 구현 범위를 정하는 기준이 생긴다.

## 4. 처리 기준

- 플랫폼 자체 배포가 필요한 조건과 아직 필요하지 않은 조건을 구분한다.
- 타겟서비스 배포 책임은 기본적으로 타겟서비스 repo에 두는지 검토한다.
- JKADH 플랫폼의 초기 책임을 Control Plane, 검증 자동화, 작업 상태/근거 관리로 둘지 결정한다.
- 타겟서비스 첫 작업 시나리오를 한 턴 실행 사이클로 정의한다.
- `platform`, `service`, `integration` 작업 유형별 Issue, branch, PR 분리 기준을 정리한다.
- GitHub Actions의 초기 적용 대상을 `test/check`, Harness 상태 정합성, DB dry-run/validate/check, 타겟서비스 테스트 중에서 우선순위화한다.
- 해결 문서는 설계 문서 또는 운영 문서로 작성한다.
- 본 Backlog는 타겟서비스 개발 운영구조와 배포 필요성 판단을 다루는 설계 또는 운영 문서가 작성되고, 후속 구현/검증 Issue 후보가 분리되면 Resolved로 전환한다.
