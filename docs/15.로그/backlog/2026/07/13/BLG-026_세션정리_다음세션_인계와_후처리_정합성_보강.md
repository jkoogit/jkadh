## 1. 내용

`#세션정리` 실행 결과에서 다음 세션 전달 정보가 사용자에게 명확히 표시되지 않는 문제를 보강한다.

세션정리 결과는 회고 문서 작성, Issue 종료, PR 병합, dev/stg/main 승급까지 수행하지만, 완료 후 사용자가 바로 확인해야 하는 다음 세션 인계 정보와 후처리 검증 결과가 출력 중심에 놓이지 않는다.

## 2. 발생 배경

011 세션정리에서 `RET-010`은 생성되었지만, 실제 실행 결과 출력은 PR 업데이트/병합/승급 단계 로그와 HCP 상태 요약 중심이었다.

이 과정에서 다음 세션 인계 내용은 회고 문서 안에는 있었으나 최종 사용자 응답과 CLI 실행 요약에서 독립 섹션으로 드러나지 않았다. 또한 세션정리 커밋이 이미 병합된 PR에 연결되는 형태로 처리되어, 후속 수동 보정 PR이 필요했다.

## 3. 확인된 미처리 건

| 항목 | 증상 | 영향 |
|---|---|---|
| 다음 세션 인계 표시 | `session close --execute` 완료 출력에서 다음 세션 추천 세션명, 시작점, 남은 Backlog, 검증 상태가 별도 섹션으로 표시되지 않음 | 다음 세션 시작 시 사용자가 회고 문서를 직접 열어야 함 |
| 세션정리 PR 재사용 | 이전에 병합된 작업 PR 제목이 세션정리 PR 제목으로 갱신되고, 새 회고 커밋은 dev/main에 반영되지 않은 상태가 발생함 | 세션정리 완료로 보였지만 실제 산출물이 기준 브랜치에 누락됨 |
| 승급 검증 기준 | 세션정리 커밋이 반영되지 않았는데도 dev/stg/main 정렬 완료처럼 보일 수 있음 | 세션 종료 기준선이 실제 산출물과 어긋남 |
| 회고 생성 시점의 상태값 | 회고 생성 시점에는 HCP 세션 상태가 아직 `active`라서 생성 회고에 stale 상태가 들어갈 수 있음 | 세션정리 후 회고 문서가 별도 보정을 필요로 함 |
| 사후 검증 자동화 | 회고 문서 존재, README 인덱스 반영, Issue/PR open 여부, dev/stg/main 동일 커밋 여부를 세션정리 플로우가 끝에서 재검증하지 않음 | 사용자가 수동으로 최종 정합성을 다시 확인해야 함 |

## 4. 보완 방향

- `session close` 완료 출력에 `다음 세션 인계` 섹션을 추가한다.
- 인계 섹션에는 최소한 세션명 후보, 다음 시작점, 남은 Backlog, 열린 Issue/PR, dev/stg/main 정렬 커밋, HCP 상태 파일 경로를 표시한다.
- 종료된 세션의 기준 Issue 댓글에는 세션 관리정보를 남길 수 있도록 댓글 대상, 댓글 내용, Issue 유지/종료 판단을 출력에 표시한다.
- 세션정리 회고 커밋은 이미 병합된 PR을 재사용하지 않는다. OPEN PR이 감지되면 기본적으로 보류하고, 세션정리용 동일 head PR임이 확인되며 사용자가 `#세션정리.PR재사용`으로 명시 승인한 경우에만 기존 PR을 갱신한다.
- 회고 문서 생성 시점과 세션 완료 시점의 상태 차이를 줄이기 위해 최종 HCP 상태 반영 후 회고의 HCP 요약을 갱신하거나, 회고에는 실행 시점 값이 아닌 완료 후 확정 가능한 표현을 사용한다.
- `session close --execute` 마지막 단계에서 산출물 포함 여부와 원격 브랜치 정렬 여부를 자동 검증한다.

## 5. 처리 기준

- `#세션정리` 완료 후 사용자에게 다음 세션 인계 정보가 별도 섹션으로 출력된다.
- 세션정리 산출물 커밋이 dev/stg/main에 실제 포함되지 않으면 완료로 보고하지 않는다.
- 이미 `MERGED` 상태인 PR은 신규 세션정리 커밋 병합 대상으로 재사용하지 않는다.
- `OPEN` 상태인 PR은 자동 재사용하지 않고, 세션정리용 PR인지 확인한 뒤 명시 승인 주문을 받아 갱신한다.
- 생성된 RET 문서의 HCP 상태가 `active`로 남지 않는다.
- 세션정리 후 열린 Issue/PR, dev/stg/main 정렬, RET 문서/README 반영 여부를 자동 검증하고 결과를 표시한다.

## 6. 기대 효과

- 다음 세션 시작자가 회고 문서를 열기 전에도 인계 정보를 즉시 확인할 수 있다.
- 세션정리 완료 보고와 실제 원격 기준선이 일치한다.
- 세션정리 후 수동 보정 PR이 필요한 상황을 줄인다.
- HCP 상태 파일, 회고 문서, GitHub Issue/PR, dev/stg/main 기준선의 정합성을 한 번에 확인할 수 있다.
