## 1. 내용

여러 에이전트가 동시에 작업할 때 Backlog ID, Issue 전용 순번, 문서 ID, 세션번호 같은 채번값이 충돌하지 않게 하는 운영방식을 검토한다.

세션번호를 별도로 관리하는 방안도 후보로 두되, 더 단순하고 충돌에 강한 대안이 있는지 함께 검토한다.

## 2. 발생 배경

현재 백로그와 작업 문서는 사람이 순차적으로 다음 번호를 확인해 채번한다. 단일 에이전트 작업에서는 충분하지만, 여러 에이전트가 동시에 백로그나 문서를 만들면 같은 번호를 선택하거나 PR 병합 시 충돌이 발생할 수 있다.

특히 `BLG-###`, Issue 제목의 전용 순번, 작업 브랜치의 `{seq}`, 세션명이 서로 연동될수록 채번 충돌이 작업 흐름 전체에 영향을 줄 수 있다.

## 3. 검토 후보

| 후보 | 장점 | 우려 |
|---|---|---|
| 세션번호 중앙 관리 | 세션 단위 추적이 명확하다. | 중앙 파일 충돌이 새 병목이 될 수 있다. |
| 날짜+에이전트 prefix 사용 | 동시 작업 충돌 가능성이 낮다. | 번호 체계가 길어지고 사람이 읽기 어려울 수 있다. |
| GitHub Issue 번호 기준 | GitHub가 원자적으로 채번한다. | 로컬 백로그 작성이 Issue 생성에 의존한다. |
| 임시 ID 후 정리 시 보정 | 작업 시작이 빠르고 충돌에 유연하다. | 정리 단계에서 보정 누락 위험이 있다. |
| 충돌 감지 체크리스트 강화 | 기존 체계를 크게 바꾸지 않는다. | 충돌 자체를 예방하기보다 사후 발견에 가깝다. |

## 4. 처리 기준

다음 항목을 비교해 결정한다.

- 동시 작업 시 사람이 수동으로 관리해야 하는 값이 최소인가?
- 로컬 작업과 GitHub Issue 작업 모두에 적용 가능한가?
- PR 병합 충돌이 발생했을 때 복구 절차가 단순한가?
- 세션명, 브랜치명, Issue 제목, Backlog ID가 과도하게 강결합되지 않는가?
- `#태스크시작`과 `#태스크정리` 중 어느 시점에 채번을 확정하는 것이 안전한가?

## 5. 기대 효과

- 여러 에이전트가 동시에 백로그나 문서를 만들어도 번호 충돌을 줄인다.
- 채번 충돌이 발생해도 정리 단계에서 빠르게 감지하고 보정할 수 있다.
- 세션번호, 브랜치명, Issue, PR, Backlog 사이의 연결 기준을 더 명확히 할 수 있다.
