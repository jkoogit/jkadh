## 1. 목적

본 문서는 2026-07-05 세션에서 진행한 AI개발플랫폼 Harness 운영모델 정리 작업을 회고하고, 다음 세션에서 이어갈 판단 지점과 절차 개선 사항을 남긴다.

## 2. 세션 요약

이번 세션은 새로 만든 프로젝트에서 이전 논의의 맥락을 유지하면서 AI개발플랫폼의 운영 기준을 문서로 정리하는 데 집중했다.

핵심 방향은 개별 AI 세션에 흩어지는 작업 맥락을 Harness 중심의 운영 흐름으로 묶고, Work Order, 역할 배정, Token 운영, 검증, 결과 수용, 개선사항 분석, 운영데이터 저장소까지 단계적으로 정의하는 것이다.

## 4. 잘된 점

- Harness를 단순 실행 에이전트가 아니라 작업 접수, Work Order 작성, 역할 배정, 검증 근거 수집, 후속 흐름 연결을 담당하는 운영 시스템으로 정리했다.
- `이거 해줘` 수준의 요청을 바로 실행하지 않고 Work Order Build와 Review를 거치는 흐름을 명확히 했다.
- 작업 분량 분석과 세부 태스크 분리 기준을 Harness 상태 모델에 반영했다.
- Token을 비용 정보가 아니라 작업 지속, 축소, 분할, 재배치, 중단을 판단하는 운영 제약으로 정의했다.
- Low-risk 수용 후보, 사람 승인 경계, 근거 기반 결과 수용 기준을 분리해 자동화와 사람 판단의 경계를 나눴다.
- 수정결과 개선사항 분석 기준을 별도 문서로 두어 구현 후 설계 개선점과 리팩토링 후보를 남길 수 있게 했다.
- 운영데이터 API 서비스 방향을 정리해 Work Order, 배정, Token, 검증, 리뷰, Report 데이터를 향후 서비스화할 기반을 만들었다.

## 5. 아쉬운 점

- `#태스크정리` 단계에서 Issue를 먼저 만들고 PR과 연결해야 했지만, PR #8을 먼저 만들고 병합한 뒤 Issue #9를 소급 생성했다.
- 브랜치명이 중간에 `task_002-ai-dev-platform-harness-ops`로 바뀌었다가 `task_codex/002-ai-dev-platform-harness-ops`로 다시 보정되었다.
- 문서 헤더의 blockquote 표기 방식은 VSCode 보기 모드에서 가독성이 떨어질 수 있다는 문제가 확인되었지만, 템플릿 개선은 아직 처리하지 않았다.
- 여러 핵심 개념이 초안 수준으로 빠르게 정리되어 구현 단계에서 엔티티, API, 상태 전이, 승인 조건의 구체화가 필요하다.

## 6. 개선 사항

- 다음 작업부터는 `#태스크정리` 전에 Issue가 없으면 먼저 Issue를 생성하고, PR 본문에 Issue를 연결한다.
- `#태스크시작` 또는 `#세션시작` 단계에서 작업 브랜치명을 `task_{agent}/{seq}-{scope}` 형식으로 확정한다.
- 공식 문서에는 확정된 기준만 남기고, 검토 후보와 후속 결정은 Backlog 또는 회고에 분리한다.
- 상태값, 승인값, Token 상태처럼 운영 중 반복되는 값은 영문 상태와 한글명을 병기한다.
- 구현 단계에서는 `DSN-002`의 Store와 API 후보를 실제 엔티티, 자료형, 인덱스, 권한 경계로 구체화한다.
