## 1. 목적

본 문서는 AI가 JKADH 작업 세션을 시작할 때 확인해야 하는 절차와 읽기 순서를 정의한다.

세션과 사용하는 AI가 바뀌어도 같은 맥락에서 작업을 시작하기 위한 기준이다.

## 2. 적용 범위

이 문서는 JKADH에서 작업하는 모든 AI 에이전트에 적용한다.

다음 상황에서 먼저 확인한다.

- 새 세션 시작
- 새 작업 브랜치 생성 전
- 역할 전환 전
- PR 또는 승급 작업 전
- 기존 대화 맥락이 불명확할 때
- 사용자 주문과 표준 절차가 충돌하거나 해석이 갈릴 때

## 3. 시작 절차

AI는 세션을 시작하면 다음 순서로 확인한다.

```text
원격 저장소 확인
↓
dev / stg / main 일치 여부 확인
↓
현재 브랜치와 작업트리 확인
↓
역할 정체성 확인
↓
필수 문서 읽기
↓
최신 회고 확인
↓
Backlog 상태 확인
↓
세션명, 브랜치명, Issue 확인
↓
작업 범위 확정
```

## 4. Git 상태 확인

세션 시작 시 다음을 확인한다.

- 원격 저장소가 `jkoogit/jkadh`인지 확인한다.
- 원격 `dev`, `stg`, `main`이 같은 커밋인지 확인한다.
- 현재 브랜치가 무엇인지 확인한다.
- 작업트리에 미완료 변경이 있는지 확인한다.
- 새 작업은 원격 `main`을 현행화한 뒤 `main` 기준으로 시작한다.

작업 브랜치와 승급 규칙은 [POL-003 Git 작업관리방안](../03.정책/POL-003_Git_작업관리방안.md)을 따른다.

## 5. 역할 정체성 확인

작업을 시작하기 전에 이번 사안의 에이전트 역할을 확인한다.

역할이 명확하지 않으면 짧게 질문한다.

```text
이번 사안은 CTO 관점에서 정책을 정리할까요, 구현 관점에서 작업 범위를 정리할까요?
```

역할 기준은 [ORG-001 에이전트 공통역할](../02.조직/ORG-001_에이전트_공통역할.md)과 [ORG-002 에이전트 역할별 운영규칙](../02.조직/ORG-002_에이전트_역할별_운영규칙.md)을 따른다.

## 7. 최신 회고 확인

작업 전 [회고 폴더](../12.회고/README.md)의 최신 `RET-*` 문서를 확인한다.

최신 회고에서 다음 항목을 세션 시작 보고에 반영한다.

- 다음 세션 권장 시작점
- 남은 Backlog
- 절차 개선 사항
- 지난 작업에서 보완 예정으로 남긴 문서 항목

회고의 권장 작업과 Backlog 상태가 다르면 Backlog 인덱스를 기준으로 판단하되, 차이를 사용자에게 보고한다.

## 8. Backlog 확인

작업 전 Backlog 상태를 확인한다.

| 상태 | 한글명 | 시작 시 처리 |
|---|---|---|
| Ready | 준비 | 다음 Issue 후보로 검토한다. |
| Blocked | 차단 | 의존 대상이 해결되었는지 확인한다. |
| Deferred | 보류 | 현재 작업 범위에 포함하지 않는다. |
| Issue Linked | 이슈 연결 | 연결 Issue와 PR 상태를 확인한다. |

현재 Issue 범위 밖의 새 논점은 Backlog로 등록한다.

## 9. 세션명, 브랜치명, Issue 확인

작업 범위를 확정하기 전에 다음 항목을 먼저 고정한다.

- 세션명 번호와 주제를 확인한다. 예: `002_*`
- 작업 브랜치명을 `task_{agent}/{seq}-{scope}` 형식으로 확정한다.
- 관련 Issue가 있는지 확인한다.
- 관련 Issue가 없고 작업이 단순 확인을 넘어서면 Issue 등록 단계를 먼저 수행한다.
- Issue, 브랜치명, PR 제목 또는 본문의 작업 번호와 범위가 서로 어긋나지 않게 맞춘다.
- 결론이 확정된 사안도 현재 PR 포함, 별도 PR, Backlog, 새 Issue 후보 중 처리 경로를 확인한다.
- 동시작업 가능성이 있으면 GitHub Issue 번호를 기준으로 Backlog ID, 브랜치명, PR 순번 충돌 여부를 확인한다.

Issue 등록과 브랜치 규칙은 [POL-003 Git 작업관리방안](../03.정책/POL-003_Git_작업관리방안.md)을 따른다.

## 10. 작업 시작 기준

작업을 시작하기 전에 다음 항목이 명확해야 한다.

- 현재 역할
- 기준 브랜치
- 세션명 번호와 주제
- 작업 브랜치명
- 관련 Issue 또는 Backlog
- 작업 범위
- 제외 범위
- 완료 조건
- 검증 방법
- 확정 사안의 처리 경로
- Issue 기준 채번 충돌 여부

불명확한 항목이 있으면 작업 전에 확인한다.

`#태스크시작` 없이 대화 중 작업이 먼저 진행된 경우에는 작업을 중단하기보다 `#태스크정리`에서 누락된 항목을 사후 보정한다. 단, 큰 작업을 Issue 없이 계속 진행하지 않고 현재 Issue 반영, `[번외]`, Backlog, 새 Issue 후보 중 하나로 분류한다.

사용자 주문과 표준 태그 절차가 충돌하거나, 절차 생략 승인으로 해석될 여지가 있으면 실행 전에 확인한다. 명시 태그가 필요한 단계는 자연어 주문만으로 자동 실행하지 않는다.

## 11. 세션 시작 보고 형식

`#세션시작`을 수행한 뒤에는 다음 항목을 요약해 보고한다.

- 원격 `dev`, `stg`, `main` 일치 여부
- 현재 브랜치와 작업트리 상태
- 최신 회고 문서와 다음 세션 권장 시작점
- 미해결 Backlog 요약
- 세션명 번호와 추천 작업 브랜치명
- 관련 Issue 존재 여부와 필요 시 Issue 등록 단계
- 추천 다음 작업

## 12. 세션 정리 절차

`#세션정리`를 수행할 때는 다음 항목을 확인한다.

- 세션명이 현재 Issue, 브랜치명, PR 제목 또는 후속 작업 범위를 반영하는지 현행화한다.
- 세션 회고는 별도 Issue로 분리하지 않고 현재 Issue의 마무리 태스크로 정리한다.
- 회고 결과가 독립적인 정책, 설계, 운영 개선 작업으로 커지면 Backlog 또는 새 Issue 후보로 분리한다.
- 다음 세션명, 남은 작업, 확인할 Issue와 PR, 관련 문서 링크를 종료 보고에 남긴다.

## 13. 세션 태그

AI는 세션 중 다음 태그와 별칭을 같은 명령으로 인식한다.

| 표준 태그 | 허용 별칭 | 목적 |
|---|---|---|
| `#세션시작` | `#sessionstart`, `#session-start` | 세션 시작 절차 수행 |
| `#태스크시작` | `#테스크시작`, `#task시작`, `#taskstart`, `#task-start` | 작업 범위 확인과 브랜치 준비 |
| `#태스크정리` | `#테스크정리`, `#task정리`, `#taskclose`, `#task-close` | 작업 검증, PR, `dev` 머지 정리 |
| `#태스크승급` | `#테스크승급`, `#task승급`, `#taskpromote` | 자동 승급 |
| `#태스크승급-확인` | `#테스크승급-확인`, `#task승급-check`, `#taskpromote-check` | `main` 승급 전 확인 |
| `#태스크승급-검증` | `#테스크승급-검증`, `#task승급-stg`, `#taskpromote-stg` | `stg` 검증 후 피드백 대기 |
| `#세션정리` | `#sessionclose`, `#session-close` | 세션 회고와 종료 정리 |

승급 태그의 상세 절차는 [POL-003 Git 작업관리방안](../03.정책/POL-003_Git_작업관리방안.md)을 따른다.
