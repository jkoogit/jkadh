## 1. 목적

본 문서는 AI가 JKADH 작업 세션을 시작할 때 확인해야 하는 절차와 읽기 순서를 정의한다.

세션과 사용하는 AI가 바뀌어도 같은 맥락에서 작업을 시작하기 위한 기준이다.

## 2. 적용 범위

이 문서는 JKADH에서 작업하는 모든 AI 에이전트에 적용한다.

다음 상황에서 먼저 확인한다.

- 새 세션 시작
- 새 작업 브랜치 생성 전
- 역할 전환 전
- PR 또는 승급 작업 전
- 기존 대화 맥락이 불명확할 때
- 사용자 주문과 표준 절차가 충돌하거나 해석이 갈릴 때

## 3. 시작 절차

AI는 세션을 시작하면 다음 순서로 확인한다.

```text
원격 저장소 확인
↓
dev / stg / main 일치 여부 확인
↓
현재 브랜치와 작업트리 확인
↓
역할 정체성 확인
↓
필수 문서 읽기
↓
최신 회고 확인
↓
Backlog 상태 확인
↓
세션명, 브랜치명, Issue 확인
↓
작업 범위 확정
```

## 4. Git 상태 확인

세션 시작 시 다음을 확인한다.

- 원격 저장소가 `jkoogit/jkadh`인지 확인한다.
- 원격 `dev`, `stg`, `main`이 같은 커밋인지 확인한다.
- 현재 브랜치가 무엇인지 확인한다.
- 작업트리에 미완료 변경이 있는지 확인한다.
- 새 작업은 원격 `main`을 현행화한 뒤 `main` 기준으로 시작한다.

작업 브랜치와 승급 규칙은 [POL-003 Git 작업관리방안](../03.정책/POL-003_Git_작업관리방안.md)을 따른다.

## 5. 역할 정체성 확인

역할이 명확하지 않으면 짧게 질문한다.

```text
이번 사안은 CTO 관점에서 정책을 정리할까요, 구현 관점에서 작업 범위를 정리할까요?
```

역할 기준은 [ORG-001 에이전트 공통역할](../02.조직/ORG-001_에이전트_공통역할.md)과 [ORG-002 에이전트 역할별 운영규칙](../02.조직/ORG-002_에이전트_역할별_운영규칙.md)을 따른다.

## 7. 최신 회고 확인

작업 전 [회고 폴더](../12.회고/README.md)의 최신 `RET-*` 문서를 확인한다.

최신 회고에서 다음 항목을 세션 시작 보고에 반영한다.

- 다음 세션 권장 시작점
- 남은 Backlog
- 절차 개선 사항
- 지난 작업에서 보완 예정으로 남긴 문서 항목
- 다음 작업 후보에 맞춘 초기 세션명

회고의 권장 작업과 Backlog 상태가 다르면 Backlog 인덱스를 기준으로 판단하되, 차이를 사용자에게 보고한다.

세션 시작 시에는 최신 회고의 다음 세션 권장 시작점과 Backlog 인덱스를 대조해 초기 세션명을 제안한다. 초기 세션명은 확정값이 아니라 작업 범위 확인 전 가설이며, 관련 Issue 또는 Backlog가 확정되면 세션명, 브랜치명, PR 제목 기준을 다시 맞춘다.

## 8. Backlog 확인

작업 전 Backlog 상태를 확인한다.

| 상태 | 한글명 | 시작 시 처리 |
|---|---|---|
| Ready | 준비 | 다음 Issue 후보로 검토한다. |
| Blocked | 차단 | 의존 대상이 해결되었는지 확인한다. |
| Deferred | 보류 | 현재 작업 범위에 포함하지 않는다. |
| Issue Linked | 이슈 연결 | 연결 Issue와 PR 상태를 확인한다. |

Backlog 등록은 진행 중인 태스크가 있고, 그 태스크 범위 밖의 번외 또는 후속 주제가 들어온 경우에만 수행한다.

진행 중인 태스크가 없는 상태에서 새 작업 주제가 제시되면 Backlog로 우회하지 않고 `#태스크시작` 대상으로 본다.

## 9. 세션명, 브랜치명, Issue 확인

작업 범위를 확정하기 전에 다음 항목을 먼저 고정한다.

- 세션명 번호와 주제를 확인한다. 예: `002_*`
- 작업 브랜치명을 `task_{agent}/{seq}-{scope}` 형식으로 확정한다.
- 관련 Issue가 있는지 확인한다.
- 관련 Issue가 없고 작업이 단순 확인을 넘어서면 Issue 등록 단계를 먼저 수행한다.
- Issue, 브랜치명, PR 제목 또는 본문의 작업 번호와 범위가 서로 어긋나지 않게 맞춘다.
- 결론이 확정된 사안도 현재 PR 포함, 별도 PR, Backlog, 새 Issue 후보 중 처리 경로를 확인한다.
- 동시작업 가능성이 있으면 GitHub Issue 번호를 기준으로 Backlog ID, 브랜치명, PR 순번 충돌 여부를 확인한다.

Issue 등록과 브랜치 규칙은 [POL-003 Git 작업관리방안](../03.정책/POL-003_Git_작업관리방안.md)을 따른다.

## 10. 작업 시작 기준

작업을 시작하기 전에 다음 항목이 명확해야 한다.

- 현재 역할
- 기준 브랜치
- 세션명 번호와 주제
- 작업 브랜치명
- 관련 Issue 또는 Backlog
- 작업 범위
- 제외 범위
- 완료 조건
- 검증 방법
- 확정 사안의 처리 경로
- Issue 기준 채번 충돌 여부

불명확한 항목이 있으면 작업 전에 확인한다.

`#태스크시작` 없이 대화 중 작업이 먼저 진행된 경우에는 작업을 중단하기보다 `#태스크정리`에서 누락된 항목을 사후 보정한다. 단, 큰 작업을 Issue 없이 계속 진행하지 않고 현재 Issue 반영, `[번외]`, Backlog, 새 Issue 후보 중 하나로 분류한다.

사용자 주문과 표준 태그 절차가 충돌하거나, 절차 생략 승인으로 해석될 여지가 있으면 실행 전에 확인한다. 명시 태그가 필요한 단계는 자연어 주문만으로 자동 실행하지 않는다.

세션명 현행화처럼 단순 주문으로 보이는 요청도 실제 처리에 Issue, 작업 브랜치, PR, 승급 같은 절차 상태 변경이 필요하면 실행 전에 현재 절차 조건을 확인하고 사용자 승인을 받는다.

### 요청 해석 게이트

AI는 사용자 요청을 실행하기 전에 요청 유형을 먼저 판정한다.

| 요청 유형 | 기준 | 실행 범위 |
|---|---|---|
| 질문/검토 | 설명, 검토, 비교, 가능 여부 판단처럼 결과물이 답변인 요청 | 답변만 수행하고 저장소 또는 외부 시스템 상태를 변경하지 않는다. |
| 작업 방향 합의 | “정리하자”, “검토하자”, “방향을 잡자”처럼 범위나 다음 절차를 정하는 요청 | 범위, 후보, 필요한 태그를 보고하고 상태 변경은 수행하지 않는다. |
| 절차 실행 | `#태스크시작`, `#태스크정리`, `#태스크승급`, `#세션정리`처럼 명시 태그가 있거나 상태 변경 대상과 승인 범위가 명확한 요청 | 해당 태그의 필수 실행조건을 확인한 뒤 절차를 수행한다. |

Issue 생성/종료, 작업 브랜치 생성, 파일 수정, 커밋, push, PR 생성/Ready 전환/병합, `dev/stg/main` 승급은 상태 변경으로 본다.

상태 변경이 필요한 요청은 실행 전에 다음 항목을 보고한다.

- 요청 유형
- 필요한 태그 또는 명시 승인
- 변경 대상
- 중단 조건
- 현재 실행 가능한 범위

## 11. 세션 시작 보고 형식

`#세션시작`을 수행한 뒤에는 다음 항목을 요약해 보고한다.

- 원격 `dev`, `stg`, `main` 일치 여부
- 현재 브랜치와 작업트리 상태
- 최신 회고 문서와 다음 세션 권장 시작점
- 미해결 Backlog 요약
- 최신 회고와 다음 작업 후보를 반영한 초기 세션명
- 세션명 번호와 추천 작업 브랜치명
- 관련 Issue 존재 여부와 필요 시 Issue 등록 단계
- 추천 다음 작업

## 12. 세션 정리 절차

`#세션정리`를 수행할 때는 다음 항목을 확인한다.

| 순서 | 확인 항목 | 완료 기준 |
|---|---|---|
| 1 | 저장소 상태 | 원격 `dev`, `stg`, `main` 일치 여부와 작업트리 상태를 확인한다. |
| 2 | 현재 Issue | 제목, 본문, 완료 조건, 관련 PR, 남은 범위가 실제 작업 결과와 맞는지 확인한다. |
| 3 | 세션명 | 세션명이 현재 Issue, PR, 후속 작업 범위를 반영하는지 현행화한다. |
| 4 | 회고 | 세션 회고를 현재 Issue의 마무리 태스크로 정리한다. 별도 Issue는 만들지 않는다. |
| 5 | Backlog | 해결된 항목은 `Resolved`로 갱신하고, 미해결 인덱스가 최신인지 확인한다. |
| 6 | 다음 세션 | 다음 세션명, 권장 시작점, 남은 Issue와 Backlog, 관련 문서 링크를 고정한다. |
| 7 | 미처리 항목 | 세션 정리 중 처리하지 못한 항목을 명시한다. |
| 8 | 자기 재점검 | 세션정리 절차 자체를 변경했다면 반영과 승급 후 변경된 기준으로 다시 확인한다. |

회고는 기본적으로 저장소 문서 또는 Issue 댓글처럼 다음 세션에서 다시 찾을 수 있는 위치에 남긴다. 회고 결과가 독립적인 정책, 설계, 운영 개선 작업으로 커지면 Backlog 또는 새 Issue 후보로 분리한다.

`#태스크시작`, `#태스크정리`, `#태스크승급`, `#세션정리`를 논스톱으로 진행하려면 사용자가 네 단계의 연속 실행과 중간 확인 생략을 명시해야 한다. 논스톱 진행 중에도 충돌, 실패, 새 판단 지점이 생기면 중단하고 확인한다.

세션 정리 보고에는 다음 세션에서 그대로 사용할 수 있는 시작 프롬프트를 포함한다.

세션 정리 보고에는 `세션명 현행화` 블록을 별도로 포함한다. 이 블록은 다음 항목을 빠짐없이 적는다.

- 이전 세션명
- 현재 작업 기준 세션명
- 다음 세션 추천 세션명
- Issue, PR, 브랜치와의 정합성 판정
- 불일치가 있으면 수행한 보정 또는 남은 보정 항목

```markdown
#세션시작

이전 세션 종료 기준:
- 이전 세션명: <이전 세션명>
- 현재 작업 기준 세션명: <현재 세션명>
- 다음 세션 추천 세션명: <다음 세션명>
- 최신 회고: <RET 문서 링크>
- 다음 세션 권장 시작점: <Backlog 또는 Issue>
- 남은 Backlog: <미해결 Backlog 요약>
- 확인할 Issue/PR: <Issue와 PR 링크>
- 기준 브랜치/커밋: <dev/stg/main 일치 커밋>

요청:
1. 원격 dev/stg/main 일치 여부와 작업트리 상태를 먼저 확인해줘.
2. 최신 회고와 Backlog 인덱스를 대조해 다음 작업 후보를 확인해줘.
3. 다음 작업 후보를 기준으로 초기 세션명을 제안하고, 관련 Issue 또는 Backlog 확정 전까지 가설로 표시해줘.
4. 작업 시작 전 세션명, 브랜치명, Issue/Backlog, 작업 범위, 제외 범위, 완료 조건, 검증 방법을 보고해줘.
5. 사용자 주문과 표준 절차가 충돌하거나 해석이 갈리면 실행 전에 확인해줘.
```

## 13. 세션 태그

AI는 세션 중 다음 태그와 별칭을 같은 명령으로 인식한다.

| 표준 태그 | 허용 별칭 | 목적 |
|---|---|---|
| `#세션시작` | `#sessionstart`, `#session-start` | 세션 시작 절차 수행 |
| `#태스크시작` | `#테스크시작`, `#task시작`, `#taskstart`, `#task-start` | 작업 범위 확인과 브랜치 준비 |
| `#태스크정리` | `#테스크정리`, `#task정리`, `#taskclose`, `#task-close` | 작업 검증, PR, `dev` 머지 정리 |
| `#태스크승급` | `#테스크승급`, `#task승급`, `#taskpromote` | 자동 승급 |
| `#태스크승급-확인` | `#테스크승급-확인`, `#task승급-check`, `#taskpromote-check` | `main` 승급 전 확인 |
| `#태스크승급-검증` | `#테스크승급-검증`, `#task승급-stg`, `#taskpromote-stg` | `stg` 검증 후 피드백 대기 |
| `#세션정리` | `#sessionclose`, `#session-close` | 세션 회고와 종료 정리 |

승급 태그의 상세 절차는 [POL-003 Git 작업관리방안](../03.정책/POL-003_Git_작업관리방안.md)을 따른다.

## 14. 세션 태그 실행 게이트

세션 태그는 단순 보고 명령이 아니라 실행 게이트로 처리한다.

AI는 태그를 인식하면 다음 순서로 진행한다.

```text
태그 인식
↓
필수 실행조건 확인
↓
미충족 항목 전환 판단
↓
실행 범위 보고
↓
태그 절차 수행
↓
보정 발생 시 보정 보고
↓
검증 결과 보고
```

필수 실행조건이 비어 있으면 바로 실행하지 않는다. 미충족 항목은 [POL-005 사람 승인 경계 정책](../03.정책/POL-005_사람_승인_경계_정책.md)의 기준에 따라 `Need Approval`, `Stop`, `사후 보정`, `Backlog` 중 하나로 전환한다.

| 태그 | 필수 실행조건 | 실행 전 확인 | 미충족 시 기본 전환 |
|---|---|---|---|
| `#세션시작` | 원격 확인 가능, 현재 저장소 확인 가능 | `dev/stg/main` 일치 여부, 현재 브랜치, 작업트리, 최신 회고, Backlog, 작업 브랜치 생성하지 않음 | 원격 또는 저장소 확인 불가 시 Stop |
| `#태스크시작` | 관련 Backlog 또는 작업 목적 존재, 작업 범위 초안 존재 | Issue 존재 여부, 세션명, 브랜치명, 제외 범위, 완료 조건, 검증 방법, 작업 범위와 Issue 확인 전 작업 브랜치 생성하지 않음 | Issue 없는 큰 작업은 Issue 생성 우선, 범위 불명확 시 Need Approval |
| `#태스크정리` | 작업 결과 존재, 관련 Issue 또는 Backlog 식별 가능 | diff, 문서 탈고, PR 연결, 후속 Backlog, 완료 조건 충족 여부 | 누락이 경미하면 사후 보정, 완료 조건 미충족 시 Stop |
| `#태스크승급` | PR이 `dev`에 병합됨, 검증 결과 존재, 승급 승인 범위 명확 | `dev -> stg -> main` 대상 커밋, 완료 조건, 리뷰/렌더링 확인, 원격 브랜치 상태 | 검증 미완료 또는 승인 불명확 시 Need Approval 또는 Stop |
| `#태스크승급-확인` | PR이 `dev`에 병합됨, `stg` 검증 목적 명확 | `stg` 반영 대상, 기본 검증 방법, `main` 승급 전 확인 지점 | `stg` 검증 조건 불명확 시 Need Approval |
| `#태스크승급-검증` | PR이 `dev`에 병합됨, 검증 피드백 대기 기준 명확 | `stg` 반영 대상, 검증 항목, 실패 기록 위치 | 검증 방법 없음 또는 기록 위치 불명확 시 Stop |
| `#세션정리` | 현재 세션 산출물 식별 가능 | 저장소 상태, Issue/PR, 회고, Backlog, 다음 세션 시작점, 미처리 항목 | 고정해야 할 산출물 누락 시 사후 보정 또는 Stop |

## 15. 태그별 보고 기준

태그 실행 보고에는 다음을 포함한다.

| 태그 | 보고 기준 |
|---|---|
| `#세션시작` | 원격 브랜치 일치 여부, 현재 브랜치, 작업트리, 최신 회고, 미해결 Backlog, 추천 다음 작업 |
| `#태스크시작` | 관련 Issue, 작업 브랜치, 작업 범위, 제외 범위, 완료 조건, 검증 방법, 승인 상태 |
| `#태스크정리` | 변경 요약, 검증 결과, 세션명 현행화 결과, PR 상태, `dev` 머지 완료 여부, 미완료 시 보류 사유와 다음 조치, Issue 완료 조건 충족 여부, 후속 Backlog |
| `#태스크승급` | 승급 대상 커밋, `stg` 검증 결과, `main` 반영 결과, `dev/stg/main` 일치 여부, Issue 미종료 상태 |
| `#태스크승급-확인` | `stg` 반영 결과, 기본 검증 결과, `main` 승급 전 확인 요청 |
| `#태스크승급-검증` | `stg` 반영 결과, 검증 대기 항목, 실패 시 기록 위치 |
| `#세션정리` | 세션명 현행화 블록, 회고 위치, 남은 Issue/PR/Backlog, 다음 세션 시작 프롬프트, 작업트리 상태 |

`#태스크승급`은 Issue를 종료하지 않는다. 승급 후에는 Issue 상태를 확인해 보고만 하고, Issue 종료는 `#태스크정리` 또는 `#세션정리`에서 완료 조건, 후속 Backlog, 남은 범위를 별도로 확인한 뒤 명시 승인 또는 별도 종료 절차로 수행한다.

## 16. 보정 보고 기준

태그 실행 중 브랜치, Issue, PR, Backlog, 검증 조건 같은 필수 실행조건을 보정한 경우 AI는 보정 직후 다음 항목을 보고한다.

- 보정 전 막힌 조건
- 수행한 보정 조치
- 보정 후 현재 상태
- 아직 남은 필수 실행조건
- 다음 진행 선택지

보정 보고는 최종 결과 보고로 대체하지 않는다. 보정으로 실행 흐름이 회복되면 남은 필수 실행조건을 확인한 뒤 다음 절차로 진행한다.
