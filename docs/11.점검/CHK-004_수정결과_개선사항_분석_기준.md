## 1. 목적

본 문서는 프로그램 수정 이후 변경 결과를 분석해 개발자가 인사이트를 얻을 수 있는 개선사항 보고 기준을 정의한다.

수정결과 개선사항 분석은 작업 완료 검증을 대체하지 않는다. 검증은 결과가 완료 조건을 만족했는지 확인하고, 본 문서는 변경 과정에서 드러난 설계 개선점, 리팩토링 후보, 구현 백로그, 향후 대응 조건을 정리한다.

## 2. 적용 범위

본 문서는 다음 작업에 적용한다.

- 코드 변경이 포함된 Issue
- 하나의 주제로 여러 파일이 함께 변경된 작업
- 반복 수정이 예상되는 구조나 흐름을 건드린 작업
- 현재 범위에서는 수용하지만 향후 설계 개선이 필요할 수 있는 작업
- Harness Report에 개발자 관점의 후속 판단을 남겨야 하는 작업

문서 오탈자 수정, 단순 링크 수정, 인덱스 갱신처럼 구조적 개선 판단이 필요 없는 Low-risk 작업에는 간단한 생략 사유만 남길 수 있다.

## 3. 기본 원칙

수정결과 개선사항 분석의 기본 원칙은 다음과 같다.

- 현재 Issue의 완료 조건과 후속 개선 후보를 분리한다.
- 작업 범위 밖의 개선은 현재 결과에 섞지 않고 Backlog로 분리한다.
- 좋은 설계 방향이 있어도 지금 적용하지 않는다면 이유를 기록한다.
- 리팩토링 필요성은 감상이 아니라 변경 비용, 반복 수정 가능성, 결합도 근거로 설명한다.
- 검증 실패나 미완료를 개선사항 보고로 포장하지 않는다.
- 개발자가 다음 수정 때 판단할 수 있는 구체 단서로 남긴다.

## 4. 분석 관점

Harness 또는 구현 에이전트는 수정 결과를 다음 관점으로 분석한다.

| 분석 관점 | 질문 | 보고 내용 |
|---|---|---|
| 변경 범위 | 어떤 파일과 모듈이 변경되었는가? | 변경 파일, 변경 이유, 공통 주제 |
| 공통 원인 | 여러 파일 변경이 같은 원인에서 발생했는가? | 중복 책임, 공통 인터페이스, 정책/모델 변경 여부 |
| 결합도 | 변경이 여러 위치로 번진 이유가 무엇인가? | 강한 결합, 암묵적 계약, 분산된 규칙 |
| 설계 개선 | 더 나은 구조가 보였는가? | 개선 후보와 기대 효과 |
| 리팩토링 | 지금은 아니지만 정리해야 할 구현 부채가 있는가? | 리팩토링 후보와 발생 조건 |
| 유지 판단 | 좋은 설계가 있지만 현 상태를 유지한 이유가 있는가? | 범위, 비용, 위험, 검증 부족 사유 |
| 후속 작업 | 구현 Backlog 또는 추가 Issue가 필요한가? | Backlog 후보, 우선순위, 선행 조건 |
| 재검토 조건 | 다음 수정 때 다시 봐야 할 신호가 있는가? | 반복 변경, 테스트 실패, 변경 파일 증가 조건 |

분석 관점은 모든 작업에 기계적으로 채우지 않는다. 해당되는 관점만 짧고 명확하게 기록한다.

## 5. 보고 상태

수정결과 개선사항 분석은 다음 상태 중 하나로 정리한다.

| 상태 | 한글명 | 의미 | 후속 처리 |
|---|---|---|---|
| No Finding | 발견 없음 | 구조적 개선 후보가 확인되지 않음 | 생략 사유 기록 |
| Insight Only | 인사이트만 있음 | 당장 작업은 없지만 참고할 판단이 있음 | Report에 기록 |
| Backlog Candidate | 백로그 후보 | 후속 작업으로 분리할 개선점이 있음 | Backlog 등록 검토 |
| Refactor Candidate | 리팩토링 후보 | 구현 구조 정리가 필요할 수 있음 | 구현 Backlog 또는 별도 Issue 후보 |
| Design Review Needed | 설계 검토 필요 | 설계 판단이 필요해 현재 작업에서 확정하기 어려움 | 설계 또는 CTO 검토 요청 |

보고 상태는 결과 수용 상태가 아니다. 결과 수용은 [POL-006 근거 기반 결과 수용 기준](../03.정책/POL-006_근거_기반_결과_수용_기준.md)을 따른다.

## 6. Harness Report 기록 기준

Harness Report에는 필요한 경우 다음 항목을 포함한다.

```markdown
## 수정결과 개선사항 분석

- 보고 상태: No Finding(발견 없음) / Insight Only(인사이트만 있음) / Backlog Candidate(백로그 후보) / Refactor Candidate(리팩토링 후보) / Design Review Needed(설계 검토 필요)
- 변경 범위:
- 여러 파일 변경의 공통 원인:
- 설계 개선 후보:
- 리팩토링 후보:
- 지금 처리하지 않는 이유:
- 구현 Backlog 후보:
- 다음 수정 시 재검토 조건:
```

현재 작업에서 후속 개선을 처리하지 않는 경우에도, 이유를 남기면 이후 작업자가 같은 판단을 반복하지 않을 수 있다.

## 7. Backlog 분리 기준

다음 조건 중 하나라도 해당하면 개선사항을 현재 작업에 섞지 않고 Backlog 후보로 분리한다.

- 현재 Issue의 완료 조건을 바꿔야 한다.
- 변경 범위가 기존 수행 범위를 넘어선다.
- 테스트 또는 검증 범위를 새로 설계해야 한다.
- 구조 변경으로 영향 분석이 필요하다.
- 정책, 설계, 운영 기준 변경이 필요하다.
- 리팩토링 이득은 있지만 현재 작업의 위험이 커진다.

Backlog 후보는 구체적인 작업 단위로 적는다. 단순히 “리팩토링 필요”라고 쓰지 않고, 어떤 책임이나 파일 묶음을 어떤 이유로 정리해야 하는지 남긴다.

## 8. 생략 기준

다음 경우에는 상세 분석을 생략할 수 있다.

- 변경이 단일 문서의 오탈자 또는 표현 정리다.
- 링크 또는 인덱스 갱신처럼 구조적 판단이 없다.
- 작업 결과가 삭제 없이 단순 상태 갱신에 그친다.
- 개선 후보가 현재 문서나 Backlog에 이미 등록되어 있다.

생략하더라도 Report에는 `No Finding(발견 없음)` 또는 생략 사유를 한 줄로 남긴다.
